# QA/테스트 프로세스 가이드

## 1. 개요

본 문서는 Konda 프로젝트의 QA/테스트 프로세스를 정의합니다. 외주사에서 1차 QA를 진행하고, 내부에서 최종 검수를 수행합니다.

---


## 3. 외주사 1차 QA

### 3.1 QA 범위

외주사가 담당하는 QA 범위:

| 영역 | QA 항목 |
|------|---------|
| BE(비금융) | API 기능 테스트, 단위 테스트 |
| 랜딩 페이지 | UI 테스트, 크로스 브라우저 테스트 |
| 기획 검증 | 기획서 대비 구현 확인 |

### 3.2 QA 체크리스트 (외주사)

#### 기능 테스트
- [ ] 모든 요구사항 구현 확인
- [ ] 정상 케이스 동작 확인
- [ ] 예외 케이스 처리 확인
- [ ] 에러 메시지 표시 확인

#### API 테스트
- [ ] 엔드포인트 정상 동작
- [ ] 요청/응답 형식 확인
- [ ] 인증/인가 동작 확인
- [ ] 에러 코드 반환 확인

#### 크로스 브라우저 (랜딩)
- [ ] Chrome
- [ ] Safari
- [ ] Firefox
- [ ] Edge

### 3.3 QA 완료 알림

```markdown
**[QA 완료] 기능명**

## QA 범위
- [테스트한 기능/화면 목록]

## QA 결과
- 테스트 케이스: N건
- 통과: N건
- 버그 발견 후 수정: N건
- 미해결: N건 (있다면 상세 기재)

## 테스트 환경
- 환경: [스테이징/로컬]
- 브라우저: [크롬 등]
- 버전: [코드 버전/커밋]

## 특이 사항
- [있다면 기재]

@내부PM @내부QA
```

## 4. 내부 최종 검수

### 4.1 검수 범위

내부에서 담당하는 최종 검수 범위:

| 영역 | 검수 항목 |
|------|----------|
| App 통합 | App과 API 통합 테스트 |
| BE(금융) | 금융 API 기능 테스트 |
| 전체 플로우 | E2E 시나리오 테스트 |
| 보안 | 보안 취약점 검토 |
| 성능 | 기본 성능 확인 |

### 4.2 검수 체크리스트 (내부)

#### 통합 테스트
- [ ] App ↔ BE(비금융) 연동 정상
- [ ] App ↔ BE(금융) 연동 정상
- [ ] 전체 사용자 플로우 테스트
- [ ] 엣지 케이스 테스트

#### 보안 검수
- [ ] 인증/인가 정상 동작
- [ ] 민감 정보 노출 없음
- [ ] 입력값 검증 확인
- [ ] API 접근 권한 확인

#### UI/UX 검수
- [ ] 디자인 시안 대비 구현 확인
- [ ] 반응형 동작 확인
- [ ] 로딩/에러 상태 표시 확인

## 5. 버그 리포팅

### 5.1 버그 리포팅 채널

**Slack #proj-konda-dev** 채널에서 버그를 리포팅합니다.

### 5.2 버그 리포트 템플릿

```markdown
**[버그] 제목**

## 심각도
- [ ] Critical: 서비스 전체 장애
- [ ] High: 주요 기능 사용 불가
- [ ] Medium: 일부 기능 이상
- [ ] Low: 사소한 문제

## 환경
- 환경: 스테이징 / 프로덕션
- 기기/브라우저:
- 앱 버전: (해당 시)

## 재현 단계
1. [1단계]
2. [2단계]
3. [3단계]

## 예상 동작
- [정상적으로 기대하는 동작]

## 실제 동작
- [실제로 발생한 문제]

## 스크린샷/영상
[첨부]

## 로그 (있다면)
```
[에러 로그]
```

@담당개발팀
```

### 5.3 버그 심각도 분류

| 심각도 | 정의 | 수정 기한 | 예시 |
|--------|------|----------|------|
| Critical | 서비스 전체 장애 | 즉시 | 서버 다운, 보안 취약점 |
| High | 주요 기능 사용 불가 | 24시간 내 | 결제 실패, 로그인 불가 |
| Medium | 일부 기능 이상 | 스프린트 내 | 화면 깨짐, 특정 조건 오류 |
| Low | 사소한 문제 | 백로그 | 오타, 미세 UI 이슈 |

## 6. 버그 수정 프로세스

### 6.1 수정 흐름

```
버그 리포트 접수
      │
      ▼
담당자 할당 (Slack 스레드에서)
      │
      ▼
버그 원인 분석 & 수정
      │
      ▼
수정 완료 알림
      │
      ▼
QA 재테스트
      │
      ▼
버그 종료 (Resolved 이모지)
```

### 6.2 수정 완료 알림

버그 리포트 스레드에 답글:

```markdown
**[수정 완료]**

- 원인: [버그 원인 간략 설명]
- 수정 내용: [수정 사항]
- 배포: [스테이징 배포 완료 / 프로덕션 배포 예정]
- 커밋: [SHA 또는 PR 링크]

재테스트 부탁드립니다.
@QA담당자
```

### 6.3 버그 상태 관리

Slack 이모지로 상태 표시:

| 이모지 | 상태 |
|--------|------|
| 👀 | 확인 중 |
| 🔧 | 수정 중 |
| ✅ | 수정 완료 (재테스트 필요) |
| ✔️ | 재테스트 통과 (종료) |
| ❌ | 재현 불가 / 반려 |

## 7. 테스트 케이스 작성

### 7.1 테스트 케이스 형식

```markdown
## TC-[번호]: [테스트 케이스명]

**전제 조건**
- [테스트 전 필요한 상태/조건]

**테스트 단계**
1. [단계 1]
2. [단계 2]
3. [단계 3]

**예상 결과**
- [기대하는 결과]

**실제 결과**
- [ ] Pass
- [ ] Fail: [실패 시 내용]

**테스트 데이터**
- [사용한 테스트 데이터]
```

### 7.2 테스트 케이스 분류

| 유형 | 설명 |
|------|------|
| 정상 케이스 | 정상적인 사용 시나리오 |
| 경계값 테스트 | 최소/최대값, 경계 조건 |
| 에러 케이스 | 잘못된 입력, 예외 상황 |
| 권한 테스트 | 인증/인가 관련 |

## 8. 테스트 환경

### 8.1 환경 구분

| 환경 | 용도 | 데이터 |
|------|------|--------|
| 개발 (Dev) | 개발자 테스트 | 더미 데이터 |
| 스테이징 (Staging) | QA 테스트 | 테스트 데이터 |
| 프로덕션 (Prod) | 실서비스 | 실제 데이터 |

### 8.2 테스트 데이터 원칙

- 프로덕션 데이터 복사 금지
- 개인정보 포함 금지
- 테스트 전용 계정 사용
- 금융 거래는 테스트 모드 사용
