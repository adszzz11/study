# 기획 프로세스 가이드

## 1. 개요

본 문서는 Konda 프로젝트의 기획 프로세스를 정의합니다. 기획서는 외주사에서 작성하고, 내부에서 검토 및 승인하는 프로세스를 따릅니다.

---

## 2. 기획 프로세스 흐름

```
┌──────────────────────────────────────────────────────────────┐
│                      기획 프로세스                             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 요구사항 수집 ──→ 2. 기획서 초안 작성 ──→ 3. 내부 검토    │
│       (협의)              (외주사)              (슈퍼블록)     │
│                                                    │         │
│                              ┌────────────────────┘         │
│                              ▼                              │
│                    ┌─────────────────┐                      │
│                    │   피드백 있음?   │                      │
│                    └────────┬────────┘                      │
│                             │                               │
│              Yes ◄──────────┼──────────► No                 │
│               │             │             │                  │
│               ▼             │             ▼                  │
│         4. 피드백 반영      │      5. 기획서 확정            │
│            (외주사)         │         (승인)                │
│               │             │             │                  │
│               └─────────────┘             ▼                  │
│                                    6. 디자인 착수            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 3. 기획서 작성 가이드

### 3.1 기획서 종류

| 문서 유형 | 용도 | 작성 시점 |
|----------|------|----------|
| 서비스 기획서 | 전체 서비스 개요 및 구조 | 프로젝트 초기 |
| 기능 명세서 | 개별 기능 상세 정의 | 기능 개발 전 |
| 화면 정의서 | 화면별 UI 요소 및 동작 정의 | 디자인 착수 전 |
| 정책 문서 | 서비스 운영 정책 정의 | 필요 시 |

### 3.2 기획서 작성 형식

## 4. 기획서 검토 프로세스

### 4.1 검토 단계

```
외주사 기획서 작성 완료
        │
        ▼
Slack #konda-planning 채널에 검토 요청
        │
        ▼
내부 검토 (PM + 개발팀)
  - 검토 기간: 영업일 기준 2일 이내
        │
        ▼
피드백 전달 (Slack 또는 GitHub Issue)
        │
        ▼
외주사 피드백 반영
        │
        ▼
재검토 (필요 시)
        │
        ▼
최종 승인 (PM)
```

### 4.2 검토 요청 형식

```markdown
**[기획 검토 요청] 제목**

- 문서 위치: [GitHub Wiki 링크]
- 기능 요약: [1-2줄 요약]
- 검토 요청 사항:
  - [특별히 검토받고 싶은 부분]
- 희망 검토 완료일: YYYY-MM-DD

@PM @개발팀
```

### 4.3 피드백 형식

```markdown
**[기획 피드백] 제목**

## 검토 결과: 승인 / 수정 후 재검토 / 반려

## 피드백 내용

### 필수 수정 사항
1. [수정 필요 항목]

### 권장 수정 사항
1. [권장 수정 항목]

### 질의 사항
1. [확인 필요 사항]

## 다음 단계
- [수정 후 재검토 요청 / 디자인 착수 가능 등]
```

### 4.4 검토 기준

| 검토 항목 | 확인 내용 |
|----------|----------|
| 완성도 | 필수 항목 모두 포함 여부 |
| 명확성 | 요구사항이 명확하게 기술되었는지 |
| 실현 가능성 | 기술적으로 구현 가능한지 |
| 일관성 | 기존 기획/서비스와 일관성 있는지 |
| 보안 | 보안 정책에 부합하는지 |

## 5. 기획 변경 관리

### 5.1 변경 요청 프로세스

```
변경 필요성 인지
      │
      ▼
Slack #konda-planning 채널에서 협의
      │
      ▼
변경 영향도 분석
  - 일정 영향
  - 개발 범위 영향
  - 디자인 영향
      │
      ▼
변경 승인 (PM)
      │
      ▼
기획서 업데이트
      │
      ▼
관련 팀 공유
```

### 5.2 변경 요청 형식

```markdown
**[기획 변경 요청] 제목**

## 변경 대상
- 문서: [기획서 링크]
- 해당 기능: [기능명]

## 변경 내용
### 현재
- [현재 기획 내용]

### 변경 후
- [변경하려는 내용]

## 변경 사유
- [변경이 필요한 이유]

## 영향 범위
- 디자인: [영향 있음/없음]
- 개발: [영향 있음/없음]
- 일정: [영향 있음/없음]

@PM
```

### 5.3 변경 이력 관리

기획서 변경 시:
1. 문서 내 변경 이력 테이블 업데이트
2. 버전 번호 증가 (예: 1.0 → 1.1)
3. 변경 내용 Slack 공유

## 6. 기획서 관리

### 6.1 저장 위치

- **GitHub Wiki**: 확정된 기획서
- **작업 중 문서**: GitHub Wiki 또는 협의된 위치

### 6.2 네이밍 규칙

```
[기능영역]-[기능명]-기획서.md

예시:
- user-login-기획서.md
- payment-card-registration-기획서.md
- content-feed-기획서.md
```

### 6.3 버전 관리

| 상태 | 버전 예시 | 설명 |
|------|----------|------|
| 초안 | 0.1, 0.2 | 검토 전 초안 |
| 검토중 | 0.9 | 검토 요청 상태 |
| 확정 | 1.0 | 최초 확정 버전 |
| 수정 | 1.1, 1.2 | 확정 후 변경 |
| 대규모 변경 | 2.0 | 큰 변경 시 메이저 버전 증가 |
