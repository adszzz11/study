# 커뮤니케이션 가이드

## 1. 개요

본 문서는 Konda 프로젝트의 내부 팀과 외주사 간 원활한 소통을 위한 커뮤니케이션 가이드라인을 정의합니다.

---

## 2. 커뮤니케이션 채널

### 2.1 Slack

**주요 소통 채널**로 Slack을 사용합니다.

#### 채널 구조

| 채널명 | 용도 | 참여자 |
|--------|------|--------|
| `#konda-general` | 일반 공지, 전체 소통 | 전체 |
| `#konda-dev` | 개발 관련 논의 | 개발팀 전체 |
| `#konda-design` | 디자인 관련 논의 | 디자인팀, App팀, PM |
| `#konda-planning` | 기획 관련 논의 | 기획팀, PM, 개발팀 |
| `#konda-bug` | 버그 리포팅 | QA, 개발팀 전체 |
| `#konda-deploy` | 배포 관련 알림 | 개발팀 전체, PM |

#### Slack 사용 가이드

**메시지 작성 시**
- 긴급도에 따라 적절한 채널 사용
- 담당자 멘션(@)으로 명확한 전달
- 스레드 사용으로 대화 정리

**응답 시간 기준**
| 긴급도 | 응답 시간 |
|--------|----------|
| 긴급 (Critical) | 30분 이내 |
| 높음 (High) | 2시간 이내 |
| 보통 (Normal) | 당일 내 |
| 낮음 (Low) | 2일 이내 |

**긴급 메시지 형식**
```
[긴급] 제목

내용:
- 상황 설명
- 요청 사항
- 기대 응답 시간

@담당자
```

## 3. 정기 미팅

### 3.1 미팅 일정

**주 2-3회** 정기 미팅을 진행합니다.

| 미팅 유형 | 주기 | 시간 | 참석자 |
|----------|------|------|--------|
| 전체 싱크 미팅 | 주 1회 (월요일) | 10:00-11:00 | 전체 |
| 개발 싱크 미팅 | 주 1-2회 | 14:00-14:30 | 개발팀 |
| 기획/디자인 리뷰 | 필요 시 | 협의 | PM, 기획, 디자인 |

### 3.2 미팅 목적별 안내

#### 전체 싱크 미팅 (주 1회)
- **목적**: 전체 진행 상황 공유, 이슈 논의
- **참석자**: PM, 개발팀, 기획팀, 디자인팀
- **아젠다**:
  1. 지난 주 진행 상황 공유 (10분)
  2. 이번 주 계획 공유 (10분)
  3. 블로커/이슈 논의 (20분)
  4. Q&A (10분)

#### 개발 싱크 미팅 (주 1-2회)
- **목적**: 개발 진행 상황, 기술 이슈 논의
- **참석자**: App팀, BE팀(금융), BE팀(비금융)
- **아젠다**:
  1. 개발 진행 상황 (10분)
  2. API 연동 이슈 (10분)
  3. 기술 이슈 논의 (10분)

### 3.3 미팅 아젠다 템플릿

```markdown
# [미팅 유형] - YYYY-MM-DD

## 참석자
- 내부: [이름]
- 외주사: [이름]

## 아젠다
1. [ ] 항목 1
2. [ ] 항목 2
3. [ ] 항목 3

## 논의 내용
### 1. 항목 1
- 내용

## 결정 사항
- [ ] 결정 1 (담당: @담당자, 기한: YYYY-MM-DD)

## 다음 미팅
- 일시: YYYY-MM-DD HH:MM
- 아젠다:
```

### 3.4 회의록 관리
- 미팅 종료 후 24시간 내 회의록 작성
- GitHub Wiki에 업로드
- Slack `#konda-general`에 링크 공유

## 4. 에스컬레이션 프로세스

### 4.1 에스컬레이션 단계

```
Level 1: 담당자 직접 소통 (Slack)
    ↓ (해결 불가 시)
Level 2: 팀 리드 개입
    ↓ (해결 불가 시)
Level 3: PM 에스컬레이션
    ↓ (긴급 시)
Level 4: 전화 연락
```

### 4.2 이슈 유형별 대응

| 이슈 유형 | 긴급도 | 초기 대응 | 에스컬레이션 기준 |
|----------|--------|----------|------------------|
| 서비스 장애 | Critical | 즉시 전화 | 즉시 PM 보고 |
| 보안 이슈 | Critical | 즉시 전화 | 즉시 PM 보고 |
| 일정 지연 | High | Slack 논의 | 1일 내 미해결 시 |
| 기술 이슈 | Medium | Slack 논의 | 2일 내 미해결 시 |
| 일반 문의 | Low | Slack 논의 | 3일 내 미응답 시 |

### 4.3 긴급 연락망

**긴급 상황 발생 시 연락 순서:**

```
1단계: Slack 멘션 → 30분 대기
2단계: 전화 연락 → 즉시
3단계: PM 에스컬레이션 → 1시간 내 해결 불가 시
```

#### 긴급 연락처

| 역할 | 담당자 | 전화번호 |
|------|--------|---------|
| 내부 PM | [TBD] | [TBD] |
| 외주사 PM | [TBD] | [TBD] |
| 내부 개발 리드 | [TBD] | [TBD] |
| 외주사 개발 리드 | [TBD] | [TBD] |

## 5. 소통 에티켓

### 5.1 기본 원칙
- 명확하고 간결한 메시지 작성
- 적절한 채널 사용
- 담당자 멘션으로 명확한 전달
- 요청 시 기한 명시
- 응답 시 확인 메시지 전송

### 5.2 피해야 할 사항
- 업무 시간 외 긴급 연락 (긴급 상황 제외)
- 불명확한 요청
- 여러 채널에 동일 메시지 중복 발송
- 멘션 없이 중요 메시지 전송

### 5.3 업무 시간
- **기본 업무 시간**: 평일 09:00 - 18:00
- **긴급 연락 가능 시간**: 평일 09:00 - 21:00
- **주말/공휴일**: 긴급 상황 시에만 연락

## 6. 요청/피드백 프로세스

### 6.1 일반 요청

```
요청자 → Slack 해당 채널에 요청 작성
         ↓
담당자 → 확인 응답 (이모지 또는 메시지)
         ↓
담당자 → 처리 후 완료 보고
```

### 6.2 요청 메시지 형식

```markdown
**[요청] 제목**

- 요청 내용:
- 요청 사유:
- 기한: YYYY-MM-DD
- 담당자: @담당자

참고 자료: [링크]
```

### 6.3 피드백 제공

```markdown
**[피드백] 제목**

- 대상: [기획서/디자인/개발물 등]
- 피드백 내용:
  1. 항목 1
  2. 항목 2
- 우선순위: [높음/보통/낮음]
- 기한: YYYY-MM-DD
```
