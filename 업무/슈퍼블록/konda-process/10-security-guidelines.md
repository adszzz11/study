# 보안 가이드라인

## 1. 개요

본 문서는 Konda 프로젝트의 보안 가이드라인을 정의합니다. 금융 서비스 특성상 높은 수준의 보안이 요구되며, 모든 참여자는 이 가이드라인을 준수해야 합니다.

---

## 2. 보안 코딩 가이드

### 2.1 입력 값 검증

#### 필수 원칙
- 모든 사용자 입력은 서버에서 검증
- 화이트리스트 방식 적용 (허용된 값만 통과)
- 입력 길이 및 형식 제한

#### 예시
```javascript
// Bad
function processInput(data) {
  return db.query(`SELECT * FROM users WHERE id = ${data.id}`);
}

// Good
function processInput(data) {
  const id = parseInt(data.id, 10);
  if (isNaN(id) || id <= 0) {
    throw new ValidationError('Invalid ID');
  }
  return db.query('SELECT * FROM users WHERE id = ?', [id]);
}
```

### 2.2 SQL Injection 방지

- Prepared Statement / Parameterized Query 사용 필수
- ORM 사용 시에도 raw query 주의
- 동적 쿼리 생성 금지

### 2.3 XSS (Cross-Site Scripting) 방지

- 출력 시 HTML 이스케이프 처리
- Content-Security-Policy 헤더 설정
- 사용자 입력을 HTML에 직접 삽입 금지

### 2.4 인증/인가

#### 인증
- 강력한 비밀번호 정책 적용
- 세션 타임아웃 설정
- 로그인 시도 횟수 제한
- 2단계 인증(2FA) 지원

#### 인가
- 최소 권한 원칙 적용
- 리소스 접근 시 권한 검증
- 수평적/수직적 권한 상승 방지

### 2.5 민감 정보 처리

```
[민감 정보 분류]
- Level 1 (최고): 금융 거래 정보, 결제 정보, 비밀번호
- Level 2 (높음): 개인식별정보, 계좌 정보
- Level 3 (보통): 연락처, 이메일
- Level 4 (낮음): 일반 서비스 데이터
```

#### 처리 원칙
- 민감 정보 암호화 저장
- 전송 시 TLS 1.2 이상 사용
- 로그에 민감 정보 출력 금지
- 필요 최소한의 정보만 수집

## 3. 데이터 격리 정책

### 3.1 금융/비금융 데이터 격리

```
┌─────────────────────────────────────────────────────────────┐
│                     데이터 격리 구조                          │
├─────────────────────────┬───────────────────────────────────┤
│    비금융 데이터 영역     │         금융 데이터 영역            │
│    (외주사 접근 가능)     │         (내부만 접근)              │
├─────────────────────────┼───────────────────────────────────┤
│  - 사용자 프로필         │  - 계좌 정보                       │
│  - 콘텐츠 데이터         │  - 거래 내역                       │
│  - 일반 서비스 데이터     │  - 결제 정보                       │
│                         │  - 자산 정보                       │
└─────────────────────────┴───────────────────────────────────┘
```

### 3.2 격리 원칙

1. **네트워크 격리**
   - 금융 시스템은 별도 네트워크 구성
   - 외주사 개발 환경에서 금융 시스템 직접 접근 불가

2. **데이터베이스 격리**
   - 금융/비금융 데이터베이스 물리적 분리
   - 외주사에 금융 DB 접근 권한 미부여

3. **API 격리**
   - 금융 API는 내부에서만 개발/관리
   - 외주사는 비금융 API만 개발

### 3.3 데이터 접근 매트릭스

| 데이터 유형 | 내부 개발팀 | 외주사 | 비고 |
|------------|-----------|-------|------|
| 사용자 기본 정보 | R/W | R (마스킹) | 테스트 데이터만 |
| 콘텐츠 데이터 | R/W | R/W | - |
| 계좌 정보 | R/W | X | 접근 불가 |
| 거래 내역 | R/W | X | 접근 불가 |
| 결제 정보 | R/W | X | 접근 불가 |

## 4. 로깅 정책

### 4.1 로깅 원칙

#### 로깅 대상
- 사용자 인증 이벤트 (로그인, 로그아웃, 실패)
- 권한 변경 이벤트
- 데이터 접근/변경 이벤트
- 시스템 오류 및 예외
- API 호출 기록

#### 로깅 제외 대상
- 비밀번호 (평문/해시 모두)
- 신용카드 번호
- 주민등록번호
- 계좌번호 (마스킹 필수)

### 4.2 로그 형식

```json
{
  "timestamp": "2024-01-15T10:30:00.000Z",
  "level": "INFO",
  "service": "konda-api",
  "traceId": "abc123",
  "userId": "user_***456",
  "action": "LOGIN_SUCCESS",
  "ip": "192.168.x.x",
  "userAgent": "...",
  "message": "User login successful",
  "metadata": {
    // 민감 정보 제외
  }
}
```

### 4.3 마스킹 규칙

| 데이터 유형 | 마스킹 방식 | 예시 |
|------------|-----------|------|
| 이메일 | 앞 3자 + *** + 도메인 | `abc***@example.com` |
| 전화번호 | 뒤 4자리만 표시 | `***-****-1234` |
| 계좌번호 | 뒤 4자리만 표시 | `*********1234` |
| 카드번호 | 앞 6자리 + 뒤 4자리 | `123456******7890` |
| 주민번호 | 앞 6자리만 | `900101-*******` |

### 4.4 로그 보관

- **보관 기간**: 최소 1년 (법적 요구사항에 따라 조정)
- **접근 권한**: 내부 보안 담당자만
- **암호화**: 저장 시 암호화 적용

## 5. 금지 사항 목록

### 5.1 개발 관련 금지 사항

| 항목 | 금지 사항 | 대안 |
|------|----------|------|
| 하드코딩 | API 키, 비밀번호 코드 내 하드코딩 | 환경변수, Secret Manager 사용 |
| 디버그 정보 | 프로덕션에서 디버그 모드 활성화 | 환경별 설정 분리 |
| 에러 노출 | 스택 트레이스 사용자에게 노출 | 일반화된 에러 메시지 |
| 로깅 | 민감 정보 로그 출력 | 마스킹 처리 |
| 암호화 | 자체 암호화 알고리즘 사용 | 표준 암호화 라이브러리 |

### 5.2 인프라 관련 금지 사항

| 항목 | 금지 사항 |
|------|----------|
| 포트 | 불필요한 포트 개방 |
| 접근 | 프로덕션 DB 직접 접근 |
| 권한 | root/admin 계정으로 서비스 실행 |
| 전송 | HTTP 사용 (HTTPS 필수) |

### 5.3 데이터 관련 금지 사항

| 항목 | 금지 사항 |
|------|----------|
| 복사 | 프로덕션 데이터 로컬 환경 복사 |
| 공유 | 기밀 정보 외부 공유 |
| 저장 | 개인 기기에 업무 데이터 저장 |
| 전송 | 암호화되지 않은 채널로 민감 정보 전송 |

## 6. 보안 체크리스트

### 6.1 개발 전 체크리스트

- [ ] 보안 교육 이수 완료
- [ ] NDA 및 보안서약 서명 완료
- [ ] 2단계 인증(2FA) 설정 완료
- [ ] 개발 환경 보안 설정 확인

### 6.2 코드 작성 시 체크리스트

- [ ] 모든 입력 값 검증 구현
- [ ] SQL Injection 방지 (Prepared Statement 사용)
- [ ] XSS 방지 (출력 이스케이프)
- [ ] 인증/인가 검증 구현
- [ ] 민감 정보 암호화
- [ ] 에러 메시지 일반화
- [ ] 로그에 민감 정보 미포함
- [ ] 하드코딩된 시크릿 없음

### 6.3 코드 리뷰 시 체크리스트

- [ ] 보안 취약점 검토
- [ ] 입력 검증 로직 확인
- [ ] 권한 검증 로직 확인
- [ ] 민감 정보 처리 방식 확인
- [ ] 로깅 내용 확인
- [ ] 외부 라이브러리 보안 검토

### 6.4 배포 전 체크리스트

- [ ] 디버그 모드 비활성화
- [ ] 테스트 계정/데이터 제거
- [ ] 환경 변수 설정 확인
- [ ] HTTPS 설정 확인
- [ ] 보안 헤더 설정 확인
- [ ] 취약점 스캔 완료

## 7. 보안 사고 대응

### 7.1 보안 사고 유형

| 유형 | 예시 | 심각도 |
|------|------|--------|
| 정보 유출 | 개인정보 유출, 소스코드 유출 | Critical |
| 시스템 침해 | 해킹, 악성코드 감염 | Critical |
| 서비스 장애 | DDoS, 시스템 다운 | High |
| 권한 남용 | 무단 접근, 권한 상승 | High |
| 정책 위반 | 보안 정책 미준수 | Medium |

### 7.2 보안 사고 보고

```
보안 사고 발생 시:

1. 발견 즉시 (1시간 이내)
   → 내부 PM 및 보안 담당자에게 보고

2. 보고 내용:
   - 사고 발생 일시
   - 사고 유형 및 내용
   - 영향 범위 (추정)
   - 초기 대응 조치

3. 보고 채널:
   - 긴급: 전화
   - 일반: Slack #konda-security (생성 필요)
```
