# 개발 프로세스 가이드

## 1. 개요

본 문서는 Konda 프로젝트의 개발 프로세스를 정의합니다. 내부 팀과 외주사가 분리된 저장소에서 개발하며, 브랜치 전략은 외주사 자율로 운영합니다.

---

## 2. 저장소 구조

### 2.1 전체 구조

```
GitHub Organization: [Konda-Project]
│
├── 내부 Repository (Private)
│   ├── konda-app          # iOS/Android 앱
│   ├── konda-api-finance  # 금융 API 서버
│   └── konda-infra        # 인프라 설정
│
└── 외부 Repository (외주사 접근 가능)
    ├── konda-api          # 비금융 API 서버
    └── konda-landing      # 랜딩 페이지
```

### 2.2 접근 권한

| Repository | 내부 팀 | 외주사 |
|------------|--------|-------|
| konda-app | Read/Write | X (접근 불가) |
| konda-api-finance | Read/Write | X (접근 불가) |
| konda-api | Read/Write | Read/Write |
| konda-landing | Read/Write | Read/Write |

### 2.3 외주사 계정 관리

- GitHub Organization 초대 (외주사 팀원)
- 해당 Repository에만 권한 부여
- 프로젝트 종료 시 권한 회수

---

## 3. 브랜치 전략

### 3.1 외주사 브랜치 전략

**외주사 자율 운영**

외주사는 자체 브랜치 전략을 사용합니다. 다만, 아래 권장사항을 참고해 주세요.

#### 권장 사항 (참고용)

```
main (또는 master)
  └── 항상 배포 가능한 상태 유지
  └── 직접 커밋 금지, PR을 통해서만 병합

develop (선택)
  └── 개발 통합 브랜치
  └── feature 브랜치 병합 대상

feature/*
  └── 기능 개발 브랜치
  └── 예: feature/login, feature/payment

hotfix/*
  └── 긴급 버그 수정
```

### 3.2 내부 브랜치 전략

내부 Repository는 아래 전략을 따릅니다.

```
main
  └── 프로덕션 배포 브랜치

develop
  └── 개발 통합 브랜치

feature/[기능명]
  └── 기능 개발

hotfix/[이슈번호]
  └── 긴급 수정
```

## 4. 코드 컨벤션 가이드

### 4.1 일반 원칙

- 가독성 우선
- 일관된 스타일 유지
- 자체 문서화 코드 작성
- 주석은 "왜(Why)"를 설명

### 4.2 네이밍 컨벤션

| 대상 | 스타일 | 예시 |
|------|--------|------|
| 변수 | camelCase | `userName`, `totalAmount` |
| 함수 | camelCase | `getUserInfo()`, `calculateTotal()` |
| 클래스 | PascalCase | `UserService`, `PaymentController` |
| 상수 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT`, `API_BASE_URL` |
| 파일명 | kebab-case 또는 camelCase | `user-service.js`, `UserService.ts` |

### 4.3 코드 포맷팅

- **들여쓰기**: 2 spaces 또는 4 spaces (프로젝트별 통일)
- **최대 줄 길이**: 100-120자
- **포맷터**: Prettier, ESLint 등 자동화 도구 권장

### 4.4 언어별 가이드 (참고)

각 기술 스택에 맞는 공식 스타일 가이드를 따르되, 프로젝트 내 일관성을 유지합니다.

## 5. 커밋 메시지 컨벤션

### 5.1 커밋 메시지 형식

```
[type] 제목 (50자 이내)

본문 (선택, 72자마다 줄바꿈)
- 무엇을 왜 변경했는지 설명
- 이슈 번호 참조 (Fixes #123)
```

### 5.2 커밋 타입

| Type | 설명 | 예시 |
|------|------|------|
| `feat` | 새로운 기능 | `[feat] 로그인 화면 추가` |
| `fix` | 버그 수정 | `[fix] 결제 오류 수정` |
| `docs` | 문서 변경 | `[docs] README 업데이트` |
| `style` | 코드 스타일 (포맷팅 등) | `[style] 코드 포맷팅` |
| `refactor` | 리팩토링 | `[refactor] 인증 로직 개선` |
| `test` | 테스트 | `[test] 로그인 테스트 추가` |
| `chore` | 빌드, 설정 등 | `[chore] 의존성 업데이트` |

### 5.3 좋은 커밋 메시지 예시

```
[feat] 사용자 프로필 조회 API 추가

- GET /api/users/{id}/profile 엔드포인트 구현
- 프로필 이미지 URL 포함
- 캐싱 적용 (TTL: 5분)

Refs: #45
```

```
[fix] 결제 금액 계산 오류 수정

할인 적용 시 소수점 버림 처리가 누락되어
1원 단위 오차 발생

Fixes: #78
```

## 6. Pull Request 가이드

### 6.1 PR 생성 규칙

- 하나의 PR은 하나의 목적
- 작은 단위로 PR 생성 (300줄 이하 권장)
- 관련 이슈 연결
- 리뷰어 지정

### 6.2 PR 템플릿

```markdown
## 변경 사항
<!-- 이 PR에서 변경된 내용 -->

## 변경 유형
- [ ] 새 기능
- [ ] 버그 수정
- [ ] 리팩토링
- [ ] 문서/설정

## 관련 이슈
<!-- Fixes #이슈번호 -->

## 테스트
<!-- 테스트 방법/결과 -->

## 체크리스트
- [ ] 코드 컨벤션 준수
- [ ] 테스트 코드 작성/통과
- [ ] 문서 업데이트 (필요 시)

## 스크린샷 (UI 변경 시)
<!-- 변경 전/후 스크린샷 -->
```

### 6.3 코드 리뷰

#### 외주사 내부 리뷰
- 외주사 자체 코드 리뷰 진행
- 리뷰 완료 후 main 브랜치 병합

#### 내부 리뷰 (필요 시)
- 주요 기능 또는 아키텍처 변경 시
- 보안 관련 코드 변경 시
- 내부 팀에 리뷰 요청

## 7. 개발 워크플로우

### 7.1 기능 개발 흐름

```
1. 기획/디자인 확정
       │
       ▼
2. GitHub Issue 생성 (또는 할당)
       │
       ▼
3. 브랜치 생성 (feature/xxx)
       │
       ▼
4. 개발 & 로컬 테스트
       │
       ▼
5. PR 생성 & 코드 리뷰
       │
       ▼
6. 리뷰 반영 & 머지
       │
       ▼
7. QA 환경 배포 & 테스트
       │
       ▼
8. 버그 수정 (필요 시)
       │
       ▼
9. 배포 요청
```

### 7.2 핫픽스 흐름

```
1. 버그 발견 & 이슈 생성
       │
       ▼
2. hotfix 브랜치 생성
       │
       ▼
3. 수정 & 테스트
       │
       ▼
4. 긴급 리뷰 & 머지
       │
       ▼
5. 즉시 배포
```

## 8. 외주사 → 내부 코드 전달

### 8.1 전달 방식

외주사는 자체 Repository에서 개발하고, 내부 팀이 해당 코드를 배포합니다.

```
외주사 개발 완료
      │
      ▼
main 브랜치 최신화
      │
      ▼
Slack #proj-konda-dev에 배포 요청
      │
      ▼
내부 팀 코드 확인
      │
      ▼
내부 인프라에 배포
```

### 8.2 배포 요청 형식

```markdown
**[배포 요청] konda-api**

- Repository: [링크]
- Branch: main
- Commit: [SHA]
- 배포 대상 환경: 스테이징 / 프로덕션
- 변경 내용 요약:
  - [변경 1]
  - [변경 2]
- 배포 희망일: YYYY-MM-DD

@내부PM @내부개발팀
```
