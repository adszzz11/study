# 위험 관리 프로세스

## 1. 개요

본 문서는 Konda 프로젝트의 위험 관리 프로세스를 정의합니다. 프로젝트 진행 중 발생할 수 있는 위험을 사전에 식별하고 대응 방안을 수립합니다.

---

## 2. 위험 관리 프로세스

### 2.1 전체 흐름

```
┌──────────────────────────────────────────────────────────────┐
│                    위험 관리 프로세스                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 위험 식별 ──→ 2. 위험 평가 ──→ 3. 대응 계획 수립          │
│                                          │                   │
│                                          ▼                   │
│                               4. 대응 실행 & 모니터링          │
│                                          │                   │
│                                          ▼                   │
│                               5. 정기 검토 & 업데이트          │
│                                          │                   │
│                    └─────────────────────┘                   │
│                           (반복)                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 위험 관리 주기

| 활동 | 주기 | 담당 |
|------|------|------|
| 위험 식별 | 상시 + 스프린트 시작 시 | 전체 |
| 위험 평가 | 위험 식별 시 | PM |
| 대응 계획 수립 | 위험 평가 후 즉시 | PM + 담당팀 |
| 모니터링 | 상시 | PM |
| 정기 검토 | 주간 미팅 시 | PM + 전체 |

## 3. 위험 식별 체계

### 3.1 위험 식별 방법

| 방법 | 설명 | 시점 |
|------|------|------|
| 정기 미팅 | 주간 미팅에서 위험 논의 | 주 1회 |
| Slack 보고 | 위험 발견 시 즉시 보고 | 상시 |
| 스프린트 계획 | 스프린트 시작 시 점검 | 스프린트당 |
| 회고 미팅 | 과거 위험 분석 | 스프린트당 |

### 3.2 위험 식별 질문

위험 식별 시 고려할 질문:

```
- 일정에 영향을 줄 수 있는 요소는?
- 기술적으로 어려운 부분은?
- 의존성이 있는 항목은?
- 외부 요인으로 인한 리스크는?
- 리소스 관련 이슈는?
- 보안 관련 우려 사항은?
- 커뮤니케이션 관련 이슈는?
```

### 3.3 위험 보고 형식

```markdown
**[위험 보고]**

## 위험 식별 정보
- 보고자:
- 보고일:

## 위험 내용
- 위험명:
- 설명:
- 영향 받는 영역:

## 예상 영향
- 일정 영향: [있음/없음] - 상세:
- 품질 영향: [있음/없음] - 상세:
- 비용 영향: [있음/없음] - 상세:

## 발생 가능성
- [높음/중간/낮음]

## 제안 대응책
- [대응 방안 제안]

@PM
```

## 4. 위험 유형 분류

### 4.1 위험 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **일정 위험** | 프로젝트 일정에 영향 | 개발 지연, 리뷰 지연, 의존성 지연 |
| **기술 위험** | 기술적 문제 발생 | 기술 난이도, 성능 이슈, 호환성 문제 |
| **리소스 위험** | 인력/자원 관련 문제 | 핵심 인력 이탈, 리소스 부족 |
| **커뮤니케이션 위험** | 소통 관련 문제 | 요구사항 오해, 응답 지연 |
| **외부 위험** | 외부 요인 | 외부 API 변경, 규제 변경 |
| **보안 위험** | 보안 관련 문제 | 보안 취약점, 데이터 유출 |
| **품질 위험** | 품질 관련 문제 | 버그 다발, 성능 미달 |

### 4.2 일반적인 외주 프로젝트 위험

| 위험 | 발생 가능성 | 대응 |
|------|------------|------|
| 요구사항 변경 빈번 | 높음 | 변경 관리 프로세스 적용 |
| 커뮤니케이션 지연 | 중간 | 정기 미팅, 응답 시간 기준 설정 |
| 일정 지연 | 중간 | 버퍼 일정, 조기 경보 |
| 품질 이슈 | 중간 | 코드 리뷰, QA 강화 |
| 핵심 인력 이탈 | 낮음 | 문서화, 지식 공유 |
| 보안 사고 | 낮음 | 보안 가이드라인, 모니터링 |

## 5. 위험 평가 매트릭스

### 5.1 평가 기준

#### 발생 가능성 (Probability)

| 등급 | 설명 | 기준 |
|------|------|------|
| 높음 (H) | 발생 가능성 높음 | 70% 이상 |
| 중간 (M) | 발생 가능성 있음 | 30-70% |
| 낮음 (L) | 발생 가능성 낮음 | 30% 미만 |

#### 영향도 (Impact)

| 등급 | 설명 | 기준 |
|------|------|------|
| 높음 (H) | 심각한 영향 | 일정 2주+ 지연, 주요 기능 불가 |
| 중간 (M) | 상당한 영향 | 일정 1-2주 지연, 일부 기능 영향 |
| 낮음 (L) | 미미한 영향 | 일정 1주 미만, 사소한 영향 |

### 5.2 위험 등급 매트릭스

```
         │ 낮음 (L) │ 중간 (M) │ 높음 (H) │
         │   영향   │   영향   │   영향   │
─────────┼──────────┼──────────┼──────────┤
높음 (H) │   중간   │   높음   │  치명적  │
가능성   │          │          │          │
─────────┼──────────┼──────────┼──────────┤
중간 (M) │   낮음   │   중간   │   높음   │
가능성   │          │          │          │
─────────┼──────────┼──────────┼──────────┤
낮음 (L) │   낮음   │   낮음   │   중간   │
가능성   │          │          │          │
─────────┴──────────┴──────────┴──────────┘
```

### 5.3 위험 등급별 대응

| 등급 | 색상 | 대응 수준 |
|------|------|----------|
| 치명적 | 🔴 Red | 즉시 대응, PM 직접 관리 |
| 높음 | 🟠 Orange | 우선 대응, 주간 모니터링 |
| 중간 | 🟡 Yellow | 대응 계획 수립, 정기 모니터링 |
| 낮음 | 🟢 Green | 모니터링, 필요시 대응 |

## 6. 대응 계획 수립

### 6.1 대응 전략

| 전략 | 설명 | 적용 상황 |
|------|------|----------|
| **회피** | 위험 원인 제거 | 위험 발생 전 예방 가능 시 |
| **완화** | 발생 가능성/영향 감소 | 위험 통제 가능 시 |
| **전가** | 제3자에게 위험 이전 | 외부 전문가 활용 시 |
| **수용** | 위험 감수 | 낮은 위험, 대응 비용 과다 시 |

### 6.2 대응 계획 형식

```markdown
**위험 대응 계획**

## 위험 정보
- 위험 ID: RISK-001
- 위험명:
- 위험 등급: [치명적/높음/중간/낮음]

## 대응 전략
- 전략: [회피/완화/전가/수용]

## 대응 계획
### 예방 조치
1. [사전 예방 활동]

### 발생 시 조치
1. [발생 시 대응 활동]

### 비상 계획
1. [최악의 경우 대응]

## 담당자
- 대응 담당: [담당자]
- 모니터링 담당: [담당자]

## 마감일
- 예방 조치 완료: YYYY-MM-DD
- 모니터링 종료: YYYY-MM-DD
```

## 7. 모니터링 및 보고

### 7.1 모니터링 방법

| 위험 등급 | 모니터링 주기 | 보고 대상 |
|----------|--------------|----------|
| 치명적 | 매일 | PM + 경영진 |
| 높음 | 주 2-3회 | PM |
| 중간 | 주 1회 | PM |
| 낮음 | 격주 | PM |

### 7.2 위험 상태 추적

위험 상태:
- **Open**: 활성 위험, 모니터링 중
- **Mitigated**: 완화됨, 영향 감소
- **Occurred**: 발생함, 대응 중
- **Closed**: 종료됨 (해결 또는 소멸)

### 7.3 주간 위험 보고

주간 미팅에서 위험 현황 공유:

```markdown
**[주간 위험 현황] YYYY-MM-DD**

## 요약
- 총 위험: N건
- 신규: N건
- 종료: N건
- 발생: N건

## 주요 위험 현황

### RISK-001: [위험명]
- 등급: [높음]
- 상태: [Open]
- 변동: [악화/유지/개선]
- 대응 현황: [현재 상태]

### RISK-002: [위험명]
- 등급: [중간]
- 상태: [Mitigated]
- 변동: [개선]
- 대응 현황: [현재 상태]

## 신규 위험
- [신규 식별된 위험]

## 다음 주 중점 사항
- [집중 모니터링 항목]
```
