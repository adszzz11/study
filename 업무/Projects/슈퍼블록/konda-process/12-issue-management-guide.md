## 3. GitHub Issues 사용 가이드

### 3.1 이슈 작성

#### 이슈 생성 시점
- 새로운 기능 개발 시
- 버그 발견 시
- 개선 사항 발생 시
- 기술 부채 정리 시

#### 이슈 템플릿

**기능 요청 (Feature)**
```markdown
## 기능 설명
<!-- 구현할 기능에 대한 설명 -->

## 배경/목적
<!-- 이 기능이 필요한 이유 -->

## 요구사항
- [ ] 요구사항 1
- [ ] 요구사항 2

## 관련 문서
- 기획서: [링크]
- 디자인: [Figma 링크]

## 참고 사항
<!-- 개발 시 참고할 사항 -->
```

**버그 리포트 (Bug)**
```markdown
## 버그 설명
<!-- 발생한 버그에 대한 설명 -->

## 재현 단계
1. 단계 1
2. 단계 2
3. 단계 3

## 예상 동작
<!-- 정상적으로 기대하는 동작 -->

## 실제 동작
<!-- 실제로 발생한 문제 -->

## 환경
- 환경: 스테이징 / 프로덕션
- 브라우저/기기:
- 버전:

## 스크린샷
<!-- 해당 시 첨부 -->

## 로그
<!-- 에러 로그 (해당 시) -->
```

**작업 (Task)**
```markdown
## 작업 설명
<!-- 수행할 작업에 대한 설명 -->

## 체크리스트
- [ ] 항목 1
- [ ] 항목 2
- [ ] 항목 3

## 완료 조건
<!-- 이 작업이 완료되었다고 판단하는 기준 -->
```

### 3.2 라벨 체계

#### 이슈 유형 라벨

| 라벨 | 색상 | 설명 |
|------|------|------|
| `type: feature` | 🟢 Green | 새로운 기능 |
| `type: bug` | 🔴 Red | 버그 |
| `type: enhancement` | 🔵 Blue | 기존 기능 개선 |
| `type: task` | 🟡 Yellow | 일반 작업 |
| `type: docs` | ⚪ Gray | 문서 작업 |

#### 우선순위 라벨

| 라벨 | 색상 | 설명 |
|------|------|------|
| `priority: critical` | 🔴 Red | 즉시 처리 |
| `priority: high` | 🟠 Orange | 높은 우선순위 |
| `priority: medium` | 🟡 Yellow | 중간 우선순위 |
| `priority: low` | 🟢 Green | 낮은 우선순위 |

#### 상태 라벨

| 라벨 | 색상 | 설명 |
|------|------|------|
| `status: backlog` | ⚪ Gray | 백로그 |
| `status: ready` | 🔵 Blue | 작업 준비 완료 |
| `status: in-progress` | 🟣 Purple | 진행 중 |
| `status: review` | 🟡 Yellow | 리뷰 중 |
| `status: blocked` | 🔴 Red | 블로킹 |

#### 영역 라벨

| 라벨 | 설명 |
|------|------|
| `area: api` | API 관련 |
| `area: frontend` | 프론트엔드 관련 |
| `area: database` | 데이터베이스 관련 |
| `area: infra` | 인프라 관련 |

## 4. GitHub Projects 사용 가이드

### 4.1 프로젝트 보드 구조

#### 칸반 보드 컬럼

| 컬럼 | 설명 |
|------|------|
| Backlog | 예정된 작업 |
| To Do | 이번 스프린트 작업 |
| In Progress | 진행 중 |
| Review | 코드 리뷰 중 |
| Done | 완료 |

### 4.2 보드 뷰

| 뷰 | 용도 |
|------|------|
| Board (칸반) | 전체 작업 현황 파악 |
| Table | 상세 정보 확인 |
| Roadmap | 일정/마일스톤 확인 |

### 4.3 이슈 → 프로젝트 연결

1. 이슈 생성
2. 이슈 우측 "Projects" 에서 프로젝트 선택
3. 적절한 컬럼에 배치

## 5. 작업 흐름

### 5.1 이슈 생명주기

```
1. 이슈 생성
       │
       ▼
2. 라벨/마일스톤 할당
       │
       ▼
3. 담당자 할당
       │
       ▼
4. 작업 시작 (In Progress)
       │
       ▼
5. PR 생성 & 리뷰 (Review)
       │
       ▼
6. 머지 & 이슈 종료 (Done)
```

### 5.2 이슈와 PR 연결

PR에서 이슈 참조:
```
Fixes #123
Closes #123
Resolves #123
```

PR 설명에 관련 이슈 링크:
```markdown
## 관련 이슈
- #123: [이슈 제목]
```

### 5.3 일일 작업 관리

```
1. 아침: 오늘 할 이슈 To Do → In Progress 이동
2. 작업 중: 진행 상황 코멘트 (필요 시)
3. 완료 시: PR 생성, Review 이동
4. 머지 후: Done 이동
```

## 6. 스프린트 관리

### 6.1 스프린트 계획

```
스프린트 시작 미팅
      │
      ▼
백로그에서 이슈 선정
      │
      ▼
마일스톤/프로젝트에 이슈 할당
      │
      ▼
스프린트 진행
      │
      ▼
스프린트 종료/회고
```

### 6.2 스프린트 현황 공유

GitHub Projects 보드를 활용하여:
- 완료된 이슈 수
- 진행 중인 이슈
- 블로커 이슈
- 스프린트 진행률

## 7. 이슈 관리 Best Practices

### 7.1 좋은 이슈 작성

- 명확하고 구체적인 제목
- 충분한 컨텍스트 제공
- 완료 조건 명시
- 관련 링크 첨부

### 7.2 이슈 관리 원칙

- 하나의 이슈에 하나의 작업
- 너무 큰 이슈는 분할
- 정기적으로 오래된 이슈 정리
- 완료된 이슈는 바로 Close

### 7.3 피해야 할 사항

- 제목만 있는 빈 이슈
- 너무 광범위한 이슈
- 라벨/담당자 없는 이슈 방치
- 오래된 이슈 방치
