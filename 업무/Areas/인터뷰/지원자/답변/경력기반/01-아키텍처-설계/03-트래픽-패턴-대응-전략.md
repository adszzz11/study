# 적립 거래와 충전 거래의 트래픽 패턴이 다른 상황에서 오버 스펙 대비 없이 어떻게 해결했는지 구체적으로 설명해주세요.

## 답변

적립 거래와 충전 거래는 트래픽 패턴이 완전히 다릅니다. 적립은 하루 평균 10만 TPS로 거의 일정한 반면, 충전은 평균 1만 TPS이지만 프로모션 시작 직후 5분간 8만 TPS까지 스파이크가 발생했습니다.

오버 스펙 없이 해결한 핵심 전략은 **서비스 분리 + 트래픽 패턴별 차별화된 스케일링 전략**입니다:

1. **마이크로서비스 분리**: 적립 서비스와 충전 서비스를 독립적으로 배포하여 각각의 트래픽 패턴에 맞게 스케일링
2. **HPA 정책 차별화**: 적립은 CPU 기반 정적 스케일링(replica 20개 고정), 충전은 동적 HPA(최소 5개, 최대 30개)
3. **비동기 처리 + 메시지 큐**: 충전 요청의 급증 트래픽을 Kafka로 버퍼링하여 처리량 평활화
4. **우선순위 큐 도입**: 충전 요청을 일반/VIP로 구분하여 중요 트래픽 보호

결과적으로 서버 대수를 30% 줄이면서도(60대 → 42대) P99 레이턴시를 120ms에서 80ms로 개선했습니다. 클라우드 비용은 월 $15,000에서 $10,500으로 30% 절감되었습니다.

## 핵심 키워드

- 트래픽 패턴 분석
- 탄력적 확장 (Elastic Scaling)
- 리소스 최적화
- 비용 효율성
- 부하 분산

## 트래픽 패턴 분석

### 적립 거래
- **트래픽 특성**: 평균 10만 TPS, 변동폭 ±5% 이내
- **시간대별 패턴**:
  - 오전 9시~오후 11시: 일정한 트래픽 유지 (95,000~105,000 TPS)
  - 심야 시간대: 80,000 TPS로 소폭 감소
  - 주말/평일 차이 거의 없음 (±3%)
- **요청 특성**:
  - 경량 트랜잭션 (평균 응답 시간 15ms)
  - 읽기:쓰기 비율 = 3:7 (잔액 조회 30%, 적립 처리 70%)
  - 배치로 묶어 처리 가능 (최대 100건 배치)
- **예측 가능성**: 높음 - 과거 1개월 데이터 기반 98% 정확도로 예측

### 충전 거래
- **트래픽 특성**: 평균 1만 TPS, 하지만 높은 변동성
- **스파이크 패턴**:
  - 프로모션 시작 직후: 0→80,000 TPS (30초 내 급증)
  - 지속 시간: 5분간 고점 유지 후 점진적 감소
  - 빈도: 주 2-3회 (수요일, 금요일 오후 8시 집중)
- **요청 특성**:
  - 중량 트랜잭션 (평균 응답 시간 80ms)
  - PG사 연동 포함 (외부 API 호출 50ms)
  - 읽기:쓰기 비율 = 1:9 (대부분 충전 처리)
  - 배치 처리 불가 - 즉시 응답 필요
- **예측 가능성**: 중간 - 프로모션 일정은 알지만 정확한 트래픽량 예측 어려움 (오차 ±40%)

## 해결 방안

### 1. 아키텍처 레벨

#### 1.1 마이크로서비스 분리

기존 모놀리식 구조를 도메인별로 분리하여 독립적인 스케일링 가능하도록 설계:

```yaml
# 기존: 단일 서비스
payment-service (replica: 40개 고정)
  ├── 적립 API
  └── 충전 API

# 개선: 서비스 분리
point-earn-service (replica: 20개 고정)
  └── 적립 전용, 안정적 트래픽 처리

point-charge-service (replica: 5~30개 동적)
  └── 충전 전용, 탄력적 스케일링
```

**분리 기준**:
- 트래픽 패턴의 차이 (일정 vs 스파이크)
- 처리 복잡도 (단순 적립 vs PG 연동 충전)
- SLA 요구사항 (P99 20ms vs 100ms)

#### 1.2 비동기 처리 아키텍처

충전 요청을 즉시 처리하지 않고 Kafka를 통해 버퍼링:

```java
@RestController
public class ChargeController {

    @PostMapping("/v2/charge")
    public ResponseEntity<ChargeResponse> charge(@RequestBody ChargeRequest request) {
        // 1. 요청 검증 (5ms)
        validator.validate(request);

        // 2. Kafka 발행 (2ms) - 즉시 반환
        String requestId = UUID.randomUUID().toString();
        ChargeEvent event = ChargeEvent.builder()
            .requestId(requestId)
            .userId(request.getUserId())
            .amount(request.getAmount())
            .priority(determinePriority(request.getUserId()))
            .build();

        kafkaTemplate.send("charge-requests", event);

        // 3. 클라이언트에게 즉시 응답 (총 7ms)
        return ResponseEntity.accepted()
            .body(ChargeResponse.builder()
                .requestId(requestId)
                .status("PROCESSING")
                .build());
    }
}

@Service
public class ChargeConsumer {

    @KafkaListener(topics = "charge-requests", concurrency = "30")
    public void processCharge(ChargeEvent event) {
        try {
            // 1. 멱등성 체크 (Redis, 2ms)
            if (isDuplicate(event.getRequestId())) {
                return;
            }

            // 2. PG사 결제 처리 (50ms)
            PaymentResult result = pgService.pay(event);

            // 3. 잔액 업데이트 (10ms)
            balanceService.increase(event.getUserId(), event.getAmount());

            // 4. 완료 알림 (WebSocket/SSE, 3ms)
            notificationService.notify(event.getUserId(), result);

        } catch (Exception e) {
            // DLQ로 전송하여 재처리
            dlqPublisher.send(event);
        }
    }
}
```

**효과**:
- API 응답 시간: 80ms → 7ms (91% 개선)
- 스파이크 흡수: Kafka가 초당 80만 건 버퍼링 가능
- 장애 격리: PG사 장애 시에도 API는 정상 응답

#### 1.3 우선순위 큐 도입

VIP 사용자와 일반 사용자의 충전 요청을 분리 처리:

### 2. 인프라 레벨

#### 2.1 HPA(Horizontal Pod Autoscaler) 정책 차별화

**적립 서비스 - 정적 스케일링**:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: point-earn-service
spec:
  replicas: 20  # 고정값
  template:
    spec:
      containers:
      - name: app
        resources:
          requests:
            cpu: 500m
            memory: 1Gi
          limits:
            cpu: 1000m
            memory: 2Gi
      affinity:
        podAntiAffinity:  # 가용성 확보
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - point-earn-service
            topologyKey: kubernetes.io/hostname
```

**충전 서비스 - 동적 HPA**:
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: point-charge-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: point-charge-service
  minReplicas: 5
  maxReplicas: 30
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Pods
    pods:
      metric:
        name: kafka_consumer_lag
      target:
        type: AverageValue
        averageValue: "1000"  # Consumer Lag 기반 스케일링
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0  # 즉시 스케일 업
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
      - type: Pods
        value: 5
        periodSeconds: 15
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300  # 5분 후 스케일 다운
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
```

**HPA 정책 선택 이유**:
- 적립: 트래픽 예측 가능 → 오토스케일링 오버헤드 불필요
- 충전: 급격한 스파이크 → Kafka Lag 기반 선제적 스케일업
- Scale-up은 빠르게(15초), Scale-down은 천천히(5분) → 불필요한 재시작 방지

#### 2.2 Cluster Autoscaler 설정

노드 레벨에서도 탄력적 확장:

```yaml
apiVersion: v1
kind: ConfigMap

  name: cluster-autoscaler-priority-expander
```

**노드 풀 구성**:
- **Spot 인스턴스 풀**: 최대 20노드, 비용 70% 절감
  - 충전 서비스는 Stateless → Spot 종료 시 재시작 가능
- **On-Demand 풀**: 최대 10노드, 안정성 보장
  - 적립 서비스는 On-Demand 노드에서만 실행

#### 2.3 데이터베이스 Read Replica 활용

적립 조회 트래픽을 Read Replica로 분산:

```java
@Configuration
public class DataSourceConfig {

    @Bean
    public DataSource routingDataSource() {
        ReplicationRoutingDataSource dataSource = new ReplicationRoutingDataSource();

        DataSource master = createMasterDataSource();
        DataSource replica1 = createReplicaDataSource("replica-1");
        DataSource replica2 = createReplicaDataSource("replica-2");
        DataSource replica3 = createReplicaDataSource("replica-3");

        Map<Object, Object> sources = new HashMap<>();
        sources.put("master", master);
        sources.put("replica", new LazyConnectionDataSourceProxy(
            new RoundRobinDataSource(replica1, replica2, replica3)
        ));

        dataSource.setTargetDataSources(sources);
        dataSource.setDefaultTargetDataSource(master);

        return dataSource;
    }
}

@Service
public class PointEarnService {

    @Transactional(readOnly = true)  // Read Replica 사용
    public PointBalance getBalance(String userId) {
        return pointRepository.findByUserId(userId);
    }

    @Transactional  // Master 사용
    public void earnPoints(String userId, long amount) {
        pointRepository.updateBalance(userId, amount);
    }
}
```

**효과**:
- Master DB CPU: 85% → 45%
- Replica 3대로 읽기 트래픽 분산 (각 30k TPS)
- 적립 조회 P99: 25ms → 8ms

### 3. 애플리케이션 레벨

#### 3.1 배치 처리 최적화

적립 요청을 배치로 묶어 DB 부하 감소:

```java
@Service
public class PointEarnBatchService {

    private final BlockingQueue<EarnRequest> batchQueue =
        new ArrayBlockingQueue<>(10000);

    @Scheduled(fixedDelay = 100)  // 100ms마다 배치 처리
    public void processBatch() {
        List<EarnRequest> batch = new ArrayList<>();
        batchQueue.drainTo(batch, 100);  // 최대 100건씩

        if (batch.isEmpty()) return;

        // 단일 트랜잭션으로 처리
        transactionTemplate.execute(status -> {
            jdbcTemplate.batchUpdate(
                "UPDATE point_balance SET amount = amount + ? WHERE user_id = ?",
                batch,
                100,
                (ps, request) -> {
                    ps.setLong(1, request.getAmount());
                    ps.setString(2, request.getUserId());
                }
            );
            return null;
        });
    }

    public CompletableFuture<Void> earnAsync(EarnRequest request) {
        CompletableFuture<Void> future = new CompletableFuture<>();

        // 큐에 추가
        boolean added = batchQueue.offer(request);
        if (!added) {
            future.completeExceptionally(
                new RuntimeException("Queue is full"));
        } else {
            future.complete(null);
        }

        return future;
    }
}
```

**배치 처리 효과**:
- DB 트랜잭션 수: 100,000 TPS → 1,000 TPS (99% 감소)
- DB CPU 사용률: 80% → 35%
- 처리 지연: 평균 50ms (배치 대기 시간 포함)

#### 3.3 캐시 전략 차별화

적립 서비스에 적극적인 캐싱 적용:

```java
@Service
public class PointBalanceCacheService {

    private final RedisTemplate<String, PointBalance> redisTemplate;

    @Cacheable(value = "point-balance", key = "#userId")
    public PointBalance getBalance(String userId) {
        return pointRepository.findByUserId(userId);
    }

    @CacheEvict(value = "point-balance", key = "#userId")
    public void evictBalance(String userId) {
        // 적립/충전 후 캐시 무효화
    }

    // Write-Through 캐시: DB 업데이트와 동시에 캐시 갱신
    public void updateBalance(String userId, long amount) {
        pointRepository.updateBalance(userId, amount);

        PointBalance updated = pointRepository.findByUserId(userId);
        redisTemplate.opsForValue().set(
            "point-balance:" + userId,
            updated,
            Duration.ofMinutes(30)
        );
    }
}
```

**캐시 효과**:
- 적립 조회 캐시 히트율: 85%
- DB 읽기 부하: 30,000 TPS → 4,500 TPS (85% 감소)
- 조회 응답 시간: 8ms → 2ms (Redis 응답)

## 효과 및 결과

### 인프라 비용 절감
- **서버 대수**: 60대 → 42대 (30% 감소)
  - 적립 서비스: 40개 → 20개 (50% 감소)
  - 충전 서비스: 평균 10개 (기존 20개 대비 50% 감소), 피크 시 30개
- **클라우드 비용**: 월 $15,000 → $10,500 (30% 절감)
  - Spot 인스턴스 활용: 월 $3,500 추가 절감
  - Total 비용: 월 $7,000 (기존 대비 53% 절감)

### 성능 개선
- **P99 레이턴시**:
  - 적립 API: 120ms → 80ms → 15ms (최종 87% 개선)
  - 충전 API: 150ms → 7ms (초기 응답, 95% 개선)
- **처리량**:
  - 적립: 100,000 TPS 안정적 처리 (기존과 동일)
  - 충전: 80,000 TPS 스파이크 무장애 처리 (기존 50,000 TPS)
- **데이터베이스 부하**:
  - Master CPU: 85% → 45% (47% 감소)
  - Replica 3대 추가로 읽기 분산
  - Connection 사용률: 90% → 55%

### 가용성 향상
- **서비스 가동률**: 99.9% → 99.99%
  - 연간 다운타임: 8.76시간 → 52.6분
- **장애 격리**: PG사 장애 시에도 충전 API는 정상 응답 (7ms)
- **자동 복구**: HPA + Kafka Retry로 장애 자동 복구

### 운영 효율화
- **배포 시간**: 30분 → 5분 (서비스 분리로 독립 배포)
- **모니터링 정확도**: 서비스별 메트릭 분리로 장애 탐지 시간 10분 → 1분
- **온콜 호출 감소**: 월 15회 → 3회 (80% 감소)

## Trade-off 및 학습

### 선택한 Trade-off
1. **충전 API 응답 방식 변경**:
   - Before: 동기 처리 (80ms 응답)
   - After: 비동기 처리 (7ms 응답 + WebSocket 알림)
   - Trade-off: 클라이언트 코드 변경 필요, 하지만 UX는 개선 (로딩 스피너 대신 실시간 알림)

2. **적립 배치 처리**:
   - Before: 즉시 처리 (지연 0ms)
   - After: 배치 처리 (평균 지연 50ms)
   - Trade-off: 약간의 지연 발생, 하지만 대부분 사용자는 인지 못함 (50ms < 100ms JND)

3. **Spot 인스턴스 활용**:
   - Before: On-Demand만 사용 (안정성 100%)
   - After: Spot 70% + On-Demand 30% 혼용
   - Trade-off: 평균 주 1회 Spot 종료, 하지만 자동 재시작으로 무장애 (총 다운타임 < 10초/주)

### 시도했으나 실패한 방법
1. **Auto Scaling 단독 사용**:
   - CPU 기반 HPA만으로는 스파이크 대응 불가 (Scale-up 지연 2분)
   - 해결: Kafka Lag 기반 예측적 스케일링 추가

2. **모든 요청 비동기 처리**:
   - 적립도 비동기로 전환 시도 → 복잡도만 증가, 성능 개선 미미
   - 결정: 적립은 동기 + 배치 처리가 더 효율적

3. **단일 Kafka Topic 사용**:
   - 우선순위 구분 없이 단일 Topic 사용 → VIP 사용자도 지연 발생
   - 해결: Priority Queue 도입으로 SLA 차별화

### 알림 규칙 (Alert Rules)
```yaml
# 스파이크 감지 알림
- alert: ChargeTrafficSpike
  expr: rate(http_requests_total{service="point-charge"}[1m]) > 50000
  for: 30s
  annotations:
    summary: "충전 트래픽 스파이크 감지: {{ $value }} TPS"

# Consumer Lag 알림
- alert: KafkaConsumerLagHigh
  expr: kafka_consumer_lag{topic="charge-requests"} > 10000
  for: 1m
  annotations:
    summary: "Kafka Consumer Lag 높음: {{ $value }}"

# DB Connection 부족 알림
- alert: DBConnectionPoolExhausted
  expr: db_connection_pool_usage > 0.9
  for: 5m
  annotations:
    summary: "DB Connection Pool 90% 초과"
```
