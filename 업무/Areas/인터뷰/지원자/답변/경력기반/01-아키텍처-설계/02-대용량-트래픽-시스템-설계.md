# 분당 10만 건 이상의 거래를 처리하는 시스템 설계 시 가장 중요하게 고려한 점은?

## 답변

선불통합관리서비스는 **분당 최대 10만 건(초당 약 1,667건)**의 적립 거래를 처리해야 하는 요구사항이 있었습니다. 가장 중요하게 고려한 점은 **병목 지점 제거**와 **수평적 확장 가능한 구조 설계**였습니다.

특히 금융 시스템 특성상 단순히 성능만 좋은 것이 아니라 **정합성 보장**과 **장애 허용**이 함께 고려되어야 했기 때문에, 성능-안정성-정합성의 균형점을 찾는 것이 핵심이었습니다.

## 핵심 키워드

- 처리량 (Throughput): 1,667 TPS
- 응답 시간 (Response Time): p95 < 500ms
- 확장성 (Horizontal Scalability)
- 병목 지점 분석 및 제거
- 비동기 처리 & 이벤트 기반 아키텍처

## 주요 고려사항

### 1. 아키텍처 설계

#### 비동기 처리 패턴
```
[API Gateway] → [Message Queue] → [Worker Pool] → [Database]
     ↓                                    ↓
[즉시 응답]                          [실제 처리]
```

**설계 결정:**
- 동기 처리: 충전 거래 (즉시 응답 필요, 금액 중요)
- 비동기 처리: 적립 거래 (지연 허용 가능, 대용량)

**적립 거래 처리 Flow:**
```kotlin
// 1. API에서 즉시 수신 확인 후 큐에 적재
@PostMapping("/points/earn")
fun earnPoints(@RequestBody request: EarnRequest): ResponseEntity<EarnResponse> {
    val messageId = messageQueue.publish(request)
    return ResponseEntity.accepted()
        .body(EarnResponse(messageId, "ACCEPTED"))
}

// 2. Worker가 큐에서 꺼내 처리
@Consumer("point-earn-queue")
fun processEarnPoint(message: EarnMessage) {
    pointService.earnPoint(message)
}
```

#### 수평적 확장 구조
- Stateless Application 설계
- Session을 DB 또는 Redis에 외부화
- 로드 밸런서를 통한 트래픽 분산

### 2. 데이터베이스 설계

#### 파티셔닝 전략
```sql
-- 거래 테이블을 일자별로 파티셔닝
CREATE TABLE transaction (
    txn_id VARCHAR(50),
    txn_date DATE,
    amount NUMBER,
    ...
) PARTITION BY RANGE (txn_date) (
    PARTITION txn_202401 VALUES LESS THAN ('2024-02-01'),
    PARTITION txn_202402 VALUES LESS THAN ('2024-03-01'),
    ...
);
```

**효과:**
- 단일 파티션 스캔으로 조회 성능 80% 향상
- 오래된 파티션 Drop으로 관리 효율성 증가

#### Connection Pool 최적화
```yaml
# HikariCP 설정
spring:
  datasource:
    hikari:
      maximum-pool-size: 50  # CPU * 2 + Disk 수
      minimum-idle: 20
      connection-timeout: 10000
      idle-timeout: 600000
      max-lifetime: 1800000
```

**근거:**
- 최적 Pool Size = (CPU 코어 수 × 2) + 디스크 수
- 서버당 16 Core → Pool Size 50

#### 인덱스 전략
```sql
-- 복합 인덱스 설계
CREATE INDEX idx_txn_user_date
ON transaction(user_id, txn_date, status);

-- 커버링 인덱스로 테이블 액세스 최소화
CREATE INDEX idx_txn_covering
ON transaction(txn_date, user_id)
INCLUDE (amount, status);
```

### 3. 캐싱 전략

#### 다층 캐시 구조
```
L1 Cache (Local - Caffeine)
    ↓ (Cache Miss)
L2 Cache (Distributed - Redis)
    ↓ (Cache Miss)
Database
```

**캐싱 대상:**
- 사용자 기본 정보: TTL 1시간
- 상품 정보: TTL 6시간
- 포인트 적립률: TTL 24시간

```kotlin
@Cacheable(
    cacheNames = ["user-info"],
    key = "#userId",
    cacheManager = "redisCacheManager"
)
fun getUserInfo(userId: String): UserInfo {
    return userRepository.findById(userId)
}
```

#### Redis 클러스터 구성
- Master-Slave 구조 (1 Master + 2 Slave)
- 읽기는 Slave로 분산
- Sentinel을 통한 자동 Failover

## 성능 테스트 결과

### 부하 테스트 시나리오
- 도구: nGrinder
- 시나리오: 분당 10만 건 적립 거래 + 1만 건 충전 거래

### 결과

| 메트릭 | 목표 | 실제 결과 |
|--------|------|-----------|
| TPS | 1,800 | 2,100 |
| 평균 응답시간 | < 300ms | 245ms |
| P95 응답시간 | < 500ms | 420ms |
| P99 응답시간 | < 1s | 780ms |
| Error Rate | < 0.1% | 0.03% |
| CPU 사용률 | < 70% | 58% |
| Memory 사용률 | < 80% | 65% |

### 병목 지점 분석 및 해결

**Before (최초 설계):**
- TPS: 800
- P95 응답시간: 1.2s
- 병목: DB Connection Pool 고갈

**개선 작업:**
1. Connection Pool 크기 조정 (20 → 50)
2. Read 쿼리 Redis 캐싱
3. Batch Insert 활용 (건별 Insert → 100건씩 묶음)

**After:**
- TPS: 2,100
- P95 응답시간: 420ms
- **성능 2.6배 향상**

## 추가 최적화

### 1. Batch Insert 최적화
```kotlin
// Before: 건별 Insert (N번 DB 호출)
transactions.forEach {
    transactionRepository.save(it)
}

// After: Batch Insert (1번 DB 호출)
transactionRepository.saveAll(transactions)
```

**효과:** DB 호출 100배 감소

### 2. 비동기 로깅
```kotlin
// Async Logger 설정
@Async("loggingExecutor")
fun logTransaction(transaction: Transaction) {
    logger.info("Transaction processed: {}", transaction)
}
```

**효과:** 로깅으로 인한 지연 시간 95% 감소

### 3. Database Partitioning
- Range Partitioning으로 월별 데이터 분리
- 조회 성능 3배 향상
