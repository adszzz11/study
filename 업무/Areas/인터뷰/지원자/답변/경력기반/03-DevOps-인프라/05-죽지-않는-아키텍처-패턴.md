# '죽지 않는 아키텍처'를 위해 어떤 패턴들을 적용했나요?

## 답변

금융 시스템은 **절대 멈추지 않아야 하는 시스템**입니다. 거래 시간 동안 1분의 장애도 수십억 원의 손실과 신뢰 하락으로 이어질 수 있기 때문에, 아키텍처 설계 단계부터 장애를 전제로 한 Resilience(회복 탄력성) 패턴들을 적용했습니다.

'죽지 않는 아키텍처'를 위해 다음 원칙을 따랐습니다:
1. **Fail Fast**: 장애는 빨리 감지하고 빠르게 실패하라
2. **Graceful Degradation**: 전체 시스템이 죽는 것보다 일부 기능만 제한적으로 동작하는 것이 낫다
3. **Isolation**: 한 서비스의 장애가 다른 서비스로 전파되지 않도록 격리
4. **Redundancy**: 단일 장애점(SPOF)을 제거하고 중복성 확보

이를 위해 Circuit Breaker, Retry/Timeout, Bulkhead, Health Check, Rate Limiting 등의 패턴을 Spring Cloud, Resilience4j, Kubernetes 등을 활용하여 구현했습니다.

## 핵심 키워드

- 고가용성 (High Availability)
- 장애 허용 (Fault Tolerance)
- 회복 탄력성 (Resilience)
- Circuit Breaker
- Graceful Degradation

## 적용한 패턴들

### 1. Circuit Breaker 패턴

#### 개념
외부 서비스 호출이 실패하면 즉시 차단하여 **연쇄 장애(Cascading Failure)를 방지**하는 패턴

#### 상태 전이
```
CLOSED (정상) → OPEN (차단) → HALF_OPEN (테스트) → CLOSED (복구)
```

- **CLOSED**: 모든 요청 정상 처리
- **OPEN**: 실패율이 임계값 초과 시 모든 요청 즉시 차단 (Fallback 실행)
- **HALF_OPEN**: 일정 시간 후 일부 요청만 허용하여 복구 여부 확인

#### 구현 (Resilience4j)
```java
// build.gradle
dependencies {
    implementation 'io.github.resilience4j:resilience4j-spring-boot3:2.1.0'
    implementation 'org.springframework.boot:spring-boot-starter-aop'
}

// application.yml
resilience4j:
  circuitbreaker:
    configs:
      default:
        registerHealthIndicator: true
        slidingWindowSize: 100  # 최근 100개 요청 기준
        minimumNumberOfCalls: 10  # 최소 10개 요청 후 판단
        failureRateThreshold: 50  # 실패율 50% 초과 시 OPEN
        waitDurationInOpenState: 60000  # OPEN 상태 60초 유지
        permittedNumberOfCallsInHalfOpenState: 5  # HALF_OPEN에서 5개 테스트
        slowCallRateThreshold: 50  # 느린 호출 50% 초과 시 OPEN
        slowCallDurationThreshold: 2000  # 2초 이상 응답을 느린 호출로 간주
    instances:
      externalPaymentService:
        baseConfig: default
        failureRateThreshold: 60  # 결제 서비스는 60%까지 허용

// Service Layer
@Service
@Slf4j
public class PaymentService {

    @Autowired
    private PaymentClient paymentClient;

    @CircuitBreaker(name = "externalPaymentService", fallbackMethod = "paymentFallback")
    @TimeLimiter(name = "externalPaymentService")
    public CompletableFuture<PaymentResponse> processPayment(PaymentRequest request) {
        log.info("결제 요청: {}", request);
        return CompletableFuture.supplyAsync(() -> paymentClient.pay(request));
    }

    // Fallback: 서킷 오픈 시 대체 로직
    private CompletableFuture<PaymentResponse> paymentFallback(PaymentRequest request, Exception ex) {
        log.error("결제 서비스 Circuit OPEN, 캐시 조회: {}", ex.getMessage());

        // 1. 캐시에서 이전 결제 정보 조회
        PaymentResponse cached = paymentCache.get(request.getTransactionId());
        if (cached != null && cached.getStatus() == COMPLETED) {
            return CompletableFuture.completedFuture(cached);
        }

        // 2. 대기열에 적재 (나중에 재시도)
        paymentQueue.enqueue(request);

        // 3. 임시 응답 반환
        return CompletableFuture.completedFuture(
            PaymentResponse.builder()
                .status(PENDING)
                .message("결제가 대기열에 등록되었습니다. 잠시 후 다시 확인해주세요.")
                .build()
        );
    }
}
```

#### 효과
- 외부 결제 시스템 장애 시 **전체 시스템 다운 방지**
- 평균 장애 복구 시간(MTTR): 10분 → 1분 (90% 단축)

### 2. Retry & Timeout 패턴

#### 개념
일시적 장애(Transient Failure)를 자동 재시도로 극복하되, 무한 대기 방지를 위한 타임아웃 설정

#### 구현
```java
// application.yml
resilience4j:
  retry:
    configs:
      default:
        maxAttempts: 3  # 최대 3회 재시도
        waitDuration: 1000  # 재시도 간격 1초
        retryExceptions:
          - java.net.SocketTimeoutException
          - org.springframework.web.client.ResourceAccessException
        ignoreExceptions:
          - com.example.BusinessException  # 비즈니스 예외는 재시도 안함

  timelimiter:
    configs:
      default:
        timeoutDuration: 3s  # 3초 타임아웃
        cancelRunningFuture: true  # 타임아웃 시 실행 중인 Future 취소

// Service Layer
@Service
public class ExternalApiService {

    @Retry(name = "externalApi", fallbackMethod = "apiFallback")
    @TimeLimiter(name = "externalApi")
    public CompletableFuture<String> callExternalApi(String endpoint) {
        return CompletableFuture.supplyAsync(() -> {
            log.info("외부 API 호출: {}", endpoint);
            return restTemplate.getForObject(endpoint, String.class);
        });
    }

    private CompletableFuture<String> apiFallback(String endpoint, Exception ex) {
        log.error("재시도 실패, 캐시 데이터 반환: {}", ex.getMessage());
        String cached = redisTemplate.opsForValue().get(endpoint);
        return CompletableFuture.completedFuture(
            cached != null ? cached : "SERVICE_UNAVAILABLE"
        );
    }
}
```

#### Exponential Backoff (지수 백오프)
```java
// 재시도 간격을 점진적으로 증가 (1초 → 2초 → 4초)
resilience4j:
  retry:
    configs:
      exponential:
        maxAttempts: 5
        waitDuration: 1000
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
        enableRandomizedWait: true  # 랜덤 지연 추가 (Thundering Herd 방지)
        randomizedWaitFactor: 0.5
```

### 3. Bulkhead 패턴

#### 개념
배의 격벽처럼 **리소스를 격리하여 한 서비스의 장애가 전체 시스템으로 전파되지 않도록 방지**

#### 구현 (Thread Pool Isolation)
```java
// application.yml
resilience4j:
  bulkhead:
    configs:
      default:
        maxConcurrentCalls: 10  # 동시 호출 최대 10개
        maxWaitDuration: 0  # 대기 없이 즉시 실패
    instances:
      externalApi:
        maxConcurrentCalls: 5  # 외부 API는 5개만 허용
      internalService:
        maxConcurrentCalls: 20  # 내부 서비스는 20개 허용

  thread-pool-bulkhead:
    configs:
      default:
        coreThreadPoolSize: 5
        maxThreadPoolSize: 10
        queueCapacity: 100
        keepAliveDuration: 20ms

// Service Layer
@Service
public class OrderService {

    @Bulkhead(name = "externalApi", type = Bulkhead.Type.THREADPOOL)
    public CompletableFuture<String> getInventory(String productId) {
        // 외부 재고 조회 API (스레드 풀 격리)
        return CompletableFuture.supplyAsync(() ->
            inventoryClient.getStock(productId)
        );
    }

    @Bulkhead(name = "internalService", type = Bulkhead.Type.SEMAPHORE)
    public Order createOrder(OrderRequest request) {
        // 내부 주문 생성 (세마포어 격리, 가벼운 작업)
        return orderRepository.save(request.toEntity());
    }
}
```

#### 효과
- 외부 API가 느려져도 **내부 서비스는 정상 동작**
- Thread Pool 고갈로 인한 전체 시스템 멈춤 방지

### 4. Health Check 패턴

#### 개념
서비스의 **상태를 주기적으로 확인**하여 비정상 인스턴스를 로드밸런서에서 제거

#### 구현 (Spring Boot Actuator + Kubernetes)
```java
// build.gradle
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-actuator'
}

// application.yml
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
  endpoint:
    health:
      probes:
        enabled: true  # Kubernetes Probes 활성화
      show-details: always
  health:
    livenessState:
      enabled: true
    readinessState:
      enabled: true

// Custom Health Indicator
@Component
public class DatabaseHealthIndicator implements HealthIndicator {

    @Autowired
    private DataSource dataSource;

    @Override
    public Health health() {
        try (Connection conn = dataSource.getConnection()) {
            if (conn.isValid(1)) {
                return Health.up()
                    .withDetail("database", "PostgreSQL")
                    .withDetail("status", "Connected")
                    .build();
            }
        } catch (Exception e) {
            return Health.down()
                .withDetail("error", e.getMessage())
                .build();
        }
        return Health.down().build();
    }
}

@Component
public class ExternalApiHealthIndicator implements HealthIndicator {

    @Autowired
    private PaymentClient paymentClient;

    @Override
    public Health health() {
        try {
            HealthResponse response = paymentClient.checkHealth();
            return response.isOk()
                ? Health.up().withDetail("payment-api", "Available").build()
                : Health.down().withDetail("payment-api", "Unavailable").build();
        } catch (Exception e) {
            return Health.down().withDetail("error", e.getMessage()).build();
        }
    }
}
```

#### Kubernetes Probes 설정
```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: trading-api
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: app
        image: trading-api:1.0.0
        ports:
        - containerPort: 8080

        # Liveness Probe: 살아있는지 확인 (실패 시 재시작)
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 60  # 앱 시작 후 60초 대기
          periodSeconds: 10
          timeoutSeconds: 3
          failureThreshold: 3  # 3회 연속 실패 시 재시작

        # Readiness Probe: 트래픽 받을 준비 확인 (실패 시 트래픽 제거)
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 5
          timeoutSeconds: 2
          failureThreshold: 2

        # Startup Probe: 초기 시작 확인 (느린 시작 앱용)
        startupProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 0
          periodSeconds: 10
          failureThreshold: 30  # 최대 300초(5분) 대기
```

#### Graceful Shutdown
```java
// application.yml
server:
  shutdown: graceful  # 우아한 종료 활성화

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 종료 시 30초 대기

// Shutdown Hook
@Component
@Slf4j
public class GracefulShutdownHandler implements ApplicationListener<ContextClosedEvent> {

    @Override
    public void onApplicationEvent(ContextClosedEvent event) {
        log.info("애플리케이션 종료 시작, 진행 중인 요청 완료 대기...");
        // 진행 중인 트랜잭션 완료 대기
        // 커넥션 풀 종료
        // 캐시 플러시 등
    }
}
```

### 5. 무중단 배포

#### Rolling Update (Kubernetes)
```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: trading-api
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1  # 최대 1개 추가 Pod 생성 가능
      maxUnavailable: 1  # 최대 1개 Pod 중단 허용

  # 결과: 4 → 5 → 4 → 5 → 4 (점진적 교체)
```

#### Blue-Green Deployment
```bash
# 1. Green 버전 배포 (트래픽 받지 않음)
kubectl apply -f deployment-green.yaml

# 2. Green 버전 헬스체크 확인
kubectl wait --for=condition=available deployment/trading-api-green --timeout=300s

# 3. Service 트래픽 전환 (Blue → Green)
kubectl patch service trading-api -p '{"spec":{"selector":{"version":"green"}}}'

# 4. Blue 버전 제거 (또는 롤백 대비 유지)
kubectl delete deployment trading-api-blue
```

#### Canary Deployment (Istio)
```yaml
# VirtualService: 트래픽 10%만 신규 버전으로
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: trading-api
spec:
  hosts:
  - trading-api
  http:
  - match:
    - headers:
        canary:
          exact: "true"
    route:
    - destination:
        host: trading-api
        subset: v2
  - route:
    - destination:
        host: trading-api
        subset: v1
      weight: 90
    - destination:
        host: trading-api
        subset: v2
      weight: 10  # 10%만 신규 버전으로
```

## 구체적인 구현 사례

### 사례 1: 외부 결제 시스템 장애 대응

#### 상황
- 외부 PG(Payment Gateway) 서비스가 DDoS 공격으로 30분간 응답 불가
- 기존 시스템: 모든 결제 요청 타임아웃으로 전체 시스템 멈춤

#### 적용 패턴
- **Circuit Breaker**: 5회 연속 실패 시 자동 차단
- **Fallback**: 결제 요청을 Redis Queue에 적재 후 임시 응답 반환
- **Retry**: PG 복구 후 큐에 쌓인 요청 자동 재시도

#### 결과
- 외부 PG 장애 시에도 **거래 시스템 정상 동작**
- 사용자는 "결제가 처리 중입니다" 메시지를 받고 나중에 확인 가능
- 장애 복구 후 큐에 쌓인 20,000건 자동 처리 (평균 2분)

### 사례 2: 데이터베이스 슬로우 쿼리 전파 방지

#### 상황
- 특정 복잡한 쿼리가 5초 이상 소요되며 커넥션 풀 고갈
- 기존: 모든 API 요청이 대기하며 시스템 전체 응답 불가

#### 적용 패턴
- **Bulkhead**: 복잡한 쿼리는 별도 스레드 풀 격리 (최대 5개)
- **Timeout**: 쿼리 타임아웃 3초 설정
- **Cache**: 자주 조회되는 데이터는 Redis 캐시 (TTL 5분)

#### 결과
- 슬로우 쿼리가 발생해도 **일반 거래 API는 정상 동작**
- 평균 응답 시간: 200ms 유지 (기존 5초 → 200ms, 96% 개선)

### 사례 3: Pod 재시작 시 트래픽 손실 방지

#### 상황
- Kubernetes Pod 재시작 시 일부 요청이 실패 (502 Bad Gateway)
- 원인: Readiness Probe 체크 전에 트래픽 유입

#### 적용 패턴
- **Readiness Probe**: 애플리케이션 완전 시작 후 트래픽 수신
- **Graceful Shutdown**: 종료 시 30초간 진행 중인 요청 완료 대기
- **PreStop Hook**: Pod 종료 전 로드밸런서에서 먼저 제거

```yaml
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 15"]  # 15초 대기 후 종료
```

#### 결과
- 무중단 배포 성공률: 95% → 99.9%
- 배포 중 에러율: 2% → 0.01%

## 장애 대응 프로세스

### 1. 장애 감지 (Detection)
- **Prometheus**: 메트릭 기반 이상 탐지 (응답 시간, 에러율, CPU)
- **Loki**: 에러 로그 실시간 모니터링
- **Alert Manager**: 임계값 초과 시 Slack/PagerDuty 알림

### 2. 자동 복구 (Auto-Healing)
- **Kubernetes**: Liveness Probe 실패 시 Pod 자동 재시작
- **Circuit Breaker**: 장애 서비스 자동 차단 및 Fallback
- **HPA**: 트래픽 증가 시 자동 스케일아웃

### 3. 수동 개입 (Manual Intervention)
- **Runbook 참조**: 장애 유형별 대응 절차 문서화
- **Blue-Green 롤백**: 신규 버전 문제 시 즉시 이전 버전 복구
- **Database Failover**: Master DB 장애 시 Replica로 승격

### 4. 사후 분석 (Post-Mortem)
- 장애 원인 분석 (Root Cause Analysis)
- 재발 방지 대책 수립
- 문서화 및 팀 공유

## 모니터링 및 알림

### Prometheus Alert Rules
```yaml
# prometheus-rules.yml
groups:
  - name: trading-api-alerts
    interval: 30s
    rules:
      # 에러율 5% 초과
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "높은 에러율 감지: {{ $value }}"
          description: "5분간 에러율이 5%를 초과했습니다."

      # 응답 시간 1초 초과
      - alert: HighLatency
        expr: histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m])) > 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "응답 시간 지연: {{ $value }}s"

      # Circuit Breaker OPEN
      - alert: CircuitBreakerOpen
        expr: resilience4j_circuitbreaker_state{state="open"} == 1
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Circuit Breaker OPEN: {{ $labels.name }}"
```
