빌링,정산 도메인은 규모가 커질수록 복잡도가 기하급수적으로 증가합니다.
저는 이러한 환경에서 운영 안정성과 데이터 정합성을 최우선으로 시스템을 설계해 온 백엔드 엔지니어입니다.

거래량 증가에 따라 반복적으로 발생하는 정산 오류, 성능 저하, 운영 리스크를
구조적 개선과 자동화를 통해 해결해 왔습니다.

저는 업무에서 직면하는 문제를 단일 기능 단위가 아닌, 인프라,데이터 흐름,운영 시나리오 전체 관점에서 바라봅니다.

비즈니스 로직과 데이터 모델이 충분히 정리되지 않으면,
유지보수 비용과 장애 가능성이 급격히 증가한다는 것을 여러 차례 경험했습니다.

반복 처리 영역은 Batch 기반 자동화를 통해 사람의 개입 없이도 재처리와 검증이 가능한 운영 구조를 구축해 왔습니다.

저의 설계 철학은 "변경과 장애를 전제로 한 구조”입니다.
운영 중 발생할 수 있는 리스크를 사전에 가정하고,
문제가 발생하더라도 빠르게 원인을 추적하고 복구할 수 있는 구조를 만드는 데 강점이 있습니다.
초기 설계 단계에서 고민이 많아 시간이 더 소요되는 편이지만,
이는 높은 품질의 소프트웨어를 만들기 위해 반드시 필요한 과정이라고 생각합니다.
실제로 이러한 접근은 팀 전체의 개발,운영 효율을 높이는 결과로 이어져왔습니다.

네이버 클라우드의 빌링,정산 플랫폼은
다양한 상품과 정책, 대규모 데이터와 트래픽을
높은 신뢰도로 처리해야 하는 핵심 영역입니다.
이는 제가 지속적으로 마주해 온 문제들과 깊게 맞닿아 있습니다.

# [필수] 지원 직무와 관련된 자신의 대표적인 프로젝트 한 가지를 적어 주시되, 어떠한 고민을 가지고 어떻게 결과물을 만들어 냈는지, 기여도는 어떠했는지 구체적으로 설명해 주세요

5년 이상 Stored Procedure 중심으로 운영되던 레거시 포인트 시스템을 사내에서 개발한 선불 통합관리시스템으로 마이그레이션하는 프로젝트에 참여해,
원장 이관 배치 설계와 중계 미들웨어 개발을 주도적으로 담당했습니다.
프로젝트의 핵심 과제는 대규모 유저 데이터를 대상으로 서비스 중단 없이 원장을 이관하면서 정산 정합성을 유지하는 것이었습니다.

기존 시스템은 정산 로직이 Stored Procedure에 집중되어 있어 데이터 흐름을 추적하기 어려웠고,
대량 데이터 처리 시 성능 병목과 정합성 이슈가 발생할 가능성이 컸습니다.
또한 신규 시스템의 안정성이 충분히 검증되기 전까지는 언제든지 레거시 시스템으로 복귀할 수 있는 구조가 필요했습니다.
이관 이후에도 레거시 시스템에는 아카이빙 목적의 데이터 적재가 지속되어야 했습니다.

저는 단순한 데이터 이전이 아닌, 정합성과 복구 가능성을 전제로 한 이관 구조가 우선되어야 한다고 판단했습니다.

신규,레거시 원장을 동시에 비교,검증할 수 있도록 중간 원장 테이블과 이관 결과 관리 테이블을 직접 설계했고,
이관 단위를 유저 기준으로 분리해 쿼리 범위를 명확히 통제했습니다.
그 결과 대사 불일치가 발생하더라도 특정 유저 단위로 재처리가 가능하도록 구조를 설계할 수 있었습니다.

원장 이관은 Spring Batch 기반으로 단계적으로 수행했으며, 유저 접속이 가장 적은 시간대에 재이관 배치를 실행해 서비스 영향도를 최소화했습니다.
또한 중계 미들웨어를 구현해 빌링 서버가 유저 이관 상태에 따라 신규 또는 레거시 시스템을 선택적으로 호출하도록 했고,
2노드 이중화 구성으로 단일 장애 지점을 줄였습니다.

이 경험을 통해 저는 대규모 정산 데이터 이관에서는 데이터 흐름을 명확히 통제하고, 언제든 재현,복구 가능한 구조를 만드는 것이 핵심임을 체감했습니다.
이러한 설계,구현 경험은 클라우드 빌링,정산 플랫폼에서도 안정성과 확장성을 확보하는 데 직접적으로 기여할 수 있다고 생각합니다.

# [필수] 팀을 이뤄 협업을 하면서 성과를 이뤘던 경험을 적어주세요. 자신이 어떤 역할로 팀에 기여했고 어떤 교훈을 얻었는지를 포함해 주세요

가장 기억에 남는 경험은 다모음 백엔드 리아키텍팅과 팀 내 개발 프로세스 셋업을 병행했던 프로젝트입니다.
당시 다모음 팀은 신생 팀에 가까웠고, 구성원 대부분이 신규 입사자였던 반면 백엔드는 Python 2.7과 MySQL Stored Procedure 기반의 레거시 코드가 방치된 상태였습니다.
개인 단위 R&R 중심의 문화로 인해 DB 구조와 결제,정산 도메인 지식이 팀에 축적되지 않는 문제가 지속되고 있었습니다.

저는 백엔드를 Spring Boot와 Kotlin 기반으로 리아키텍팅하는 과제를 맡았고,
단순한 기술 전환이 아니라 결제,정산 웹 서비스의 유지보수성과 확장성을 회복하는 것을 목표로 삼았습니다.
“누가 와도 빠르게 온보딩할 수 있을 것”, “관리 포인트와 데이터 흐름을 명확히 규격화할 것”을 기준으로 삼아,
기존 팀원들과 인터뷰를 진행하며 결제 수단별 처리 흐름과 정산 로직의 히스토리를 정리해 문서화했습니다.
이를 바탕으로 결제 수단별 로직을 도메인 모델로 분리하고, Kotlin의 sealed class를 활용해 타입 안전하게 관리할 수 있도록 설계했습니다.

또한 POST 위주로 구성되어 있던 REST API를 RESTful 원칙에 맞게 재정비하고, 응답 코드를 서비스 코드 기반으로 체계화해 확장성을 확보했습니다.
공통 기준 정보는 코드 테이블로 관리해 DB 모델링을 정리했고, 약관 관리 구조를 개선해 이후 특정 약관 동의 유저를 대상으로 한 정산,정책 로직 변경이 수월해지도록 했습니다.

기술 개선과 함께 코드 리뷰 룰을 정립하고 SonarQube, GitHub–Jira 연동 워크플로우를 도입해 개발 품질과 작업 흐름을 가시화했습니다.
이 경험을 통해 저는 정리된 데이터 모델과 처리 구조, 명확한 개발 프로세스가 팀 전체의 협업 효율과 서비스 안정성을 근본적으로 끌어올린다는 것을 체감했습니다.
이러한 경험은 복잡한 정산 로직과 배치 처리가 요구되는 Cloud Billing 시스템 개발에서도 중요한 자산이 될 것이라 생각합니다.
