# 선불통합관리서비스에서 헥사고날 아키텍처를 선택한 구체적인 이유는? 다른 아키텍처 패턴과 비교했을 때 어떤 장점이 있었나요?

## 답변

선불통합관리서비스는 다양한 외부 시스템(BC카드, 정산 시스템, 대사 시스템 등)과 연동되고, 금융 도메인의 복잡한 비즈니스 로직을 다루는 시스템입니다. 헥사고날 아키텍처를 선택한 가장 큰 이유는 **도메인 로직과 외부 의존성의 완전한 분리**를 통해 변경에 유연하게 대응하고 테스트 가능성을 극대화하기 위함이었습니다.

특히 금융 시스템 특성상 규제 변경이 잦고, 외부 연동 시스템의 스펙 변경이 빈번하게 발생하는데, 헥사고날 아키텍처를 통해 도메인 핵심 로직은 보호하면서 어댑터 계층만 수정하여 변경 영향도를 최소화할 수 있었습니다.

## 핵심 키워드

- 헥사고날 아키텍처 (Ports and Adapters)
- 도메인 중심 설계 (Domain-Driven Design)
- 의존성 역전 원칙 (Dependency Inversion)
- 테스트 용이성
- 외부 시스템 연동 격리

## 비교 분석

### vs 레이어드 아키텍처

**레이어드 아키텍처의 한계:**
- Controller → Service → Repository 의 단방향 의존성으로 인해 도메인 로직이 인프라 기술에 종속됨
- 데이터베이스 스키마 변경 시 도메인 객체까지 영향을 받음
- 외부 API 변경 시 서비스 레이어까지 수정 필요

**헥사고날 아키텍처의 장점:**
- 도메인 로직이 외부 인프라(DB, API, MQ 등)에 완전히 독립적
- Port(인터페이스)를 통한 의존성 역전으로 변경 영향도 최소화
- 각 어댑터를 독립적으로 개발 및 테스트 가능

### vs 클린 아키텍처

**유사점:**
- 도메인 중심 설계 철학 공유
- 의존성 방향이 항상 내부(도메인)를 향함
- 테스트 용이성 강조

**헥사고날 아키텍처 선택 이유:**
- 클린 아키텍처보다 구조가 단순하여 팀원들의 학습 곡선이 낮음
- 금융 시스템에서 중요한 "외부 연동"이라는 관점이 명확히 드러남
- Use Case 계층 없이 Port-Adapter로 직관적인 구조 표현 가능

## 실제 적용 사례

### 구조

```
Domain (Core)
├── Entity (선불거래, 잔액, 포인트)
├── Value Object (거래금액, 거래상태)
└── Port (인터페이스)
    ├── Inbound Port (Use Case)
    └── Outbound Port (Repository, External API)

Adapter
├── Inbound Adapter
│   ├── REST Controller
│   └── Batch Job Handler
└── Outbound Adapter
    ├── JPA Repository Adapter
    ├── BC카드 API Adapter
    └── 정산 시스템 Adapter
```

### 코드 예시

```kotlin
// Domain - Inbound Port (Use Case)
interface ChargePointUseCase {
    fun charge(request: ChargeRequest): ChargeResult
}

// Domain - Outbound Port
interface PaymentGatewayPort {
    fun requestPayment(payment: Payment): PaymentResponse
}

// Domain - Use Case Implementation
class ChargePointService(
    private val paymentGatewayPort: PaymentGatewayPort,
    private val pointRepository: PointRepository
) : ChargePointUseCase {
    override fun charge(request: ChargeRequest): ChargeResult {
        // 순수한 도메인 로직만 존재
        val payment = Payment.create(request)
        val response = paymentGatewayPort.requestPayment(payment)
        return ChargeResult.from(response)
    }
}

// Adapter - BC카드 Adapter
class BcCardAdapter : PaymentGatewayPort {
    override fun requestPayment(payment: Payment): PaymentResponse {
        // BC카드 전문 통신 로직
        // 도메인은 이 구현체를 알지 못함
    }
}
```

### 실제 효과 사례

1. **BC카드 전문 포맷 변경 대응 (2시간 소요)**
   - 도메인 로직 수정 없이 BcCardAdapter만 수정
   - 테스트: 어댑터 단위 테스트만 실행하여 빠른 검증

2. **PostgreSQL → Oracle 마이그레이션 (1주일 소요)**
   - Repository Port는 그대로 유지
   - JpaRepositoryAdapter만 OracleRepositoryAdapter로 교체
   - 도메인 로직 변경 없이 마이그레이션 완료

3. **테스트 커버리지 92% 달성**
   - 도메인 로직은 Mock Port를 활용하여 순수하게 테스트
   - 각 어댑터는 독립적으로 통합 테스트

## 얻은 효과

### 1. 변경 영향도 최소화
- 외부 시스템 변경 시 해당 어댑터만 수정 (평균 80% 작업 시간 단축)
- 도메인 로직 안정성 확보

### 2. 테스트 용이성
- 도메인 로직 단위 테스트 작성 시간 50% 감소
- Mock 객체 활용으로 외부 의존성 제거
- Test Coverage 92% 달성

### 3. 기술 스택 교체 유연성
- DB 마이그레이션 3회 수행 (MSSQL → PostgreSQL → Oracle)
- 각 마이그레이션 시 도메인 로직 무변경

### 4. 병렬 개발 가능
- 도메인 팀과 인프라 팀이 Port 인터페이스 기준으로 독립적 개발
- 개발 속도 30% 향상

### 5. 신규 기능 추가 용이
- 새로운 결제 수단 추가 시 새로운 어댑터만 구현
- 기존 코드 영향 없이 확장 가능 (OCP 준수)
