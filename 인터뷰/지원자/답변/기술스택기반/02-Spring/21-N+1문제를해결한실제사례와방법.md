# N+1 문제를 해결한 실제 사례와 방법은?

## 답변

N+1 문제는 JPA를 사용하면서 가장 자주 마주치는 성능 이슈입니다. 저는 선불통합관리서비스에서 **사용자별 거래 내역 조회 API**를 개발하면서 이 문제를 직접 경험하고 해결했습니다.

초기에는 1명의 사용자 조회에 1개의 쿼리, 100건의 거래 조회에 100개의 쿼리가 발생하여 **총 101개의 쿼리**가 실행되었습니다. 이를 **Fetch Join과 BatchSize 조합**으로 해결하여 **2~3개의 쿼리**로 최적화했고, 응답 시간을 **5초에서 0.3초로 개선**했습니다.

## 핵심 키워드

- N+1 문제 (N+1 Query Problem)
- Fetch Join
- @BatchSize / default_batch_fetch_size
- @EntityGraph
- DTO Projection
- Lazy Loading vs Eager Loading

## N+1 문제 원인

### 발생 시나리오
```kotlin
// Entity 구조
@Entity
class User(
    @Id val id: String,
    val name: String,

    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    val transactions: List<Transaction> = emptyList()
)

@Entity
class Transaction(
    @Id val id: String,
    val amount: BigDecimal,

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    val user: User
)

// 문제가 있는 코드
fun getUsers(): List<User> {
    val users = userRepository.findAll()  // 1. User 조회: 쿼리 1개

    users.forEach { user ->
        println("${user.name}: ${user.transactions.size}") // 2. N번 쿼리 발생!
    }

    return users
}
```

### 실행되는 쿼리
```sql
-- 1. User 전체 조회 (1개)
SELECT * FROM user;

-- 2. 각 User마다 Transaction 조회 (N개)
SELECT * FROM transaction WHERE user_id = 'user1';
SELECT * FROM transaction WHERE user_id = 'user2';
SELECT * FROM transaction WHERE user_id = 'user3';
...
-- 총 1 + N개의 쿼리 실행!
```

### 성능 문제
- 100명의 User → 101개의 쿼리
- 1,000명의 User → 1,001개의 쿼리
- **쿼리 수가 데이터 건수에 비례하여 선형 증가**

## 해결 방법

### 1. Fetch Join (가장 효과적)

```kotlin
// Repository
interface UserRepository : JpaRepository<User, String> {
    @Query("SELECT u FROM User u JOIN FETCH u.transactions")
    fun findAllWithTransactions(): List<User>
}

// 실행되는 쿼리 (1개로 줄어듦)
SELECT u.*, t.*
FROM user u
INNER JOIN transaction t ON u.id = t.user_id;
```

**장점:**
- 1개의 쿼리로 모든 데이터 조회
- Lazy Loading 없이 즉시 모든 연관 엔티티 로드

**단점:**
- 카테시안 곱으로 인한 중복 데이터 전송
- 페이징 불가 (메모리에서 처리)
- 2개 이상의 컬렉션 Fetch Join 불가

### 2. @EntityGraph

```kotlin
@Entity
class User(
    @Id val id: String,
    val name: String,

    @OneToMany(mappedBy = "user")
    val transactions: List<Transaction> = emptyList()
)

// Repository
interface UserRepository : JpaRepository<User, String> {
    @EntityGraph(attributePaths = ["transactions"])
    fun findAll(): List<User>
}
```

**장점:**
- 쿼리 메서드에 애노테이션만 추가
- Fetch Join과 동일한 효과

**단점:**
- Fetch Join과 동일한 제약사항

### 3. @BatchSize (가장 실용적)

```kotlin
@Entity
class User(
    @Id val id: String,
    val name: String,

    @BatchSize(size = 100)  // IN절에 최대 100개씩
    @OneToMany(mappedBy = "user")
    val transactions: List<Transaction> = emptyList()
)

// 또는 application.yml에 글로벌 설정
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

**실행되는 쿼리:**
```sql
-- 1. User 조회
SELECT * FROM user;

-- 2. Transaction을 IN절로 한 번에 조회 (100개씩)
SELECT * FROM transaction
WHERE user_id IN ('user1', 'user2', ..., 'user100');

-- 3. 다음 100개 (만약 있다면)
SELECT * FROM transaction
WHERE user_id IN ('user101', 'user102', ..., 'user200');
```

**장점:**
- 페이징 가능
- 2개 이상의 컬렉션도 처리 가능
- 카테시안 곱 문제 없음
- 메모리 효율적

**단점:**
- Fetch Join보다는 쿼리 수가 많음 (하지만 N개 → N/100개로 감소)

### 4. DTO Projection (읽기 전용)

```kotlin
// DTO
data class UserTransactionDto(
    val userId: String,
    val userName: String,
    val transactionId: String,
    val amount: BigDecimal
)

// Repository
@Query("""
    SELECT new com.example.dto.UserTransactionDto(
        u.id, u.name, t.id, t.amount
    )
    FROM User u
    JOIN u.transactions t
""")
fun findUserTransactionDtos(): List<UserTransactionDto>
```

**장점:**
- 필요한 컬럼만 조회로 네트워크 트래픽 감소
- 엔티티 변환 오버헤드 없음

**단점:**
- 읽기 전용 (수정 불가)
- DTO 클래스 관리 필요

## 실제 사례

### Case 1: 사용자별 거래 내역 조회 API

#### 문제 상황
```kotlin
// Before: N+1 문제 발생
@GetMapping("/users/{userId}/transactions")
fun getUserTransactions(@PathVariable userId: String): UserTransactionResponse {
    val user = userRepository.findById(userId)
        .orElseThrow { NotFoundException() }

    // 여기서 N+1 발생!
    val transactions = user.transactions.map {
        TransactionDto.from(it)
    }

    return UserTransactionResponse(user, transactions)
}

// 실행 쿼리: 1(User) + N(Transactions) + N(각 Transaction의 연관 데이터)
// 평균 응답시간: 5초 (100건 기준)
```

#### 해결 방법
```kotlin
// After: Fetch Join + DTO Projection
interface UserRepository : JpaRepository<User, String> {
    @Query("""
        SELECT u
        FROM User u
        LEFT JOIN FETCH u.transactions t
        LEFT JOIN FETCH t.category
        WHERE u.id = :userId
    """)
    fun findByIdWithTransactions(@Param("userId") userId: String): Optional<User>
}

@GetMapping("/users/{userId}/transactions")
fun getUserTransactions(@PathVariable userId: String): UserTransactionResponse {
    val user = userRepository.findByIdWithTransactions(userId)
        .orElseThrow { NotFoundException() }

    val transactions = user.transactions.map {
        TransactionDto.from(it)
    }

    return UserTransactionResponse(user, transactions)
}

// 실행 쿼리: 1개
// 평균 응답시간: 0.3초 (100건 기준)
```

**개선 효과:**
- 쿼리 수: 101개 → 1개 (99% 감소)
- 응답 시간: 5초 → 0.3초 (94% 개선)

### Case 2: 페이징이 필요한 목록 조회

```kotlin
// Fetch Join은 페이징 불가 → @BatchSize 사용
@Entity
class User(
    @Id val id: String,
    val name: String,

    @BatchSize(size = 100)
    @OneToMany(mappedBy = "user")
    val transactions: List<Transaction> = emptyList()
)

// application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100

// Controller
@GetMapping("/users")
fun getUsers(@PageableDefault(size = 20) pageable: Pageable): Page<UserDto> {
    return userRepository.findAll(pageable)
        .map { user ->
            UserDto(
                id = user.id,
                name = user.name,
                transactionCount = user.transactions.size  // Lazy Loading이지만 BatchSize로 최적화
            )
        }
}
```

**실행 쿼리:**
```sql
-- 1. User 페이징 조회
SELECT * FROM user LIMIT 20 OFFSET 0;

-- 2. 20명의 User에 대한 Transaction을 IN절로 한 번에 조회
SELECT * FROM transaction
WHERE user_id IN ('user1', 'user2', ..., 'user20');
```

**결과:**
- 쿼리 수: 21개 → 2개 (90% 감소)
- 페이징 정상 동작

### Case 3: 복잡한 조회 - QueryDSL 활용

```kotlin
// 필터링 조건이 복잡한 경우
class UserQueryRepository(
    private val queryFactory: JPAQueryFactory
) {
    fun findUsersWithTransactions(
        startDate: LocalDate,
        endDate: LocalDate
    ): List<User> {
        val user = QUser.user
        val transaction = QTransaction.transaction

        return queryFactory
            .selectFrom(user)
            .distinct()
            .leftJoin(user.transactions, transaction).fetchJoin()
            .where(
                transaction.createdAt.between(startDate, endDate)
            )
            .fetch()
    }
}
```

## 성능 측정 결과

### 테스트 환경
- 데이터: User 100명, 각 User당 Transaction 평균 50건 (총 5,000건)
- DB: PostgreSQL 13
- JPA: Hibernate 5.6

### 결과 비교

| 방법 | 쿼리 수 | 응답 시간 | 메모리 사용량 |
|------|---------|-----------|---------------|
| N+1 발생 (원본) | 101개 | 5.2초 | 120MB |
| Fetch Join | 1개 | 0.3초 | 150MB (중복 데이터) |
| @BatchSize(100) | 2개 | 0.5초 | 100MB |
| DTO Projection | 1개 | 0.2초 | 80MB |

## Best Practices

### 1. 기본 전략
```yaml
# application.yml - 글로벌 BatchSize 설정
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100  # 적절한 크기 설정
```

### 2. 용도별 선택 기준

**Fetch Join 사용:**
- 1:N 관계에서 N이 적을 때
- 페이징이 필요 없을 때
- 즉시 모든 데이터가 필요할 때

**@BatchSize 사용:**
- 페이징이 필요할 때
- 1:N 관계에서 N이 많을 때
- 2개 이상의 컬렉션을 조회할 때

**DTO Projection 사용:**
- 읽기 전용 조회
- 대용량 데이터 조회
- API 응답 최적화
