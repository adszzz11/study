# 레거시 시스템을 현대화한 경험이 있나요

## English Question

**Have you modernized legacy systems? What was your approach?**

## 질문 번역

레거시 시스템을 현대화한 경험이 있나요? 접근 방식은 무엇이었나요?

---

## English Answer

Yes. At Danal, I led the containerization of our legacy payment systems. This was a multi-year effort.

**The Situation:**
- 10+ year old monolithic Java applications
- Running on bare metal servers
- Manual deployments with significant downtime
- No horizontal scaling capability
- "It works, don't touch it" mentality

**My Approach:**

**1. Understand Before Changing**
- Documented current architecture (often undocumented)
- Identified all dependencies
- Mapped traffic patterns and load profiles
- Found the actual pain points (not assumed ones)

**2. Strangler Fig Pattern**
We didn't rewrite everything at once:
- New features built in containerized services
- Gradually moved traffic to new services
- Legacy system shrank over time
- Never a "big bang" migration

**3. Prioritize by Pain**
Started with services that caused the most operational pain:
- Frequent scaling needs
- Complex deployments
- Often failing

**4. Build Confidence Incrementally**
- Extensive testing before each migration
- Feature flags to control rollout
- Easy rollback mechanisms
- Monitored closely after each change

**5. Document Everything**
- Each migration documented as a template
- Knowledge transferred to the team
- Playbooks for incidents

**Results:**
- Deployment time: hours → minutes
- Scaling: manual → automatic
- Incidents: weekly → monthly
- Team confidence: terrified → comfortable

**Key Lesson:**
Modernization is a journey, not an event. Patience and incremental progress beat heroic rewrites.

## 한글 번역

네. 다날에서 레거시 결제 시스템의 컨테이너화를 이끌었습니다. 다년간의 노력이었습니다.

**상황:**
- 10년 이상 된 모놀리식 Java 애플리케이션
- 베어메탈 서버에서 실행
- 상당한 다운타임과 함께 수동 배포
- 수평 확장 능력 없음
- "작동하니 건드리지 마" 정신

**제 접근 방식:**

**1. 변경 전에 이해**
- 현재 아키텍처 문서화 (종종 문서화 안 됨)
- 모든 의존성 식별
- 트래픽 패턴과 부하 프로필 매핑
- 실제 페인 포인트 찾기 (가정한 것이 아닌)

**2. 스트랭글러 피그 패턴**
모든 것을 한 번에 다시 쓰지 않음:
- 컨테이너화된 서비스에 새 기능 구축
- 점진적으로 새 서비스로 트래픽 이동
- 시간이 지남에 따라 레거시 시스템 축소
- "빅뱅" 마이그레이션 절대 없음

**3. 고통으로 우선순위**
가장 많은 운영 고통을 야기하는 서비스로 시작:
- 빈번한 스케일링 필요
- 복잡한 배포
- 자주 실패

**4. 신뢰를 점진적으로 구축**
- 각 마이그레이션 전 광범위한 테스트
- 롤아웃 제어를 위한 피처 플래그
- 쉬운 롤백 메커니즘
- 각 변경 후 면밀히 모니터링

**5. 모든 것 문서화**
- 각 마이그레이션 템플릿으로 문서화
- 팀에 지식 이전
- 인시던트용 플레이북

**결과:**
- 배포 시간: 시간 → 분
- 스케일링: 수동 → 자동
- 인시던트: 주간 → 월간
- 팀 신뢰: 겁에 질린 → 편안한

**핵심 교훈:**
현대화는 이벤트가 아닌 여정입니다. 인내와 점진적 진전이 영웅적 재작성을 이깁니다.
