# 프로덕션에서 문제를 어떻게 디버깅하나요

## English Question

**How do you debug issues in production?**

## 질문 번역

프로덕션에서 문제를 어떻게 디버깅하나요?

---

## English Answer

Production debugging at Danal was high-stakes—payment failures meant real money and angry merchants.

**My Process:**

**1. Triage First**
Before debugging, assess:
- What's the blast radius? (One user vs. all users)
- Is it getting worse?
- Can we mitigate without a fix? (Feature flag, rollback, restart)

**2. Gather Information**
- Check monitoring dashboards (Datadog, Grafana)
- Review recent deployments
- Look at error rates and patterns
- Check dependencies (database, external services)

**3. Form Hypotheses**
Based on symptoms:
- "Traffic spike → maybe resource exhaustion"
- "After deployment → likely code change"
- "Specific endpoint → narrowed scope"

**4. Narrow Down**
- Use logs to trace specific requests
- Correlation IDs are essential for distributed systems
- Check: when did it start? What changed?

**5. Reproduce (if possible)**
- Staging environment with production data snapshot
- Local reproduction if safe
- Sometimes read-only production queries

**6. Fix and Verify**
- Smallest possible fix
- Test in staging
- Gradual rollout (canary if possible)
- Monitor after deployment

**Tools I Use:**
- **Logs**: Structured logging, ELK stack
- **Metrics**: Prometheus, Grafana, Datadog
- **Tracing**: Jaeger for distributed traces
- **Profiling**: CPU/memory profilers when needed

**Post-Incident:**
- Write postmortem (blameless)
- Identify action items
- Update monitoring to catch similar issues

**My Philosophy:**
The goal isn't just to fix the bug—it's to make sure this class of bug is easier to catch next time.

## 한글 번역

다날에서 프로덕션 디버깅은 높은 위험이었습니다—결제 실패는 실제 돈과 화난 가맹점을 의미했습니다.

**제 프로세스:**

**1. 먼저 분류**
디버깅 전에 평가:
- 영향 범위는? (한 사용자 vs. 모든 사용자)
- 악화되고 있나?
- 수정 없이 완화할 수 있나? (피처 플래그, 롤백, 재시작)

**2. 정보 수집**
- 모니터링 대시보드 확인 (Datadog, Grafana)
- 최근 배포 검토
- 오류율과 패턴 확인
- 의존성 확인 (데이터베이스, 외부 서비스)

**3. 가설 형성**
증상에 기반:
- "트래픽 스파이크 → 아마도 리소스 고갈"
- "배포 후 → 아마도 코드 변경"
- "특정 엔드포인트 → 범위 좁힘"

**4. 좁히기**
- 로그를 사용해 특정 요청 추적
- 분산 시스템에서 상관 ID 필수
- 확인: 언제 시작됐나? 뭐가 바뀌었나?

**5. 재현 (가능하면)**
- 프로덕션 데이터 스냅샷이 있는 스테이징 환경
- 안전하면 로컬 재현
- 때로는 읽기 전용 프로덕션 쿼리

**6. 수정 및 검증**
- 가장 작은 수정
- 스테이징에서 테스트
- 점진적 롤아웃 (가능하면 카나리)
- 배포 후 모니터링

**사용하는 도구:**
- **로그**: 구조화된 로깅, ELK 스택
- **메트릭**: Prometheus, Grafana, Datadog
- **트레이싱**: 분산 추적용 Jaeger
- **프로파일링**: 필요시 CPU/메모리 프로파일러

**인시던트 후:**
- 포스트모템 작성 (비난 없이)
- 액션 아이템 식별
- 유사 문제 포착을 위해 모니터링 업데이트

**제 철학:**
목표는 버그 수정만이 아닙니다—이 종류의 버그를 다음에 더 쉽게 잡을 수 있게 하는 것입니다.
