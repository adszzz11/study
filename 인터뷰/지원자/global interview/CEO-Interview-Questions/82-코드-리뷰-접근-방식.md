# 코드 리뷰를 어떻게 하나요

## English Question

**How do you approach code reviews?**

## 질문 번역

코드 리뷰를 어떻게 접근하나요?

---

## English Answer

Code review is about more than catching bugs—it's about shared understanding and team growth.

**When I Review:**

**1. Start with the "Why"**
- Read the PR description first
- Understand the problem being solved
- Check if this is the right approach before diving into code

**2. Prioritize Feedback**
Not all comments are equal:
- **Blocking**: Security issues, data loss risks, broken functionality
- **Important**: Performance problems, maintainability concerns
- **Suggestions**: Style preferences, minor improvements
- **Nitpicks**: Label these clearly so author knows it's optional

**3. What I Look For**
- Does it solve the stated problem?
- Edge cases and error handling
- Test coverage for new functionality
- Readability and maintainability
- Security implications
- Performance at scale

**4. How I Communicate**
- Ask questions rather than demand changes
- Explain the "why" behind suggestions
- Praise good solutions (not just criticism)
- Offer alternatives, not just problems

**When I Submit PRs:**

**1. Make Review Easy**
- Small, focused PRs (under 400 lines when possible)
- Clear description with context
- Self-review before requesting review
- Highlight areas where I'm uncertain

**2. Respond Constructively**
- Thank reviewers for feedback
- Explain reasoning if I disagree
- Ask for clarification when needed

**At Danal:**
I reviewed 3-5 PRs daily. The best reviews I received were from people who understood the system context, not just the code syntax.

## 한글 번역

코드 리뷰는 버그 잡기 이상입니다—공유된 이해와 팀 성장에 관한 것입니다.

**리뷰할 때:**

**1. "왜"로 시작**
- PR 설명을 먼저 읽음
- 해결되는 문제 이해
- 코드에 들어가기 전에 올바른 접근인지 확인

**2. 피드백 우선순위**
모든 코멘트가 동일하지 않음:
- **차단**: 보안 문제, 데이터 손실 위험, 깨진 기능
- **중요**: 성능 문제, 유지보수성 우려
- **제안**: 스타일 선호, 사소한 개선
- **니트픽**: 선택 사항임을 알 수 있게 명확히 라벨

**3. 찾는 것**
- 명시된 문제를 해결하나?
- 엣지 케이스와 오류 처리
- 새 기능에 대한 테스트 커버리지
- 가독성과 유지보수성
- 보안 영향
- 규모에서의 성능

**4. 커뮤니케이션 방식**
- 변경 요구보다 질문
- 제안 뒤의 "왜" 설명
- 좋은 솔루션 칭찬 (비판만 아님)
- 문제만 아닌 대안 제시

**PR 제출할 때:**

**1. 리뷰 쉽게 만들기**
- 작고 집중된 PR (가능하면 400줄 미만)
- 컨텍스트가 있는 명확한 설명
- 리뷰 요청 전 셀프 리뷰
- 불확실한 영역 강조

**2. 건설적으로 응답**
- 피드백에 리뷰어 감사
- 동의하지 않으면 이유 설명
- 필요하면 명확화 요청

**다날에서:**
매일 3-5개의 PR을 리뷰했습니다. 받은 최고의 리뷰는 코드 문법만이 아니라 시스템 컨텍스트를 이해한 사람들에게서 왔습니다.
