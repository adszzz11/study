# 입사 후 첫 30일에 무엇을 하시겠습니까

## English Question

**What would you do in your first 30 days?**

## 질문 번역

입사 후 첫 30일에 무엇을 하시겠습니까?

---

## English Answer

My goal for the first 30 days: become useful as fast as possible.

**Week 1: Understand**

**Technical:**
- Set up local development environment
- Read existing codebase (architecture, patterns, conventions)
- Understand current infrastructure and deployment
- Review open issues and recent PRs

**Business:**
- Understand the product from a customer perspective
- Learn the domain: how municipal zoning actually works
- Meet the team (even virtually)—understand everyone's role

**Context:**
- What are the biggest pain points right now?
- What's blocked? What's in progress?
- What's the near-term roadmap?

**Week 2: Contribute Small**

- Pick up a small, well-defined task
- Ship something—even if minor
- Get familiar with the PR and deployment process
- Start building relationships through collaboration

**Week 3: Go Deeper**

- Take on a medium-sized feature or bug fix
- Start contributing to technical discussions
- Identify areas where I can add value
- Ask: "What's causing the most pain that I can help with?"

**Week 4: Build Momentum**

- Own a meaningful piece of work
- Document what I've learned (help future teammates)
- Propose improvements (if I see them)
- Set goals for the next 60 days with my manager

**What I Won't Do:**
- Come in with strong opinions before understanding context
- Propose big changes before I've earned credibility
- Wait to be told what to do—proactively find work

**My Measure of Success:**
After 30 days, the team should feel like I've been helpful, not like they've been onboarding me.

## 한글 번역

첫 30일의 목표: 가능한 빨리 유용해지기.

**1주차: 이해**

**기술:**
- 로컬 개발 환경 설정
- 기존 코드베이스 읽기 (아키텍처, 패턴, 컨벤션)
- 현재 인프라와 배포 이해
- 오픈 이슈와 최근 PR 리뷰

**비즈니스:**
- 고객 관점에서 제품 이해
- 도메인 배우기: 지자체 조닝이 실제로 어떻게 작동하는지
- 팀 만나기 (가상이라도)—모든 사람의 역할 이해

**컨텍스트:**
- 지금 가장 큰 페인 포인트는?
- 무엇이 막혀 있나? 무엇이 진행 중인가?
- 단기 로드맵은?

**2주차: 작게 기여**

- 작고 잘 정의된 작업 선택
- 무언가 출시—사소해도
- PR과 배포 프로세스에 익숙해지기
- 협업을 통해 관계 구축 시작

**3주차: 더 깊이**

- 중간 크기 기능이나 버그 수정 맡기
- 기술 토론에 기여 시작
- 가치를 추가할 수 있는 영역 식별
- 질문: "제가 도울 수 있는 가장 큰 고통은?"

**4주차: 모멘텀 구축**

- 의미 있는 작업 오너십
- 배운 것 문서화 (미래 팀원 돕기)
- 개선 제안 (보이면)
- 매니저와 다음 60일 목표 설정

**하지 않을 것:**
- 컨텍스트 이해 전에 강한 의견으로 들어오기
- 신뢰 얻기 전에 큰 변경 제안
- 뭘 해야 할지 듣기를 기다리기—능동적으로 일 찾기

**성공 척도:**
30일 후 팀이 제가 온보딩 중이라고 느끼지 않고 도움이 됐다고 느껴야 합니다.
