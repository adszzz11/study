# 수천 개 시정부 웹사이트에서 데이터를 수집하는 시스템을 어떻게 설계하시겠습니까

## English Question

**How would you design a system to ingest data from thousands of municipal websites?**

## 질문 번역

수천 개의 시정부 웹사이트에서 데이터를 수집하는 시스템을 어떻게 설계하시겠습니까?

---

## English Answer

This is a fascinating problem. Let me walk you through my approach:

**1. Architecture Overview**
I'd design a distributed scraping system with these components:
- **Scheduler**: Manages crawl frequency per source (some update daily, some weekly)
- **Worker Pool**: Horizontally scalable scrapers running in Kubernetes
- **Data Pipeline**: ETL process to normalize and validate data
- **Storage**: Raw data in object storage, processed data in a database

**2. Handling Scale**
With thousands of sources:
- **Rate limiting per domain** to respect robots.txt and avoid being blocked
- **Distributed queue** (Kafka or SQS) to manage work distribution
- **Incremental updates** - only fetch changed content, not full re-scrapes

**3. Handling Inconsistency**
Each municipality formats data differently. My approach:
- **Template-based parsers** for common formats
- **ML-assisted extraction** for unstructured content (meeting minutes, PDFs)
- **Data validation layer** that flags anomalies for human review

**4. Reliability**
- **Retry logic** with exponential backoff
- **Dead letter queues** for failed jobs
- **Monitoring & alerting** for source changes that break parsers

**5. From My Experience**
At Danal, I dealt with data from multiple payment networks, each with different formats and update schedules. The patterns are similar: normalize at ingestion, validate aggressively, monitor for drift.

## 한글 번역

매력적인 문제입니다. 제 접근 방식을 설명하겠습니다:

**1. 아키텍처 개요**
다음 컴포넌트로 분산 스크래핑 시스템을 설계하겠습니다:
- **스케줄러**: 소스별 크롤 빈도 관리 (일부는 매일, 일부는 매주 업데이트)
- **워커 풀**: Kubernetes에서 실행되는 수평 확장 가능한 스크래퍼
- **데이터 파이프라인**: 데이터 정규화 및 검증을 위한 ETL 프로세스
- **스토리지**: 오브젝트 스토리지에 원시 데이터, 데이터베이스에 처리된 데이터

**2. 규모 처리**
수천 개의 소스로:
- **도메인별 속도 제한**으로 robots.txt 준수 및 차단 방지
- **분산 큐** (Kafka 또는 SQS)로 작업 분배 관리
- **증분 업데이트** - 전체 재스크래핑이 아닌 변경된 콘텐츠만 가져오기

**3. 불일치 처리**
각 지자체마다 데이터 형식이 다릅니다. 제 접근:
- **템플릿 기반 파서**로 일반적인 형식 처리
- **ML 지원 추출**로 비정형 콘텐츠 (회의록, PDF) 처리
- **데이터 검증 레이어**로 인간 검토를 위해 이상 항목 플래그

**4. 신뢰성**
- 지수 백오프가 있는 **재시도 로직**
- 실패한 작업을 위한 **데드 레터 큐**
- 파서를 깨뜨리는 소스 변경에 대한 **모니터링 및 알림**

**5. 제 경험에서**
다날에서 각각 다른 형식과 업데이트 일정을 가진 여러 결제 네트워크의 데이터를 다뤘습니다. 패턴이 유사합니다: 수집 시 정규화, 적극적으로 검증, 드리프트 모니터링.
