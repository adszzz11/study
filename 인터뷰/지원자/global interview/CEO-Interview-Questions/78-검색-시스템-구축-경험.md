# 검색 시스템을 구축한 경험이 있나요

## English Question

**Have you built search systems? How would you approach building one?**

## 질문 번역

검색 시스템을 구축한 경험이 있나요? 어떻게 접근하시겠습니까?

---

## English Answer

**My Experience:**
At Danal, I worked on internal search for transaction logs and merchant data—not a user-facing search engine, but the principles apply.

**What I Built:**
- Elasticsearch cluster for transaction search
- Full-text search across merchant names, transaction IDs, descriptions
- Aggregations for analytics dashboards
- Near-real-time indexing from database

**How I'd Approach Search for Zonagent:**

**1. Understand the Use Case**
- What are users searching for? (Municipal names, project types, keywords)
- What's "good" look like? (Precision vs. recall trade-offs)
- How much data? (Thousands vs. millions of documents)

**2. Choose the Right Tool**
- **Elasticsearch/OpenSearch**: Full-text search, good for structured queries
- **Vector search**: Semantic similarity, good for "find documents like this"
- **Hybrid**: Combine both for best results

**3. Search Architecture**
```
User Query → Query Processing → Search Engine → Ranking → Results
                  ↓                                ↓
            Spell check           Relevance scoring
            Synonyms              Personalization
            Filters               Facets
```

**4. Key Components**
- **Indexing pipeline**: Keep search index in sync with source data
- **Query processing**: Tokenization, stemming, synonym expansion
- **Ranking**: TF-IDF, BM25, or ML-based ranking
- **Relevance tuning**: Boost important fields, penalize stale data

**5. Iteration**
Search is never done:
- Analyze search logs (what are people searching for?)
- Track click-through rates
- A/B test ranking changes

**For Municipal Documents:**
I'd probably use hybrid search—keyword matching for specific terms (zoning codes, addresses) plus semantic search for conceptual queries.

## 한글 번역

**제 경험:**
다날에서 트랜잭션 로그와 가맹점 데이터에 대한 내부 검색 작업을 했습니다—사용자 대면 검색 엔진은 아니지만 원칙은 적용됩니다.

**구축한 것:**
- 트랜잭션 검색용 Elasticsearch 클러스터
- 가맹점 이름, 트랜잭션 ID, 설명 전반의 전문 검색
- 분석 대시보드용 집계
- 데이터베이스에서 준실시간 인덱싱

**Zonagent 검색 접근 방식:**

**1. 사용 사례 이해**
- 사용자가 무엇을 검색하나? (지자체 이름, 프로젝트 유형, 키워드)
- "좋은" 것은 무엇인가? (정밀도 vs. 재현율 트레이드오프)
- 얼마나 많은 데이터? (수천 vs. 수백만 문서)

**2. 올바른 도구 선택**
- **Elasticsearch/OpenSearch**: 전문 검색, 구조화된 쿼리에 적합
- **벡터 검색**: 의미적 유사성, "이것과 같은 문서 찾기"에 적합
- **하이브리드**: 최상의 결과를 위해 둘 다 결합

**3. 검색 아키텍처**
```
사용자 쿼리 → 쿼리 처리 → 검색 엔진 → 랭킹 → 결과
                 ↓                        ↓
            맞춤법 검사            관련성 점수화
            동의어                 개인화
            필터                   패싯
```

**4. 핵심 컴포넌트**
- **인덱싱 파이프라인**: 검색 인덱스를 소스 데이터와 동기화 유지
- **쿼리 처리**: 토큰화, 스테밍, 동의어 확장
- **랭킹**: TF-IDF, BM25, 또는 ML 기반 랭킹
- **관련성 튜닝**: 중요 필드 부스트, 오래된 데이터 페널티

**5. 반복**
검색은 절대 완료되지 않음:
- 검색 로그 분석 (사람들이 무엇을 검색하나?)
- 클릭률 추적
- 랭킹 변경 A/B 테스트

**지자체 문서의 경우:**
아마 하이브리드 검색을 사용할 것입니다—특정 용어 (조닝 코드, 주소)에 대한 키워드 매칭과 개념적 쿼리에 대한 의미적 검색.
