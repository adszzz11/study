# 스타트업에서 기술 부채에 대한 접근 방식이 어떻게 되나요

## English Question

**What's your approach to technical debt in a startup?**

## 질문 번역

스타트업에서 기술 부채에 대한 접근 방식이 어떻게 되나요?

---

## English Answer

Technical debt isn't always bad. It's a tradeoff. The key is being intentional about it.

**My Framework:**

**1. Distinguish Good Debt from Bad Debt**
- **Good debt**: Conscious shortcuts to ship faster, with a plan to address later. "We'll use a simple solution now and refactor when we scale."
- **Bad debt**: Accidental mess that slows everyone down with no clear path forward.

I'm okay with good debt. Bad debt is what kills teams.

**2. Track It Explicitly**
Untracked debt is the most dangerous. I:
- Document shortcuts when we take them
- Create tickets for known issues
- Tag code with TODO comments that link to issues
- Review debt regularly—don't let it become invisible

**3. Pay Interest, Don't Just Accumulate**
Allocate some capacity for ongoing cleanup:
- Fix small things as you go
- Refactor during feature work when it makes sense
- Don't wait for a "big cleanup project" that never comes

**4. Know When to Pay It Off**
Some debt needs priority attention:
- When it's blocking new features
- When it's causing production issues
- When onboarding new engineers is painful because of it

**For a Startup:**
In early stages, speed matters more than perfection. I'd rather ship something that works and iterate than build something perfect that never launches. But I won't create debt that makes iteration impossible.

The goal: Stay fast while staying flexible.

## 한글 번역

기술 부채가 항상 나쁜 것은 아닙니다. 트레이드오프입니다. 핵심은 의도적으로 하는 것입니다.

**제 프레임워크:**

**1. 좋은 부채와 나쁜 부채 구분**
- **좋은 부채**: 나중에 해결할 계획으로 더 빨리 출시하기 위한 의식적 지름길. "지금은 간단한 솔루션을 쓰고 스케일할 때 리팩토링하자."
- **나쁜 부채**: 명확한 진로 없이 모든 사람을 느리게 하는 우발적 혼란.

좋은 부채는 괜찮습니다. 나쁜 부채가 팀을 죽입니다.

**2. 명시적으로 추적**
추적되지 않는 부채가 가장 위험합니다. 저는:
- 지름길을 택할 때 문서화
- 알려진 이슈에 대한 티켓 생성
- 이슈에 링크되는 TODO 코멘트로 코드 태그
- 정기적으로 부채 검토—보이지 않게 되지 않도록

**3. 축적만 하지 말고 이자 지불**
지속적인 정리를 위해 일부 용량 할당:
- 진행하면서 작은 것들 수정
- 의미 있을 때 기능 작업 중 리팩토링
- 절대 오지 않는 "대규모 정리 프로젝트"를 기다리지 않음

**4. 언제 갚을지 알기**
일부 부채는 우선 주의가 필요:
- 새 기능을 막을 때
- 프로덕션 이슈를 유발할 때
- 그것 때문에 새 엔지니어 온보딩이 고통스러울 때

**스타트업에서:**
초기 단계에서는 완벽보다 속도가 더 중요합니다. 완벽하지만 절대 출시되지 않는 것보다 작동하는 것을 출시하고 반복하는 것을 선호합니다. 하지만 반복을 불가능하게 만드는 부채는 만들지 않겠습니다.

목표: 유연하면서 빠르게 유지.
