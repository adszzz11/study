# AI 추론 파이프라인을 어떻게 설계하시겠습니까

## English Question

**How would you design an AI inference pipeline for document processing?**

## 질문 번역

문서 처리를 위한 AI 추론 파이프라인을 어떻게 설계하시겠습니까?

---

## English Answer

For Zonagent's use case—processing municipal documents at scale—here's how I'd approach it:

**Pipeline Architecture:**

```
Documents → Preprocessing → Model Inference → Post-processing → Storage
                ↓                 ↓                  ↓
            Queue/Buffer    GPU/CPU Workers     Validation
```

**1. Ingestion Layer**
- Message queue (Kafka/SQS) for document intake
- Prioritization: urgent documents processed first
- Rate limiting to prevent overwhelming downstream

**2. Preprocessing**
- Document type detection (PDF, HTML, image)
- Text extraction (PyPDF2, BeautifulSoup, OCR)
- Chunking for long documents
- Cleaning and normalization

**3. Inference Layer**
- Batch processing for efficiency (group similar documents)
- Model serving:
  - Self-hosted: vLLM, TensorRT for optimization
  - Managed: OpenAI API, Anthropic API with rate management
- Retry logic with exponential backoff
- Fallback models for reliability

**4. Post-processing**
- Confidence scoring
- Result validation (schema compliance)
- Human-in-the-loop for low-confidence results
- Structured output formatting

**5. Storage & Serving**
- Results to database with version tracking
- Cache for frequently accessed results
- API for downstream consumers

**Key Considerations:**

**Cost Management:**
- Batch to reduce API calls
- Cache repeated queries
- Use smaller models for simple tasks

**Latency vs. Throughput:**
- Real-time needs: optimize for latency
- Batch processing: optimize for throughput

**Reliability:**
- Idempotent processing
- Dead letter queues for failures
- Monitoring at each stage

## 한글 번역

Zonagent의 사용 사례—지자체 문서의 대규모 처리—에 대한 제 접근 방식:

**파이프라인 아키텍처:**

```
문서 → 전처리 → 모델 추론 → 후처리 → 저장
          ↓           ↓           ↓
      큐/버퍼    GPU/CPU 워커    검증
```

**1. 수집 레이어**
- 문서 수집용 메시지 큐 (Kafka/SQS)
- 우선순위: 긴급 문서 먼저 처리
- 다운스트림 과부하 방지를 위한 속도 제한

**2. 전처리**
- 문서 타입 감지 (PDF, HTML, 이미지)
- 텍스트 추출 (PyPDF2, BeautifulSoup, OCR)
- 긴 문서 청킹
- 정리 및 정규화

**3. 추론 레이어**
- 효율성을 위한 배치 처리 (유사 문서 그룹화)
- 모델 서빙:
  - 자체 호스팅: 최적화를 위한 vLLM, TensorRT
  - 관리형: 속도 관리와 함께 OpenAI API, Anthropic API
- 지수 백오프와 함께 재시도 로직
- 신뢰성을 위한 폴백 모델

**4. 후처리**
- 신뢰도 점수화
- 결과 검증 (스키마 준수)
- 낮은 신뢰도 결과에 대한 휴먼 인 더 루프
- 구조화된 출력 포매팅

**5. 저장 & 서빙**
- 버전 추적과 함께 데이터베이스에 결과
- 자주 액세스되는 결과용 캐시
- 다운스트림 소비자를 위한 API

**핵심 고려사항:**

**비용 관리:**
- API 호출 줄이기 위해 배치
- 반복 쿼리 캐시
- 간단한 작업에 작은 모델 사용

**지연 시간 vs. 처리량:**
- 실시간 필요: 지연 시간 최적화
- 배치 처리: 처리량 최적화

**신뢰성:**
- 멱등 처리
- 실패용 데드 레터 큐
- 각 단계에서 모니터링
