# 우리 시스템에서 데이터 품질을 어떻게 보장하시겠습니까

## English Question

**How would you ensure data quality in a system like ours?**

## 질문 번역

우리 같은 시스템에서 데이터 품질을 어떻게 보장하시겠습니까?

---

## English Answer

Data quality is everything for Zonagent. If developers can't trust the data, the product is worthless. Here's how I'd approach it:

**1. Quality at Ingestion**
Catch problems early:
- **Schema validation**: Does the data match expected structure?
- **Business rules**: Are zoning codes in valid formats? Do approval dates make sense?
- **Completeness checks**: Are required fields populated?

Failed records go to quarantine, not production.

**2. Source Reliability Scoring**
Not all sources are equal:
- Track accuracy of each municipal source over time
- Monitor freshness (when was data last updated?)
- Flag sources with high error rates for review
- Surface reliability scores to users so they know context

**3. Cross-Validation**
Where possible, verify data across multiple sources:
- If two sources disagree, flag for review
- Use public records to validate scraped data
- Implement sanity checks (e.g., approval date should be after application date)

**4. Continuous Monitoring**
Data quality degrades over time:
- Dashboards showing validation failure rates
- Alerts when sources change format or stop updating
- Regular audits comparing our data to ground truth

**5. Feedback Loop**
When users report inaccuracies:
- Easy way to flag incorrect data
- Track which sources generate most complaints
- Prioritize fixing high-impact errors

**From My Experience:**
In payments, a 0.01% error rate means real money lost. I've built systems where data accuracy isn't optional—it's existential. That mindset transfers directly to Zonagent.

## 한글 번역

데이터 품질은 Zonagent의 전부입니다. 개발자들이 데이터를 신뢰할 수 없다면 제품은 무가치합니다. 접근 방식:

**1. 수집 시 품질**
문제를 일찍 잡습니다:
- **스키마 검증**: 데이터가 예상 구조와 일치하나?
- **비즈니스 규칙**: 조닝 코드가 유효한 형식인가? 승인 날짜가 의미 있나?
- **완전성 검사**: 필수 필드가 채워져 있나?

실패한 레코드는 프로덕션이 아닌 격리 구역으로 갑니다.

**2. 소스 신뢰성 점수화**
모든 소스가 동일하지 않습니다:
- 시간에 따른 각 시정부 소스의 정확도 추적
- 신선도 모니터링 (데이터가 마지막으로 업데이트된 시점?)
- 높은 오류율의 소스를 검토 플래그
- 사용자에게 신뢰성 점수를 표시하여 맥락 파악

**3. 교차 검증**
가능한 경우 여러 소스에서 데이터 검증:
- 두 소스가 불일치하면 검토 플래그
- 공공 기록을 사용하여 스크랩된 데이터 검증
- 정상성 검사 구현 (예: 승인 날짜는 신청 날짜 이후여야 함)

**4. 지속적 모니터링**
데이터 품질은 시간이 지나면서 저하됩니다:
- 검증 실패율을 보여주는 대시보드
- 소스가 형식을 변경하거나 업데이트를 중지할 때 알림
- 우리 데이터를 실제와 비교하는 정기 감사

**5. 피드백 루프**
사용자가 부정확성을 보고할 때:
- 잘못된 데이터를 플래그하는 쉬운 방법
- 어떤 소스가 가장 많은 불만을 생성하는지 추적
- 높은 영향 오류 수정 우선순위

**제 경험에서:**
결제에서 0.01% 오류율은 실제 돈 손실을 의미합니다. 데이터 정확성이 선택이 아닌—존재적인 시스템을 구축했습니다. 그 마인드셋이 Zonagent에 직접 전이됩니다.
