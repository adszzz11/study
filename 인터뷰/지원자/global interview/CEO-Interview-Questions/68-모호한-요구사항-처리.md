# 모호한 요구사항을 어떻게 처리하나요

## English Question

**How do you handle ambiguous requirements?**

## 질문 번역

모호한 요구사항을 어떻게 처리하나요?

---

## English Answer

Ambiguity is normal in startups. Here's how I navigate it:

**1. Clarify What's Actually Fixed**
Some things are flexible, others aren't:
- Fixed deadline? Then scope must flex.
- Fixed scope? Then timeline must flex.
- Fixed both? Something will break—make that visible early.

**2. Ask Targeted Questions**
Not "what do you want?" but:
- "What problem are we solving?"
- "Who is the user for this?"
- "What does success look like?"
- "What's the minimum viable version?"

**3. Propose Concrete Options**
When requirements are vague, I draft options:
- "Here are three ways to approach this. Option A is fastest but limited. Option B is more complete but takes longer. Option C is in between."
- Let stakeholders react to something concrete.

**4. Build to Learn**
Sometimes the best way to clarify is to prototype:
- Quick mock or spike
- Show, don't just discuss
- "Is this what you meant? No? What's wrong?"

**5. Document Assumptions**
When I proceed without clarity:
- "I'm assuming X. Correct me if wrong."
- Written in PR description or Slack
- Creates opportunity for correction before work is done

**6. Set Checkpoints**
For longer work with ambiguity:
- Check in early with partial work
- "Here's the direction—are we aligned?"
- Cheaper to course-correct at 20% than at 80%

**My Mindset:**
Waiting for perfect requirements means waiting forever. Start with what you know, make assumptions explicit, and iterate.

## 한글 번역

모호함은 스타트업에서 정상입니다. 어떻게 탐색하는지:

**1. 실제로 고정된 것 명확화**
어떤 것은 유연하고 다른 것은 아님:
- 고정된 마감? 그러면 스코프가 유연해야.
- 고정된 스코프? 그러면 타임라인이 유연해야.
- 둘 다 고정? 뭔가 깨질 것—일찍 가시화.

**2. 타겟팅된 질문**
"뭘 원하세요?"가 아니라:
- "어떤 문제를 해결하나요?"
- "이것의 사용자는 누구인가요?"
- "성공은 어떻게 생겼나요?"
- "최소 viable 버전은 무엇인가요?"

**3. 구체적 옵션 제안**
요구사항이 모호할 때 옵션 초안:
- "이것에 접근하는 세 가지 방법입니다. 옵션 A는 가장 빠르지만 제한적. 옵션 B는 더 완전하지만 더 오래 걸림. 옵션 C는 중간."
- 이해관계자가 구체적인 것에 반응하게 함.

**4. 배우기 위해 만들기**
때때로 명확화하는 가장 좋은 방법은 프로토타입:
- 빠른 목업이나 스파이크
- 논의만 하지 말고 보여주기
- "이것이 의미한 것인가요? 아니요? 뭐가 잘못됐나요?"

**5. 가정 문서화**
명확성 없이 진행할 때:
- "X를 가정합니다. 틀리면 수정해주세요."
- PR 설명이나 Slack에 작성
- 작업 완료 전에 수정 기회 생성

**6. 체크포인트 설정**
모호함이 있는 긴 작업에:
- 부분 작업으로 일찍 체크인
- "방향입니다—정렬되었나요?"
- 80%에서보다 20%에서 방향 수정이 더 저렴

**제 마인드셋:**
완벽한 요구사항을 기다리면 영원히 기다립니다. 아는 것으로 시작하고, 가정을 명시적으로 만들고, 반복합니다.
