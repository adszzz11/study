# 데이터베이스 확장성을 어떻게 다루나요

## English Question

**How do you approach database scalability for high-volume applications?**

## 질문 번역

고용량 애플리케이션을 위한 데이터베이스 확장성을 어떻게 접근하나요?

---

## English Answer

At Danal, handling 100K+ transactions per minute meant database scalability was existential. Here's my approach:

**1. Read-Write Separation**
- Master for writes, replicas for reads
- We had 1 master + 3 read replicas
- Application-level routing based on query type

**2. Sharding Strategy**
When single-node limits hit:
- Shard by merchant ID (our primary access pattern)
- Consistent hashing for shard distribution
- Cross-shard queries are expensive—design to avoid them

**3. Caching Layers**
Not everything needs to hit the database:
- Redis for hot data (session, rate limits, recent transactions)
- Local cache for static reference data
- Cache invalidation strategy is critical—we used TTL + event-based invalidation

**4. Query Optimization**
- Proper indexing (compound indexes for common query patterns)
- Query analysis and slow query monitoring
- Denormalization where read performance matters

**5. Connection Pooling**
- PgBouncer for PostgreSQL connection pooling
- Prevents connection exhaustion under load

**For Zonagent:**
Your scale will be different—thousands of municipalities, not millions of transactions per second. But the principles apply:
- Understand your access patterns first
- Cache aggressively for read-heavy workloads
- Plan for growth before you need it

## 한글 번역

다날에서 분당 10만 건 이상의 트랜잭션을 처리한다는 것은 데이터베이스 확장성이 생존과 직결되었음을 의미합니다. 제 접근 방식:

**1. 읽기-쓰기 분리**
- 쓰기용 마스터, 읽기용 리플리카
- 1개 마스터 + 3개 읽기 리플리카
- 쿼리 타입에 따른 애플리케이션 레벨 라우팅

**2. 샤딩 전략**
단일 노드 한계에 도달했을 때:
- 가맹점 ID로 샤딩 (주요 액세스 패턴)
- 샤드 분배를 위한 일관된 해싱
- 크로스 샤드 쿼리는 비용이 큼—피하도록 설계

**3. 캐싱 레이어**
모든 것이 데이터베이스를 칠 필요 없음:
- 핫 데이터용 Redis (세션, 속도 제한, 최근 트랜잭션)
- 정적 참조 데이터용 로컬 캐시
- 캐시 무효화 전략이 중요—TTL + 이벤트 기반 무효화 사용

**4. 쿼리 최적화**
- 적절한 인덱싱 (일반 쿼리 패턴용 복합 인덱스)
- 쿼리 분석 및 슬로우 쿼리 모니터링
- 읽기 성능이 중요한 곳에서 비정규화

**5. 커넥션 풀링**
- PostgreSQL 커넥션 풀링용 PgBouncer
- 부하 상황에서 커넥션 고갈 방지

**Zonagent의 경우:**
규모가 다를 것입니다—초당 수백만 트랜잭션이 아닌 수천 개의 지자체. 하지만 원칙은 적용됩니다:
- 먼저 액세스 패턴 이해
- 읽기 중심 워크로드에 적극적 캐싱
- 필요하기 전에 성장 계획
