## Project 1: MVP 연동

### 목표

이 시스템이 매일 생성하는 자료를 NotebookLM에서 읽을 수 있게 한다.

### 흐름

```text
Daily job
  -> research-report.md
  -> summary.pdf
  -> Google Drive folder
  -> NotebookLM Drive source
  -> User chat / Audio Overview / study artifacts
```

### 구현 체크리스트

- [ ] `research-report.md` 생성
- [ ] `summary.pdf` 생성
- [ ] Google Drive 폴더와 파일 naming convention 정의
- [ ] NotebookLM notebook 생성
- [ ] Drive source import
- [ ] 원본 수정 후 sync latency 측정
- [ ] citation 품질 기록

### 성공 기준

| 기준 | 목표 |
|------|------|
| sync latency | 몇 분 단위 반영 확인 |
| citation quality | 핵심 claim에 source citation 표시 |
| format quality | headings, tables, bullet이 질문에 잘 반영 |
| 운영 부담 | 사용자가 최초 연결 후 반복 업로드를 하지 않아도 됨 |

## Project 2: 팀 knowledge base

### 목표

제품 스펙, 회의록, 정책 문서, FAQ를 NotebookLM shared notebook으로 묶어 onboarding과 검색을 돕는다.

### source 구성

| Source | Format | 관리 위치 |
|---|---|---|
| Product specs | Google Docs | Shared Drive |
| Meeting notes | Google Docs/Markdown | Shared Drive |
| Policy docs | PDF/Docs | Shared Drive |
| FAQ | Google Sheets/CSV | Shared Drive |
| Release notes | Markdown/PDF | Generated by system |

### 운영 규칙

- source owner를 정한다.
- 오래된 문서와 최신 문서를 분리한다.
- NotebookLM 답변을 최종 정책으로 보지 않고 source citation을 확인한다.
- 민감 문서는 Workspace/Enterprise privacy와 sharing policy를 확인한 뒤 추가한다.

## Project 3: 자동화형 RAG 대체

### 목표

사용자 앱에서 문서 업로드, 질문, citation 포함 답변을 자동화한다.

### 후보 구현

| Backend | PoC 범위 | 평가 기준 |
|---|---|---|
| Gemini API | PDF/document upload 후 structured answer 생성 | document understanding, Google ecosystem |
| Google Cloud Agent Search | data store 기반 generated answers | access control, enterprise search |
| OpenAI File Search | vector store + Responses API | API ergonomics, citation quality |

### 평가 질문

- 답변 latency가 제품 요구사항에 맞는가?
- citations가 사용자가 검증 가능한 형태인가?
- source update와 re-indexing이 자동화되는가?
- 비용, rate limit, access control을 운영할 수 있는가?

## Project 4: 하이브리드 운영

### 목표

사용자는 NotebookLM UI로 탐색하고, production workflow는 API 기반 RAG로 재현 가능하게 만든다.

```text
Shared sources
  -> NotebookLM for human exploration
  -> API RAG for product workflow
```

### 설계 원칙

- source of truth는 Drive 또는 문서 저장소로 둔다.
- NotebookLM 결과는 탐색/리서치 산출물로 취급한다.
- product feature는 Gemini API, Agent Search, OpenAI File Search 중 하나로 구현한다.
- 두 시스템의 답변 품질을 동일한 question set으로 비교한다.
