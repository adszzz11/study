  - 백엔드가 동의함

- [ ] **명세 문서 지속 업데이트** (iOS)
- [ ] **API 설명을 상세하게 작성** (FE)
- [ ] **공통코드 설명 포함** (FE)

### 인증/보안

- [ ] **로그인 토큰 갱신 로직 집중 설계** (iOS)

### 커뮤니케이션

- [ ] **배포 시 커뮤니케이션** (AOS)

## DON'T (하지 않았으면 좋겠는 것)

### API 변경 (심각)

- [ ] ❌ **미공지 API 변경** (FE)
- [ ] ❌ **소통 없는 API 변경** (iOS)
- [ ] ❌ **배포 관련 소통 없이 변경** (AOS)

### nullable 관련 (전체 공통)

- [ ] ❌ **nullable 표시 누락** (FE, iOS, AOS 전체)

### Schema 설계

- [ ] ❌ **Schema 과도하게 쪼개기** (AOS)
- [ ] ❌ 프론트에서 과도한 데이터 정제 요구 (FE)

### 문서화

- [ ] ❌ 불명확한 공통코드 설명 누락 (FE)
- [ ] ❌ 공통코드 만들어두고 설명 안 하기 (FE)

### 인증

- [ ] ❌ **토큰을 DB에서 관리하지 않는 설계** (iOS)

## 주요 이슈 포인트

### 1. nullable (최다 언급)

```
"nullable 좀 ㅅㅂ 잘써라" - AOS
"nullable 표시좀 ㅅㅂ 잘해줘라" - iOS
```

**해결책**: 모든 API 응답 필드에 nullable 여부 명시

### 2. 미공지 변경

모든 파트에서 공통적으로 가장 싫어하는 것:
- API 변경 시 사전 공지 필수
- 배포 전 커뮤니케이션 필수

### 3. 로그인 토큰 갱신

- 토큰 갱신 로직에 집중 필요
- DB에서 토큰 관리 필수

## API 설계 가이드라인

### JSON 구조

```json
// Good - Depth 4 이하
{
  "user": {
    "profile": {
      "name": "홍길동",
      "avatar": "url"
    }
  }
}

// Bad - Depth 4 초과
{
  "data": {
    "user": {
      "profile": {
        "details": {
          "name": "홍길동"  // Depth 5
        }
      }
    }
  }
}
```

### Schema 설계 원칙 (AOS)

- Entity 값을 원형으로 내려주기
- 과도하게 쪼개지 않기
- 프론트에서 필요한 데이터 선별 가능하도록

## 파트별 주요 요청 요약

| 파트 | 핵심 요청 |
|------|----------|
| **FE** | JSON Depth 4 이하, 서버에서 데이터 정제, OpenAPI Schema, 공통코드 설명 |
| **iOS** | Swagger 업데이트, nullable 명시, 토큰 갱신 로직, DB 토큰 관리 |
| **AOS** | nullable 명시, Entity 원형 전달, 배포 커뮤니케이션, Schema 단순화 |
