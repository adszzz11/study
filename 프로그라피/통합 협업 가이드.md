## 개요

이 문서는 프로그라피 프로젝트에서 각 파트 간 원활한 협업을 위해 필요한 산출물과 협업 원칙을 정리한 가이드입니다.

### 문서 활용 방법

1. **밍글링 전**: 각 파트가 자신의 섹션을 확인하고 산출물 생성 가능 여부 검토
2. **밍글링 중**: "만들자/말자" 의사결정 및 우선순위 조율
3. **프로젝트 진행 중**: 체크리스트로 활용하여 누락 방지

## 공통 원칙

> 모든 파트에 적용되는 핵심 협업 원칙

### 1. 초반 확정의 원칙

| 항목 | 설명 | 관련 파트 |
|------|------|----------|
| 구조 확정 | 아키텍처, 기획 구조는 프로젝트 초반에 확정 | 기획, 개발 전체 |
| 컨셉 확정 | 디자인 컨셉, 서비스 방향성 초기 정의 | 기획, 디자인 |
| 기술 스택 확정 | 프레임워크, 라이브러리 초기 선정 | 개발 전체 |

### 2. 변경 관리 원칙

- 테스트베드 배포 이후 기획 변경 **최소화**
- 불가피한 변경 시 **영향도 분석** 후 전파트 공유
- API 변경 시 **사전 공지** 필수

### 3. 커뮤니케이션 원칙

- 의견 제시 시 **배경과 이유** 함께 설명
- 버그/이슈 리포트 시 **재현 가능한 정보** 포함
- 정기적인 **싱크업 미팅** 운영

### 4. 문서화 원칙

- 모든 산출물은 **버전 관리**
- 변경사항은 **히스토리 기록**
- 팀 전체가 접근 가능한 **중앙 저장소** 사용

### 기획/PO 파트

> 기획자와 PO의 역할을 통합하여 정리 (규모에 따라 분리 가능)

#### 역할 구분

| 역할 | 핵심 업무 |
|------|----------|
| **PO** | 비즈니스 의사결정, 제품 방향성, "왜 만드는가" 결정 |
| **기획** | 요구사항 구체화, 정책 수립, "어떻게 만드는가" 정의 |

#### 필수 산출물

| 우선순위 | 산출물 | 형식 | 설명 | 요청 파트 |
|---------|--------|------|------|----------|
| 🔴 필수 | Wireframe | PPTX/Figma | 화면 구성의 기본 틀 | 전체 |
| 🔴 필수 | IA (Information Architecture) | 다이어그램 | 전체 화면 구성도 | iOS, 전체 |
| 🔴 필수 | Flow Chart | 다이어그램 | 사용자 흐름도 | FE, 전체 |
| 🟡 권장 | 기능 명세서 | 문서 | 각 기능별 상세 설명 | FE |
| 🟡 권장 | 정책 정의서 | 문서 | 날짜 표기, 문체, String Token 등 | iOS |
| 🟡 권장 | User Story | 문서 | 사용자 관점의 기능 설명 | FE |
| 🟡 권장 | 아이디어 기획서 | 문서 | 아이디어 + 진행 방향 + 일정 | 디자인 |
| 🟢 선택 | 레퍼런스 정리 | 문서/링크 | 참고할 서비스/디자인 | 디자인 |
| 🟢 선택 | 테스트 시나리오 | 문서 | QA 연계용 | QA |

#### 정책 정의서 포함 항목

```markdown
## 정책 정의서

### 1. 표기법
- 날짜: YYYY.MM.DD / YYYY-MM-DD
- 시간: 24시간제 / 12시간제 (오전/오후)
- 숫자: 천 단위 콤마 사용 여부

### 2. 문체
- 요체 (합니다/입니다)
- 다체 (한다/이다)

### 3. String Token 규칙
- 버튼: btn_[기능]_[상태]
- 라벨: lbl_[화면]_[위치]
- 메시지: msg_[유형]_[번호]

### 4. 에러 메시지 규칙
- 형식: [에러 코드] 사용자 친화적 메시지
```

#### 협업 요청사항

**DO (해주세요)**
- [ ] 초반에 구조/컨셉 확정
- [ ] 기획 의도를 명확하게 표현 (맞고 틀리고 상관없이)
- [ ] 실현 가능한 범위 내에서 기획
- [ ] 의견 제시 시 이유와 배경 설명
- [ ] 레퍼런스는 정돈된 형태로 전달

**DON'T (피해주세요)**
- [ ] 테스트베드 이후 기획 변경
- [ ] 아키텍처 근간을 흔드는 후반 변경
- [ ] 근거 없는 고집
- [ ] 어지러운 레퍼런스 전달

### PM 파트

> 일정 조율의 마왕, 실제로 일이 되게끔 만드는 사람

#### 핵심 역할

- 일정 조율 및 리소스 관리
- 파트 간 갈등 중재
- 커뮤니케이션 프로세스 수립
- 리스크 관리

#### 필수 산출물

| 우선순위 | 산출물 | 형식 | 설명 |
|---------|--------|------|------|
| 🔴 필수 | 프로젝트 일정표 | 문서/툴 | 마일스톤 및 세부 일정 |
| 🔴 필수 | 리소스 할당표 | 문서/툴 | 파트별 담당자 및 업무 분장 |
| 🔴 필수 | 정기 리뷰 일정 | 캘린더 | 주간/스프린트 리뷰 |
| 🟡 권장 | 회의록 | 문서 | 주요 결정사항 기록 |
| 🟡 권장 | 이슈 트래킹 | 툴 | Jira, Notion 등 |
| 🟡 권장 | 커뮤니케이션 채널 가이드 | 문서 | 어떤 상황에 어떤 채널 사용 |

#### PM이 조율해야 하는 주요 이슈

| 이슈 유형 | 관련 파트 | 중재 방향 |
|----------|----------|----------|
| 기획 변경 | 기획 ↔ 개발 | 변경 최소화 원칙, 영향도 분석 |
| 디자인 vs 구현 시간 | 디자인 ↔ 개발 | 팀 목표에 맞춰 조율 |
| API 변경 공유 | 백엔드 ↔ 클라이언트 | 사전 공유 프로세스 수립 |
| 플랫폼 특성 미고려 | 디자인 ↔ AOS | 플랫폼별 특성 교육/조율 |

#### 체크리스트

**프로젝트 시작 시**
- [ ] 파트별 인원 확인 및 역할 분담
- [ ] 초반 구조/컨셉 확정 유도
- [ ] 커뮤니케이션 채널 수립
- [ ] 정기 리뷰 일정 수립

**프로젝트 진행 중**
- [ ] 기획 변경 영향도 관리
- [ ] 파트 간 갈등 중재
- [ ] API 변경 공유 프로세스 관리
- [ ] 일정 조율 및 리스크 관리

**배포 전**
- [ ] 테스트베드 배포 후 변경사항 최소화
- [ ] QA 일정 확보
- [ ] 배포 커뮤니케이션

### 디자인 파트

> UX/UI 디자인, 디자인시스템, 브랜딩 담당

#### 필수 산출물

| 우선순위 | 산출물 | 형식 | 설명 | 요청 파트 |
|---------|--------|------|------|----------|
| 🔴 필수 | 디자인시스템 | Figma | 토큰, 컴포넌트 정의 | FE, iOS, AOS |
| 🔴 필수 | Figma 디자인 명세 | Figma | 화면별 디자인 | FE |
| 🔴 필수 | 프로토타입 | Figma | WireFrame 포함 | AOS |
| 🟡 권장 | 디자인 플로우 | Figma | 인터랙션 흐름도 | FE |
| 🟡 권장 | Animation Diagram | Figma | 인터랙션 상세 정의 | AOS, iOS |
| 🟢 선택 | 브랜드 가이드 | 문서 | BI/CI, 컬러, 폰트 | 전체 |

#### 디자인시스템 구성 요소

```markdown
## 디자인시스템 구성

### 1. 토큰 (Token)
- Primitive Token: RGBA 값 (기본 색상)
- Semantic Token: 의미 기반 정의 (primary, secondary, error 등)

### 2. 컴포넌트
- 버튼 (Button): Primary, Secondary, Tertiary, Ghost
- 입력 필드 (Input): Text, Password, Search, TextArea
- 카드 (Card): 기본, 리스트, 상세
- 모달 (Modal): Alert, Confirm, Bottom Sheet
- 네비게이션: Tab Bar, Navigation Bar, Bottom Navigation

### 3. 레이아웃
- 그리드 시스템
- 여백 규칙 (Spacing)
- 타이포그래피 스케일
```

#### 플랫폼별 고려사항

| 플랫폼 | 고려사항 | 중요도 |
|--------|----------|--------|
| **iOS** | Human Interface Guidelines 참고 | 높음 |
| **AOS** | Material Design 참고, 폴더블 대응 | 높음 |
| **AOS** | 다양한 해상도 (넓은 화면/좁은 화면) | 높음 |
| **FE** | 반응형 웹 대응 | 중간 |

#### Android 기기 다양성 대응

| 기기 유형 | 대응 필요 수준 |
|----------|---------------|
| 일반 스마트폰 | 기본 |
| 폴더블 (Fold) | **필수** |
| 플립 | 선택 |
| 특수 해상도 | 케이스별 협의 |

#### 협업 요청사항

**DO (해주세요)**
- [ ] 디자인시스템 기반 구성을 먼저 진행
- [ ] Theme, Token을 일관성 있게 구성
- [ ] 라이브러리에서 재현 가능한 수준의 디자인
- [ ] 구현 시간 고려한 대안 제시
- [ ] Android 특성 고려 (폴더블, 해상도 다양성)

**DON'T (피해주세요)**
- [ ] iPhone 디자인을 Android에 그대로 적용
- [ ] 과도한 Animation 요구 (구현 시간 미고려)
- [ ] 디자인시스템과 맞지 않는 새 컨셉 무작정 추가
- [ ] 너무 컬러풀한 디자인 (통일성 저해)

| 항목 | 중요도 | 설명 |

| 에러 코드 정의 | ⭐⭐ | 에러 유형별 코드 |

#### API 설계 가이드라인

**JSON Depth 권장**

```json
// ✅ Good - Depth 4 이하
{
  "user": {
    "profile": {
      "name": "홍길동",
      "avatar": "url"
    }
  }
}

// ❌ Bad - Depth 4 초과
{
  "data": {
    "user": {
      "profile": {
        "details": {
          "name": "홍길동"  // Depth 5
        }
      }
    }
  }
}
```

**Schema 설계 원칙**
- Entity 값을 원형으로 내려주기
- 과도하게 쪼개지 않기
- 프론트에서 필요한 데이터 선별 가능하도록
