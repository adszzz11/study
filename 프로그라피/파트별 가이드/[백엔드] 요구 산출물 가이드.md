# 백엔드 파트 요구 산출물 가이드

> 다른 파트에서 백엔드에게 요청하는 산출물 및 협업 가이드

## 핵심 역할

- 서버 개발
- API 설계 및 제공

## API 명세 필수 포함 항목

| 항목 | 중요도 | 설명 |
|------|--------|------|
| **nullable 표시** | ⭐⭐⭐ | **모든 파트에서 가장 많이 요청** |
| 공통코드 설명 | ⭐⭐ | 상세하게 작성 |
| API 설명 | ⭐⭐ | 용도, 사용법 |
| 에러 코드 정의 | ⭐⭐ | 에러 유형별 코드 |

### ⚠️ nullable 표시의 중요성

> FE, iOS, AOS 모든 파트에서 가장 많이 요청한 사항입니다

- 모든 API 응답 필드에 nullable 여부 명시
- Optional 필드는 명확하게 표시
- 클라이언트에서 안전한 코드 작성 가능

## API 설계 가이드라인

### JSON Depth 권장 (FE 요청)

```json
// ✅ Good - Depth 4 이하
{
  "user": {
    "profile": {
      "name": "홍길동",
      "avatar": "url"
    }
  }
}

// ❌ Bad - Depth 4 초과
{
  "data": {
    "user": {
      "profile": {
        "details": {
          "name": "홍길동"  // Depth 5
        }
      }
    }
  }
}
```

### Schema 설계 원칙 (AOS 요청)

- Entity 값을 원형으로 내려주기
- 과도하게 쪼개지 않기
- 프론트에서 필요한 데이터 선별 가능하도록

### 데이터 정제 (FE 요청)

- 데이터 정제는 **서버에서 처리** 권장
- 프론트에서 과도한 데이터 가공 지양

## 협업 시 주의사항

### DO (해주세요)

- [ ] **nullable 표시 정확하게** (가장 중요!)
- [ ] API 변경 시 **사전 공유**
- [ ] 배포 전 **커뮤니케이션**
- [ ] JSON Depth 4단계 이하로 구성
- [ ] 데이터 정제는 서버에서 처리
- [ ] 명세 문서 지속 업데이트
- [ ] 공통코드 설명 상세 작성
- [ ] 토큰은 DB에서 관리

### DON'T (피해주세요)

- [ ] 소통 없는 API 변경
- [ ] nullable 표시 누락
- [ ] 배포 관련 소통 없이 변경
- [ ] Schema 과도하게 쪼개기
- [ ] 토큰을 DB에서 관리하지 않는 설계

## 기획에게 요청

> 백엔드 개발자로서 기획에게 이해를 구하는 부분

- [ ] 확장성을 고려한 기획을 해주세요

- [ ] 기획 시 개발 영향도가 궁금하시면 먼저 여쭤봐 주세요

### 기획자가 알아주셨으면 하는 것

| 기능 | 개발 부담 정도 |
|------|---------------|
| 알림 기능 | 높음 (지속적인 데이터 관리 필요) |
| 복잡한 조회 | 데이터 구조에 따라 상이 |
| 실시간 기능 | 높음 (WebSocket 등 필요) |
