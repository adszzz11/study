## API 명세 필수 포함 항목

| 항목 | 중요도 | 설명 |
|------|--------|------|
| **nullable 표시** | ⭐⭐⭐ | **모든 파트에서 가장 많이 요청** |
| 공통코드 설명 | ⭐⭐ | 상세하게 작성 |
| API 설명 | ⭐⭐ | 용도, 사용법 |
| 에러 코드 정의 | ⭐⭐ | 에러 유형별 코드 |

### ⚠️ nullable 표시의 중요성

> FE, iOS, AOS 모든 파트에서 가장 많이 요청한 사항입니다

- 모든 API 응답 필드에 nullable 여부 명시
- Optional 필드는 명확하게 표시
- 클라이언트에서 안전한 코드 작성 가능

## API 설계 가이드라인

### JSON Depth 권장 (FE 요청)

```json
// ✅ Good - Depth 4 이하
{
  "user": {
    "profile": {
      "name": "홍길동",
      "avatar": "url"
    }
  }
}

// ❌ Bad - Depth 4 초과
{
  "data": {
    "user": {
      "profile": {
        "details": {
          "name": "홍길동"  // Depth 5
        }
      }
    }
  }
}
```

### Schema 설계 원칙 (AOS 요청)

- Entity 값을 원형으로 내려주기
- 과도하게 쪼개지 않기
- 프론트에서 필요한 데이터 선별 가능하도록

### 데이터 정제 (FE 요청)

- 데이터 정제는 **서버에서 처리** 권장
- 프론트에서 과도한 데이터 가공 지양

## Swagger API 명세 예시

### 좋은 API 명세 예시

```yaml
/api/v1/users/{userId}:
  get:
    summary: 사용자 정보 조회
    description: 특정 사용자의 상세 정보를 조회합니다.
    tags:
      - Users
    parameters:
      - name: userId
        in: path
        required: true
        schema:
          type: integer
        description: 사용자 고유 ID
    responses:
      200:
        description: 성공
        content:
          application/json:
            schema:
              type: object
              properties:
                id:
                  type: integer
                  example: 1
                name:
                  type: string
                  example: "홍길동"
                email:
                  type: string
                  format: email
                  example: "user@example.com"
                profileImage:
                  type: string
                  nullable: true  # ⬅️ nullable 명시!
                  example: "https://..."
                createdAt:
                  type: string
                  format: date-time
      404:
        description: 사용자를 찾을 수 없음
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ErrorResponse'
```

## 에러 코드 체계

### 에러 응답 표준 포맷

```json
{
  "success": false,
  "error": {
    "code": "USER_NOT_FOUND",
    "message": "사용자를 찾을 수 없습니다.",
    "details": null
  },
  "timestamp": "2025-01-01T12:00:00Z"
}
```

### 에러 코드 분류

| 코드 범위 | 분류 | 예시 |
|----------|------|------|
| **AUTH_xxx** | 인증 관련 | AUTH_INVALID_TOKEN, AUTH_EXPIRED_TOKEN |
| **USER_xxx** | 사용자 관련 | USER_NOT_FOUND, USER_ALREADY_EXISTS |
| **VALIDATION_xxx** | 유효성 검사 | VALIDATION_REQUIRED_FIELD, VALIDATION_INVALID_FORMAT |
| **PERMISSION_xxx** | 권한 관련 | PERMISSION_DENIED, PERMISSION_INSUFFICIENT |
| **INTERNAL_xxx** | 서버 내부 | INTERNAL_SERVER_ERROR, INTERNAL_DB_ERROR |

### 클라이언트별 에러 처리 가이드

| 에러 코드 | HTTP Status | 클라이언트 대응 |
|----------|-------------|----------------|
| AUTH_EXPIRED_TOKEN | 401 | 토큰 갱신 후 재요청 |
| AUTH_INVALID_TOKEN | 401 | 로그아웃 → 로그인 화면 |
| VALIDATION_xxx | 400 | 사용자에게 입력 오류 안내 |
| PERMISSION_DENIED | 403 | 권한 없음 안내 |
| INTERNAL_xxx | 500 | 일반 오류 메시지 표시 |

## API 버저닝 전략

### 버저닝 방식

| 방식 | 예시 | 장단점 |
|------|------|--------|
| **URL Path (권장)** | `/api/v1/users` | 명확함, 캐싱 용이 |
| Header | `API-Version: 1` | URL 깔끔, 관리 복잡 |
| Query Param | `/api/users?version=1` | 간단하지만 비표준 |

### 버전 변경 가이드

```
버전 업 기준:
├── Breaking Change 발생 시
│   ├── 필드 타입 변경
│   ├── 필드 삭제
│   └── 필수 파라미터 추가
│
└── Backward Compatible 변경 시 (버전 유지)
    ├── 새 필드 추가 (optional)
    ├── 새 endpoint 추가
    └── 에러 메시지 개선
```

### 버전 Deprecation 프로세스

```
1. 새 버전 (v2) 출시
   ↓
2. 기존 버전 (v1) Deprecated 공지 (최소 2주 전)
   ↓
3. v1 응답에 Deprecation Warning 헤더 추가
   Deprecation: true
   Sunset: 2025-03-01
   ↓
4. 마이그레이션 기간 (최소 1개월)
   ↓
5. v1 종료
```

## 협업 시 주의사항

### DO (해주세요)

- [ ] **nullable 표시 정확하게** (가장 중요!)
- [ ] API 변경 시 **사전 공유**
- [ ] 배포 전 **커뮤니케이션**
- [ ] JSON Depth 4단계 이하로 구성
- [ ] 데이터 정제는 서버에서 처리
- [ ] 명세 문서 지속 업데이트
- [ ] 공통코드 설명 상세 작성
- [ ] 토큰은 DB에서 관리

### DON'T (피해주세요)

- [ ] 소통 없는 API 변경
- [ ] nullable 표시 누락
- [ ] 배포 관련 소통 없이 변경
- [ ] Schema 과도하게 쪼개기
- [ ] 토큰을 DB에서 관리하지 않는 설계
