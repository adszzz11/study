# PM 파트 요구 산출물 가이드

> 다른 파트에서 PM에게 요청하는 산출물 및 역할 정리

---

## 핵심 역할

> PM은 **일정 조율의 마왕**, 실제로 일이 되게끔 만드는 사람

- 일정 조율 및 리소스 관리
- 파트 간 갈등 중재
- 커뮤니케이션 프로세스 수립
- 리스크 관리

---

## 요청받는 산출물

### 필수 (🔴)

| 산출물 | 형식 | 설명 |
|--------|------|------|
| **프로젝트 일정표** | 문서/툴 | 마일스톤 및 세부 일정 |
| **리소스 할당표** | 문서/툴 | 파트별 담당자 및 업무 분장 |
| **정기 리뷰 일정** | 캘린더 | 주간/스프린트 리뷰 |

### 권장 (🟡)

| 산출물 | 형식 | 설명 |
|--------|------|------|
| 회의록 | 문서 | 주요 결정사항 기록 |
| 이슈 트래킹 | 툴 | Jira, Notion 등 |
| 커뮤니케이션 채널 가이드 | 문서 | 상황별 채널 사용법 |

---

## PM이 조율해야 하는 주요 이슈

### 기획 ↔ 개발

| 이슈 | 중재 방향 |
|------|----------|
| 테스트베드 이후 기획 변경 | 변경 최소화 원칙 수립, 불가피 시 일정 재조율 |
| 아키텍처 근간을 흔드는 변경 | 초반 확정 유도, 변경 시 영향도 분석 |

### 디자인 ↔ 개발

| 이슈 | 중재 방향 |
|------|----------|
| UX/UI 수준 vs 개발 시간 | 팀 목표에 맞춰 조율 |
| Android 특성 미고려 | 플랫폼별 특성 고려하도록 조율 |

### 백엔드 ↔ 클라이언트

| 이슈 | 중재 방향 |
|------|----------|
| API 미공지 변경 | 변경 사전 공유 프로세스 수립 |
| 데이터 정제 책임 | 서버에서 정제하는 것으로 가이드 |

---

## 파트별 PM에게 원하는 것

| 파트 | 요청사항 |
|------|----------|
| **기획** | 실현 가능한 범위 조율, 일정 관리 |
| **디자인** | 초반 매니징 유연성, 다양한 산출물 고려한 일정 |
| **FE** | 초반 구조 확정, 기획 일관성 유지 |
| **iOS** | 컨셉 확정, 실현 가능 범위 조율 |
| **AOS** | 테스트베드 이후 변경 최소화, 배포 커뮤니케이션 |
| **BE** | 확장성 고려 시간, 기획 영향도 사전 공유 |

## 체크리스트

### 프로젝트 시작 시

- [ ] 파트별 인원 확인 및 역할 분담
- [ ] 초반 구조/컨셉 확정 유도
- [ ] 커뮤니케이션 채널 수립
- [ ] 정기 리뷰 일정 수립

### 프로젝트 진행 중

- [ ] 기획 변경 영향도 관리
- [ ] 파트 간 갈등 중재
- [ ] API 변경 공유 프로세스 관리
- [ ] 일정 조율 및 리스크 관리

### 배포 전

- [ ] 테스트베드 배포 후 변경사항 최소화
- [ ] QA 일정 확보
- [ ] 배포 커뮤니케이션
