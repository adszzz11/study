# 기획/PO 파트 요구 산출물 가이드

> 다른 파트에서 기획/PO에게 요청하는 산출물 정리

---

## 역할 구분

| 역할 | 핵심 업무 |
|------|----------|
| **PO** | 비즈니스 의사결정, 제품 방향성, "왜 만드는가" 결정 |
| **기획** | 요구사항 구체화, 정책 수립, "어떻게 만드는가" 정의 |

---

## 요청받는 산출물

### 필수 (🔴)

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| **Wireframe** | PPTX/Figma | 화면 구성의 기본 틀 | 전체 |
| **IA (Information Architecture)** | 다이어그램 | 전체 화면 구성도, 한눈에 볼 수 있게 | iOS, 전체 |
| **Flow Chart** | 다이어그램 | 사용자 흐름도 | FE, 전체 |

### 권장 (🟡)

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| 기능 명세서 | 문서 | 각 기능별 상세 설명 | FE |
| 정책 정의서 | 문서 | 날짜 표기, 문체, String Token 등 | iOS |
| User Story | 문서 | 사용자 관점의 기능 설명 | FE |
| 아이디어 기획서 | 문서 | 아이디어 + 진행 방향 + 일정 | 디자인 |
| 테스트 시나리오 | 문서 | QA 연계용 | QA |

### 선택 (🟢)

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| 레퍼런스 정리 | 문서/링크 | 참고할 서비스/디자인 | 디자인 |

---

## 정책 정의서 템플릿

```markdown
## 정책 정의서

### 1. 표기법
- 날짜: YYYY.MM.DD / YYYY-MM-DD
- 시간: 24시간제 / 12시간제 (오전/오후)
- 숫자: 천 단위 콤마 사용 여부

### 2. 문체
- 요체 (합니다/입니다)
- 다체 (한다/이다)

### 3. String Token 규칙
- 버튼: btn_[기능]_[상태]
- 라벨: lbl_[화면]_[위치]
- 메시지: msg_[유형]_[번호]

### 4. 에러 메시지 규칙
- 형식: [에러 코드] 사용자 친화적 메시지
```

## 다른 파트가 원하는 것

### 개발 파트 공통

| 요청사항 | 파트 |
|----------|------|
| 초반에 구조/컨셉 확정 | iOS, FE |
| 기획 의도를 명확하게 표현 | AOS |
| 테스트베드 이후 기획 변경 최소화 | AOS, FE |
| 실현 가능한 범위 내 기획 | iOS |

### 디자인 파트

| 요청사항 |
|----------|
| 의견 제시 시 이유/배경 설명 |
| 레퍼런스는 정돈된 형태로 전달 |
| UX/UI 통일성 고려 |

## 협업 시 주의사항

### DO (해주세요)

- [ ] 초반에 구조/컨셉 확정
- [ ] 기획 의도를 명확하게 표현 (맞고 틀리고 상관없이)
- [ ] 실현 가능한 범위 내에서 기획
- [ ] 의견 제시 시 이유와 배경 설명
- [ ] 레퍼런스는 정돈된 형태로 전달
- [ ] 기획 일관성 유지 (nav bar, baseline 등)

### DON'T (피해주세요)

- [ ] 테스트베드 이후 기획 변경
- [ ] 아키텍처 근간을 흔드는 후반 변경
- [ ] 근거 없는 고집
- [ ] 어지러운 레퍼런스 전달
- [ ] 기획 일관성이 떨어지는 산출물
