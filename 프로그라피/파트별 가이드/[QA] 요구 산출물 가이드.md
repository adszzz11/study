# QA 파트 요구 산출물 가이드

> 다른 파트에서 QA에게 요청하는 산출물 및 협업 가이드

---

## 핵심 역할

- 품질 보증
- 테스트 계획 및 실행
- 버그 리포트
- 결함 관리

---

## 요청받는 산출물

### 필수 (🔴)

| 산출물 | 설명 | 요청 파트 |
|--------|------|----------|
| **테스트 케이스 (시나리오)** | 기능별 테스트 항목 | iOS, AOS |
| **테스트 결과 리포트** | 실행 결과 | AOS |

### 권장 (🟡)

| 산출물 | 설명 | 요청 파트 |
|--------|------|----------|
| 에러 명세 | 에러 유형 및 대응 방안 | iOS |
| 상황 명세 | 엣지 케이스 정의 | iOS |
| QA 워크플로우 | 업무 프로세스 정의 | AOS |
| 결함 처리 방식 | 버그 리포트 및 처리 프로세스 | iOS |

---

## 품질 목표 (권장)

| 지표 | 목표 |
|------|------|
| 테스트 성공률 | 100% |
| Crash-free 비율 | 98% 이상 |

> 예측 못하는 이슈는 발생할 수 있으나 최대한 잡고 싶습니다

---

## QA 워크플로우 권장 구성

```
1. 테스트 케이스 작성
   ↓
2. 테스트 실행
   ↓
3. 결과 리포트 작성
   ↓
4. 버그 등록 (기기/환경 명시)
   ↓
5. 수정 확인
   ↓
6. 재테스트
```

> QA가 직접 업무 프로세스를 정의하면 더 명확해질 수 있습니다

### 필수 포함 항목

```markdown

[화면명] 간단한 버그 설명

### 환경
- 기기: (예: Galaxy S23, iPhone 15 Pro)
- OS 버전: (예: Android 14, iOS 17.2)
- 앱 버전: (예: 1.2.3)

### 재현 단계
1.
2.
3.

### 예상 결과


### 실제 결과


### 심각도
- [ ] Critical (앱 크래시, 데이터 손실)
- [ ] High (주요 기능 불가)
- [ ] Medium (기능 제한적 사용 가능)
- [ ] Low (사소한 UI 문제)

### 첨부
- 스크린샷/영상:
```

### 버그 리포트 예시

**❌ 개선 필요한 예시**

| 안 좋은 리포트 | 문제점 |
|---------------|--------|
| "안 돼요" | 구체적인 상황 없음 |
| "기획이랑 다른데요" | 기기/해상도 정보 없음 |
| "제 폰에서는 안 돼요" | 기기 모델명 없음 |

**✅ 좋은 예시**

```
Galaxy S23 (Android 14)에서 결제 버튼 클릭 시 앱이 크래시됩니다.

재현 단계:
1. 장바구니 진입
2. 결제하기 클릭
3. 크래시 발생

첨부: 크래시 로그, 스크린 녹화
```

```
기획서에는 A로 되어있는데, 해상도 320dp 이하 기기에서는
레이아웃이 깨져서 B로 대응하는 것이 어떨까요?

테스트 기기: Galaxy A10 (360x640)
```

## Android 테스트 시 고려사항

> AOS 개발자가 특히 요청하는 부분

### 기기 다양성

| 기기 유형 | 테스트 필요 수준 |
|----------|-----------------|
| 일반 스마트폰 | 기본 |
| **폴더블 (Fold)** | **필수** |
| 플립 | 선택 |
| **다양한 해상도** | **필수** |

### 해상도 관련

- 넓은 화면 / 좁은 화면 모두 테스트
- 해상도로 인한 이슈는 기획 문제가 아닐 수 있음
- 기획과 다르게 보인다면 기기/해상도 정보 함께 공유

## 파트별 QA에게 요청사항

### iOS가 원하는 것

| 요청사항 |
|----------|
| 구체적인 에러 상황 설명 |
| 재현 가능한 버그 리포트 |
| 에러 명세 작성 |
| 상황 명세 (엣지 케이스) 작성 |

### AOS가 원하는 것

| 요청사항 |
|----------|
| 자신만의 프로세스 정의 및 운영 |
| 구체적인 버그 리포트 (**기기 명시!**) |
| **모든 파트에 균등한 테스트** |
| 해상도 다양성 고려 |

---

## 협업 시 주의사항

### DO (해주세요)

- [ ] 구체적인 에러 상황 설명
- [ ] 재현 가능한 버그 리포트
- [ ] **기기/환경 정보 포함** (중요!)
- [ ] 모든 파트에 균등한 테스트
- [ ] 해상도 차이 고려
- [ ] 자신만의 QA 프로세스 정의

### DON'T (피해주세요)

- [ ] "안 돼요"만 말하기 (원인/상황 설명 필요)
- [ ] 기기 정보 없이 버그 리포트
- [ ] "기획에서는 저렇게 되어있는데"만 말하기 (해상도 다양성 고려)
- [ ] 특정 파트만 테스트 (다른 파트 서운함)
