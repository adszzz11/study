## 다른 파트에 요청하는 산출물

### 기획/PO에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| 기획서 (PPTX 선호) | 🔴 필수 | Figma는 제대로 쓸 게 아니면 지양 |
| 정책 정의서 | 🔴 필수 | 날짜 표기, 문체, String Token |
| IA (Information Architecture) | 🔴 필수 | 전체 화면 구성도 |
| 테스트 시나리오 | 🟡 권장 | QA 연계 |

**기획에게 요청사항:**
- [ ] 초반 구조를 확실히 잡기 (후반 롤백 방지)
- [ ] 컨셉을 명확히 정의 (초장에 잡기)
- [ ] 실현 가능한 범위 내에서 기획
- [ ] 기획 오버치지 않기

### 디자인에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| 디자인시스템 (토큰화) | 🔴 필수 | 컴포넌트 통일 |
| 컴포넌트 정의 | 🔴 필수 | 텍스트박스, 버튼 등 |
| Animation Diagram | 🟡 권장 | 인터랙션 정의 |

**디자인에게 요청사항:**
- [ ] 디자인시스템 기반 구성을 먼저 진행
- [ ] 토큰화가 잘 되어있으면 좋겠습니다
- [ ] 컴포넌트 통일화
- [ ] 구현 시간 고려한 대안 제시
- [ ] 과도한 Animation 지양

### 백엔드에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| Swagger API 명세 | 🔴 필수 | 지속적 업데이트 필수 |
| Flow Chart | 🟡 권장 | 로직 흐름도 |

**백엔드에게 요청사항:**
- [ ] 명세 문서 지속 업데이트
- [ ] **nullable 표시 명확히** (중요!)
- [ ] 로그인 토큰 갱신 로직 집중 설계
- [ ] 토큰은 DB에서 관리
- [ ] 소통 없는 API 변경 지양

### QA에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| 테스트 케이스 | 🔴 필수 | 기능별 테스트 항목 |
| 에러 명세 | 🟡 권장 | 에러 유형 및 대응 |
| 상황 명세 | 🟡 권장 | 엣지 케이스 정의 |
| 결함 처리 방식 | 🟡 권장 | 버그 리포트 프로세스 |

**QA에게 요청사항:**
- [ ] 구체적인 에러 상황 설명
- [ ] 재현 가능한 버그 리포트
- [ ] "안 돼요"만 말하지 말고 원인/상황 설명 포함

## 정책 정의서 필요 항목

iOS 개발 시 기획에서 정의해줘야 하는 항목:

```markdown
## 정책 정의서

### 1. 날짜 표기법
- 형식: YYYY.MM.DD / YYYY-MM-DD / YY.MM.DD

### 2. 문체
- 요체 (합니다/입니다)
- 다체 (한다/이다)

### 3. 디자인시스템 정의
- 컬러 토큰
- 타이포그래피

### 4. String Token 규칙
- 버튼: btn_[기능]_[상태]
- 라벨: lbl_[화면]_[위치]
- 메시지: msg_[유형]_[번호]
```

## 앱스토어 심사 체크리스트

### 심사 통과를 위한 필수 확인 사항

```markdown
## App Store Review Checklist

### 기능
- [ ] 앱 설명과 실제 기능 일치
- [ ] 모든 링크 정상 작동 (회원가입, 로그인 등)
- [ ] 데모 계정 준비 (로그인 필요 시)
- [ ] 크래시 없음 (Crashlytics 확인)

### 개인정보
- [ ] 개인정보 처리방침 URL 제공
- [ ] App Privacy 정보 정확히 입력
- [ ] 추적 권한 요청 시 ATT 구현

### 콘텐츠
- [ ] 저작권 문제 없는 콘텐츠
- [ ] 연령 등급 적절히 설정
- [ ] 불쾌한 콘텐츠 없음

### 기술
- [ ] IPv6 네트워크 지원
- [ ] 최신 SDK 사용 (보통 6개월 이내)
- [ ] 불필요한 권한 요청 없음
- [ ] In-App Purchase 정상 동작 (있는 경우)

### 메타데이터
- [ ] 스크린샷 실제 앱 화면
- [ ] 앱 이름에 불필요한 키워드 없음
- [ ] 앱 설명 명확하고 정확함
```

### 자주 리젝되는 사유

| 리젝 사유 | 대응 방안 |
|----------|----------|
| **Guideline 2.1** - 크래시/버그 | 충분한 QA 후 제출 |
| **Guideline 4.2** - 최소 기능 | 핵심 기능 완성 후 제출 |
| **Guideline 4.3** - 스팸 | 차별화 포인트 명확히 |
| **Guideline 5.1.1** - 개인정보 | 개인정보 처리방침 필수 |
| **Guideline 3.1.1** - IAP | 디지털 콘텐츠는 IAP 필수 |

## Swift 코드 컨벤션

### 네이밍 컨벤션

```swift
// ✅ 타입: UpperCamelCase
struct UserProfile { }
class NetworkManager { }
enum ButtonState { }
protocol Fetchable { }

// ✅ 변수/함수: lowerCamelCase
let userName: String
func fetchUserData() { }

// ✅ 상수: lowerCamelCase (static은 UpperCamelCase 가능)
let maximumRetryCount = 3
static let DefaultTimeout: TimeInterval = 30

// ✅ Bool 변수: is/has/should 접두사
var isLoading: Bool
var hasPermission: Bool
```
