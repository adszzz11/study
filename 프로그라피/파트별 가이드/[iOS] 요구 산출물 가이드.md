## 다른 파트에 요청하는 산출물

### 기획/PO에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| 기획서 (PPTX 선호) | 🔴 필수 | Figma는 제대로 쓸 게 아니면 지양 |
| 정책 정의서 | 🔴 필수 | 날짜 표기, 문체, String Token |
| IA (Information Architecture) | 🔴 필수 | 전체 화면 구성도 |
| 테스트 시나리오 | 🟡 권장 | QA 연계 |

**기획에게 요청사항:**
- [ ] 초반 구조를 확실히 잡기 (후반 롤백 방지)
- [ ] 컨셉을 명확히 정의 (초장에 잡기)
- [ ] 실현 가능한 범위 내에서 기획
- [ ] 기획 오버치지 않기

### 디자인에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| 디자인시스템 (토큰화) | 🔴 필수 | 컴포넌트 통일 |
| 컴포넌트 정의 | 🔴 필수 | 텍스트박스, 버튼 등 |
| Animation Diagram | 🟡 권장 | 인터랙션 정의 |

**디자인에게 요청사항:**
- [ ] 디자인시스템 기반 구성을 먼저 진행
- [ ] 토큰화가 잘 되어있으면 좋겠습니다
- [ ] 컴포넌트 통일화
- [ ] 구현 시간 고려한 대안 제시
- [ ] 과도한 Animation 지양

### 백엔드에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| Swagger API 명세 | 🔴 필수 | 지속적 업데이트 필수 |
| Flow Chart | 🟡 권장 | 로직 흐름도 |

**백엔드에게 요청사항:**
- [ ] 명세 문서 지속 업데이트
- [ ] **nullable 표시 명확히** (중요!)
- [ ] 로그인 토큰 갱신 로직 집중 설계
- [ ] 토큰은 DB에서 관리
- [ ] 소통 없는 API 변경 지양

### QA에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| 테스트 케이스 | 🔴 필수 | 기능별 테스트 항목 |
| 에러 명세 | 🟡 권장 | 에러 유형 및 대응 |
| 상황 명세 | 🟡 권장 | 엣지 케이스 정의 |
| 결함 처리 방식 | 🟡 권장 | 버그 리포트 프로세스 |

**QA에게 요청사항:**
- [ ] 구체적인 에러 상황 설명
- [ ] 재현 가능한 버그 리포트
- [ ] "안 돼요"만 말하지 말고 원인/상황 설명 포함

## 정책 정의서 필요 항목

iOS 개발 시 기획에서 정의해줘야 하는 항목:

```markdown
## 정책 정의서

### 1. 날짜 표기법
- 형식: YYYY.MM.DD / YYYY-MM-DD / YY.MM.DD

### 2. 문체
- 요체 (합니다/입니다)
- 다체 (한다/이다)

### 3. 디자인시스템 정의
- 컬러 토큰
- 타이포그래피

### 4. String Token 규칙
- 버튼: btn_[기능]_[상태]
- 라벨: lbl_[화면]_[위치]
- 메시지: msg_[유형]_[번호]
```
