# PM 파트 가이드

> 개발자/디자이너 인터뷰 기반 PM 역할 및 요구사항 정리

---

## 역할 정의

> PM은 **일정 조율의 마왕**, 실제로 일이 되게끔 만들어주는 사람

| 역할 | 핵심 업무 |
|------|----------|
| **PM** | 일정 조율, 리소스 관리, 실행 가능하게 조율 |
| **PO** | 비즈니스 의사결정, 제품 방향성 |
| **기획** | 요구사항 구체화, 정책 수립 |

## PM이 조율해야 하는 주요 이슈

| 이슈 유형 | 관련 파트 | 중재 방향 |
|----------|----------|----------|
| 테스트베드 이후 기획 변경 | AOS, FE | 변경 최소화 원칙 수립 |
| 아키텍처 근간을 흔드는 변경 | FE | 초반 확정 유도 |
| UX/UI 수준 vs 개발 시간 | FE, iOS | 팀 목표에 맞춰 조율 |
| API 미공지 변경 | 전체 | 사전 공유 프로세스 수립 |
| Android 특성 미고려 | AOS | 플랫폼별 특성 교육/조율 |

---

## 각 파트가 PM에게 원하는 것

| 파트 | 요청 사항 |
|------|----------|
| **기획** | 실현 가능한 범위 조율, 일정 관리 |
| **디자인** | 초반 매니징 유연성, 다양한 산출물 고려한 일정 |
| **FE** | 초반 구조 확정 유도, 기획 일관성 유지 |
| **iOS** | 컨셉 확정, 실현 가능 범위 조율 |
| **AOS** | 테스트베드 이후 변경 최소화, 배포 커뮤니케이션 |
| **BE** | 확장성 고려 시간, 기획 영향도 사전 공유 |

## PM이 알아야 할 파트별 호소 포인트

### 디자이너
- 시장조사, 설문 신뢰도 낮을 때 기한 연장 필요
- 할 일이 많음 (BI/CI, 디자인, 컨텐츠 등)

### 프론트엔드
- 세부정책 놓칠 수 있음 → 알려줘서 정하게끔 지도 필요

### 백엔드
- 확장성 고려한 설계 시간 필요
- 알림 등 지속적 데이터 관리 기능은 공수가 큼

## 밍글링 진행 가이드

PM은 킥오프 밍글링을 주도합니다:

1. **팀 구성 확인** (15분)
2. **프로젝트 개요 공유** (20분)
3. **산출물 합의** (30분) ⭐
4. **협업 원칙 합의** (20분)
5. **플랫폼 고려사항** (15분)
6. **도구 및 채널** (10분)
7. **정기 밍글링 일정** (5분)
