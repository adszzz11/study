# 통합 밍글링 프로세스

> 프로젝트 시작 전 팀 빌딩 협의 + 진행 중 정기 싱크업 프로세스

---

## 밍글링이란?

**밍글링(Mingling)**: 프로젝트를 함께 진행할 파트들이 모여 협업 방식, 산출물, 커뮤니케이션 채널 등을 합의하는 과정

### 밍글링의 두 가지 축

| 구분 | 설명 |
|------|------|
| **킥오프 밍글링** | 프로젝트 시작 전 초기 협의 (1회성) |
| **정기 밍글링** | 프로젝트 진행 중 정기 싱크업 (반복) |

---

### STEP 3 상세: 산출물 합의 체크리스트

> 밍글링에서 "만들자/말자"를 결정할 때 사용

#### 기획/PO 파트

**필수 (반드시 필요)**
- [ ] Wireframe 만들 수 있는가?
- [ ] IA (전체 화면 구성도) 만들 수 있는가?
- [ ] Flow Chart 만들 수 있는가?

**권장 (있으면 좋음)**
- [ ] 정책 정의서 (날짜, 문체, Token 규칙)
- [ ] 기능 명세서
- [ ] User Story

**합의 필요**
- 기획서 형식: PPTX / Figma / 문서
- 레퍼런스 공유 방식

#### PM 파트

**필수 (반드시 필요)**
- [ ] 프로젝트 일정표 만들 수 있는가?
- [ ] 리소스 할당표 만들 수 있는가?
- [ ] 정기 리뷰 일정 잡을 수 있는가?

**합의 필요**
- 이슈 트래킹 도구
- 회의록 관리 방식

#### 디자인 파트

**필수 (반드시 필요)**
- [ ] 디자인시스템 만들 수 있는가?
- [ ] Figma 디자인 명세 만들 수 있는가?
- [ ] 프로토타입 만들 수 있는가?

**권장 (있으면 좋음)**
- [ ] 디자인 플로우
- [ ] Animation Diagram

**합의 필요**
- 디자인 도구 (Figma 기본)
- 디자인 핸드오프 방식

#### 백엔드 파트

**필수 (반드시 필요)**
- [ ] Swagger API 명세 만들 수 있는가?
- [ ] nullable 표시 정확히 할 수 있는가?

**권장 (있으면 좋음)**
- [ ] OpenAPI Schema (TypeScript 타입 추출용)
- [ ] Flow Chart (로직 흐름도)
- [ ] 시스템 아키텍처 문서

**합의 필요**
- API 명세 도구 (Swagger / Postman)
- API 변경 공지 방식

#### QA 파트

**필수 (반드시 필요)**
- [ ] 테스트 케이스 만들 수 있는가?
- [ ] 버그 리포트 양식 준수할 수 있는가?

**권장 (있으면 좋음)**
- [ ] 에러 명세
- [ ] 상황 명세 (엣지 케이스)
- [ ] QA 워크플로우

**합의 필요**
- 버그 리포트 도구
- 테스트 결과 공유 방식

### STEP 4 상세: 협업 원칙 합의

#### 필수 합의 항목

| 항목 | 질문 | 권장 답변 |
|------|------|----------|
| 기획 변경 | 테스트베드 이후 기획 변경 정책은? | 변경 최소화, 불가피한 경우 영향도 분석 후 전파트 공유 |
| API 변경 | API 변경 시 언제, 어떻게 공유? | 최소 2일 전 공지, 변경 공지 템플릿 사용 |
| 버그 리포트 | 버그 리포트에 필수로 포함할 정보는? | 기기, OS 버전, 앱 버전, 재현 단계 |
| 정기 미팅 | 주간 리뷰 주기는? | 주 1회 |

#### 협업 원칙 합의서 템플릿

```markdown
## 협업 원칙 합의서

### 1. 기획 변경 정책
- 테스트베드 이후 기획 변경: [ ] 금지 / [ ] 최소화 (영향도 분석 필수)
- 기획 변경 시 공유 방식: _______________

### 2. API 변경 정책
- 변경 사전 공지 기한: ___일 전
- 변경 공지 채널: _______________
- 변경 공지 담당: _______________

### 3. 버그 리포트 정책
- 버그 리포트 도구: _______________
- 필수 포함 정보: [ ] 기기 [ ] OS 버전 [ ] 앱 버전 [ ] 재현 단계 [ ] 스크린샷
- 버그 심각도 기준: _______________

### 4. 정기 미팅
- 주간 리뷰: 매주 ___요일 ___시
- 스프린트 주기: ___주
- 스프린트 리뷰: _______________

### 5. 커뮤니케이션
- 주 커뮤니케이션 채널: _______________
- 긴급 연락 채널: _______________
- 문서 공유: _______________
```

## Part 2: 정기 밍글링

> 프로젝트 진행 중 정기적으로 진행하는 싱크업 미팅

### 정기 밍글링 종류

| 종류 | 주기 | 참여자 | 목적 |
|------|------|--------|------|
| **데일리 스탠드업** | 매일 | 파트별 | 당일 할 일 공유, 블로커 파악 |
| **주간 리뷰** | 주 1회 | 전 파트 | 진행 상황 공유, 이슈 논의 |
| **스프린트 리뷰** | 스프린트 종료 시 | 전 파트 | 결과 데모, 회고 |

---

### 데일리 스탠드업

**주기**: 매일 (10-15분)

**참여자**: 파트별 또는 전체 (팀 규모에 따라)

**형식**:
```
각자 3가지 공유:
1. 어제 한 일
2. 오늘 할 일
3. 블로커 (있다면)
```

**팁**:
- 15분 이내로 끝내기
- 상세 논의는 별도 미팅으로
- 서서 하면 짧아짐 (스탠드업)

---

### 주간 리뷰

**주기**: 주 1회 (60-90분)

**참여자**: 전 파트

**어젠다 템플릿**:

```markdown
## 주간 리뷰 - YYYY.MM.DD

### 1. 지난 주 완료 사항 (15분)
각 파트별 완료 사항 공유

### 2. 이번 주 계획 (15분)
각 파트별 이번 주 할 일

### 3. 이슈 및 블로커 (20분)
- 현재 막힌 것
- 도움 필요한 것
- 일정 리스크

### 4. 파트 간 협의 필요 사항 (20분)
- 기획 ↔ 개발 협의
- 디자인 ↔ 개발 협의
- 백엔드 ↔ 프론트 협의

### 5. 다음 주 주요 일정 (10분)
- 마일스톤
- 중요 미팅
- 휴가/부재

### Action Item
| 담당자 | 항목 | 마감일 |
|--------|------|--------|
| | | |
```

### 스프린트 리뷰 & 회고

**주기**: 스프린트 종료 시 (2-3주마다)

**참여자**: 전 파트

**Part 1: 스프린트 리뷰 (60분)**

```markdown
## 스프린트 리뷰

### 1. 완료된 기능 데모 (30분)
- 각 파트별 완료 기능 시연
- Q&A

### 2. 미완료 항목 (15분)
- 미완료 항목 목록
- 미완료 사유
- 다음 스프린트 이월 여부

### 3. 다음 스프린트 계획 (15분)
- 우선순위 높은 항목
- 예상 작업량
- 리스크 요소
```

**Part 2: 스프린트 회고 (30-60분)**

```markdown
## 스프린트 회고

### 1. 좋았던 점 (Keep)
- 계속 유지하고 싶은 것
- 잘 된 것

### 2. 아쉬웠던 점 (Problem)
- 개선이 필요한 것
- 힘들었던 것

### 3. 시도해볼 것 (Try)
- 다음 스프린트에 시도할 개선안
- Action Item

### Action Item
| 항목 | 담당자 | 다음 스프린트 목표 |
|------|--------|------------------|
| | | |
```

## 밍글링 FAQ

### Q: 킥오프 밍글링은 얼마나 걸리나요?
A: 보통 2시간 내외입니다. 파트가 많으면 2.5시간까지 걸릴 수 있습니다.

### Q: 모든 파트가 참석해야 하나요?
A: 네, 킥오프 밍글링은 모든 파트가 참석해야 합니다. 합의된 내용을 모두가 알고 있어야 하기 때문입니다.

### Q: 주간 리뷰를 매주 하기 어려우면?
A: 최소 격주로는 진행하세요. 2주 이상 싱크업이 없으면 파트 간 인식 차이가 커집니다.

### Q: 데일리 스탠드업은 꼭 해야 하나요?
A: 팀 규모와 상황에 따라 다릅니다. 소규모 팀은 필요 시 슬랙 공유로 대체 가능합니다.

### Q: 밍글링에서 결정된 내용을 바꿔도 되나요?
A: 네, 하지만 변경 시 전 파트에 공유하고 합의를 받아야 합니다.
