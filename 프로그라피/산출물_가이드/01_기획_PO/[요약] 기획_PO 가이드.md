# 기획/PO 파트 가이드

> 개발자/디자이너 인터뷰 기반 기획/PO 요구사항 정리

---

## 역할 구분

| 역할 | 핵심 업무 |
|------|----------|
| **PO** | 비즈니스 의사결정, "왜 만드는가" 결정 |
| **기획** | 요구사항 구체화, "어떻게 만드는가" 정의 |

---

## 필수 산출물

| 산출물 | 형식 | 요청 파트 |
|--------|------|----------|
| **Wireframe** | PPTX/Figma | 전체 |
| **IA (Information Architecture)** | 다이어그램 | iOS, 전체 |
| **Flow Chart** | 다이어그램 | FE, 전체 |

### 권장 산출물

| 산출물 | 형식 | 요청 파트 |
|--------|------|----------|
| 정책 정의서 | 문서 | iOS |
| 기능 명세서 | 문서 | FE |
| 아이디어 기획서 | 문서 | 디자인 |
| 레퍼런스 정리 | 문서/링크 | 디자인 |

---

## 정책 정의서 포함 항목

```markdown
### 표기법
- 날짜: YYYY.MM.DD / YYYY-MM-DD
- 시간: 24시간제 / 12시간제

### 문체
- 요체 (합니다/입니다) / 다체 (한다/이다)

### String Token 규칙
- 버튼: btn_[기능]_[상태]
- 라벨: lbl_[화면]_[위치]
```

---

## 개발자들이 기획에게 원하는 것

| 요청 파트 | 핵심 요청 |
|----------|----------|
| **FE** | 초반 구조 확정, 일관성 유지, 아키텍처 변경 최소화 |
| **iOS** | 초반 컨셉 확정, 실현 가능한 범위 조율 |
| **AOS** | 의도 명확히 전달, 테스트베드 이후 변경 최소화 |
| **디자인** | 근거 있는 제안, 정리된 레퍼런스 |

---

## DO (해주세요)

- [ ] **초반에 구조/컨셉 확정** - 후반 롤백 방지
- [ ] **기획 의도를 명확하게 표현** - 맞고 틀리고 상관없이
- [ ] **실현 가능한 범위 내에서 기획**
- [ ] **의견 제시 시 이유와 배경 설명**
- [ ] **레퍼런스는 정돈된 형태로 전달**

---

## DON'T (피해주세요)

- [ ] **테스트베드 이후 기획 변경** ⚠️ 개발자들 "척살 대상" 지목
- [ ] 아키텍처 근간을 흔드는 후반 변경
- [ ] 근거 없는 고집
- [ ] 어지러운 레퍼런스 전달
- [ ] 기획 오버치기 (과도한 기능 요구)

---

## PO 추가 주의사항 (디자이너 요청)

- 아이디어가 왜 좋은지 설명할 수 있게 준비
- 너무 컬러풀한 디자인 요구 지양 (통일성 저해)
- 새 컨셉 추가 시 기존 디자인시스템과의 조화 고려

---

## 체크리스트

### 기획서 작성 전
- [ ] 아키텍처 근간이 되는 구조가 명확한가?
- [ ] 실현 가능한 범위인가?
- [ ] 개발자/디자이너와 함께 리뷰했는가?

### 기획서 전달 시
- [ ] Wireframe이 포함되어 있는가?
- [ ] IA가 한눈에 보이는가?
- [ ] 정책이 명확하게 정의되어 있는가?

### 테스트베드 이후
- [ ] 기획 변경 요청이 있는가? → 영향도 분석 필수!
