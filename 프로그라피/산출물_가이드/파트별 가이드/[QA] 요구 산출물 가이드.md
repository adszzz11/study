# QA 파트 요구 산출물 가이드

> 다른 파트에서 QA에게 요청하는 산출물 및 협업 가이드

---

## 핵심 역할

- 품질 보증
- 테스트 계획 및 실행
- 버그 리포트
- 결함 관리

---

## 요청받는 산출물

### 필수 (🔴)

| 산출물 | 설명 | 요청 파트 |
|--------|------|----------|
| **테스트 케이스 (시나리오)** | 기능별 테스트 항목 | iOS, AOS |
| **테스트 결과 리포트** | 실행 결과 | AOS |

### 권장 (🟡)

| 산출물 | 설명 | 요청 파트 |
|--------|------|----------|
| 에러 명세 | 에러 유형 및 대응 방안 | iOS |
| 상황 명세 | 엣지 케이스 정의 | iOS |
| QA 워크플로우 | 업무 프로세스 정의 | AOS |
| 결함 처리 방식 | 버그 리포트 및 처리 프로세스 | iOS |

## 테스트 전략

### 테스트 유형별 가이드

| 테스트 유형 | 시점 | 담당 | 중요도 |
|------------|------|------|--------|
| **스모크 테스트** | 빌드 직후 | QA | 🔴 필수 |
| **기능 테스트** | 스프린트 중 | QA | 🔴 필수 |
| **회귀 테스트** | 릴리즈 전 | QA | 🔴 필수 |
| **탐색적 테스트** | 기능 완료 후 | QA | 🟡 권장 |
| **성능 테스트** | 베타 전 | QA/개발 | 🟡 권장 |

### 회귀 테스트 전략

```
1. 핵심 기능 정의
   - 로그인/회원가입
   - 메인 사용자 플로우
   - 결제/구매 기능 (있는 경우)

2. 회귀 테스트 스위트 구성
   - Critical Path: 매 릴리즈 실행
   - Full Regression: 메이저 릴리즈 실행

3. 자동화 대상 선정
   - 반복 실행되는 테스트
   - 시간이 오래 걸리는 테스트
   - 휴먼 에러가 발생하기 쉬운 테스트
```

### 테스트 커버리지 기준

| 영역 | 커버리지 목표 |
|------|-------------|
| 핵심 기능 (Critical Path) | 100% |
| 주요 기능 (Happy Path) | 90% 이상 |
| 예외 케이스 (Edge Case) | 70% 이상 |
| UI/UX | 주요 화면 100% |

## 테스트 자동화 가이드

### 자동화 도구 권장

| 플랫폼 | 도구 | 용도 |
|--------|------|------|
| Web | Playwright, Cypress | E2E 테스트 |
| iOS | XCTest, XCUITest | UI 자동화 |
| Android | Espresso, UI Automator | UI 자동화 |
| API | Postman, REST Assured | API 테스트 |
| 성능 | k6, JMeter | 부하 테스트 |

### 자동화 우선순위

```
높음 (먼저 자동화):
├── 로그인/인증 플로우
├── 핵심 비즈니스 로직
├── 데이터 CRUD 작업
└── 반복적인 회귀 테스트

중간:
├── 폼 유효성 검사
├── 네비게이션 테스트
└── 에러 핸들링

낮음 (수동 테스트 권장):
├── 시각적 UI 검증
├── 탐색적 테스트
└── 사용성 테스트
```

### E2E 테스트 예시 (Playwright)

```typescript
// login.spec.ts
import { test, expect } from '@playwright/test'

test('정상 로그인 시 메인 화면으로 이동', async ({ page }) => {
  // Given
  await page.goto('/login')

  // When
  await page.fill('[data-testid="email"]', 'user@test.com')
  await page.fill('[data-testid="password"]', 'password123')
  await page.click('[data-testid="login-button"]')

  // Then
  await expect(page).toHaveURL('/main')
  await expect(page.locator('[data-testid="welcome-message"]'))
    .toContainText('환영합니다')
})
```

## 성능 테스트 기준

### 모바일 앱 성능 기준

| 지표 | 기준 | 측정 방법 |
|------|------|----------|
| 앱 시작 시간 (Cold Start) | < 3초 | Firebase Performance |
| 화면 전환 시간 | < 1초 | Profiler |
| API 응답 대기 | < 2초 | Network Monitor |

| 배터리 소모 | 과도한 소모 없음 | Battery Historian |

### 웹 성능 기준

| 지표 | 기준 | 측정 방법 |
|------|------|----------|
| LCP | < 2.5초 | Lighthouse |
| FID | < 100ms | Lighthouse |
| CLS | < 0.1 | Lighthouse |
| TTI | < 3.5초 | Lighthouse |

## QA 워크플로우 권장 구성

```
1. 테스트 케이스 작성
   ↓
2. 테스트 실행
   ↓
3. 결과 리포트 작성
   ↓
4. 버그 등록 (기기/환경 명시)
   ↓
5. 수정 확인
   ↓
6. 재테스트
```

> QA가 직접 업무 프로세스를 정의하면 더 명확해질 수 있습니다

## Android 테스트 시 고려사항

> AOS 개발자가 특히 요청하는 부분

### 기기 다양성

| 기기 유형 | 테스트 필요 수준 |
|----------|-----------------|
| 일반 스마트폰 | 기본 |
| **폴더블 (Fold)** | **필수** |
| 플립 | 선택 |
| **다양한 해상도** | **필수** |

### 해상도 관련

- 넓은 화면 / 좁은 화면 모두 테스트
- 해상도로 인한 이슈는 기획 문제가 아닐 수 있음
- 기획과 다르게 보인다면 기기/해상도 정보 함께 공유

## 파트별 QA에게 요청사항

### iOS가 원하는 것

| 요청사항 |
|----------|
| 구체적인 에러 상황 설명 |
| 재현 가능한 버그 리포트 |
| 에러 명세 작성 |
| 상황 명세 (엣지 케이스) 작성 |

### AOS가 원하는 것

| 요청사항 |
|----------|
| 자신만의 프로세스 정의 및 운영 |
| 구체적인 버그 리포트 (**기기 명시!**) |
| **모든 파트에 균등한 테스트** |
| 해상도 다양성 고려 |

## 협업 시 주의사항

### DO (해주세요)

- [ ] 구체적인 에러 상황 설명
- [ ] 재현 가능한 버그 리포트
- [ ] **기기/환경 정보 포함** (중요!)
- [ ] 모든 파트에 균등한 테스트
- [ ] 해상도 차이 고려
- [ ] 자신만의 QA 프로세스 정의

### DON'T (피해주세요)

- [ ] "안 돼요"만 말하기 (원인/상황 설명 필요)
- [ ] 기기 정보 없이 버그 리포트
- [ ] "기획에서는 저렇게 되어있는데"만 말하기 (해상도 다양성 고려)
- [ ] 특정 파트만 테스트 (다른 파트 서운함)
