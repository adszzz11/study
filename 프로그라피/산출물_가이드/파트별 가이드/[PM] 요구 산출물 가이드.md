# PM 파트 요구 산출물 가이드

> 다른 파트에서 PM에게 요청하는 산출물 및 역할 정리

---

## 핵심 역할

> PM은 **일정 조율의 마왕**, 실제로 일이 되게끔 만드는 사람

- 일정 조율 및 리소스 관리
- 파트 간 갈등 중재
- 커뮤니케이션 프로세스 수립
- 리스크 관리

---

## 요청받는 산출물

### 필수 (🔴)

| 산출물 | 형식 | 설명 |
|--------|------|------|
| **프로젝트 일정표** | 문서/툴 | 마일스톤 및 세부 일정 |
| **리소스 할당표** | 문서/툴 | 파트별 담당자 및 업무 분장 |
| **정기 리뷰 일정** | 캘린더 | 주간/스프린트 리뷰 |

### 권장 (🟡)

| 산출물 | 형식 | 설명 |
|--------|------|------|
| 회의록 | 문서 | 주요 결정사항 기록 |
| 이슈 트래킹 | 툴 | Jira, Notion 등 |
| 커뮤니케이션 채널 가이드 | 문서 | 상황별 채널 사용법 |

## PM이 조율해야 하는 주요 이슈

### 기획 ↔ 개발

| 이슈 | 중재 방향 |
|------|----------|
| 테스트베드 이후 기획 변경 | 변경 최소화 원칙 수립, 불가피 시 일정 재조율 |
| 아키텍처 근간을 흔드는 변경 | 초반 확정 유도, 변경 시 영향도 분석 |

### 디자인 ↔ 개발

| 이슈 | 중재 방향 |
|------|----------|
| UX/UI 수준 vs 개발 시간 | 팀 목표에 맞춰 조율 |
| Android 특성 미고려 | 플랫폼별 특성 고려하도록 조율 |

### 백엔드 ↔ 클라이언트

| 이슈 | 중재 방향 |
|------|----------|
| API 미공지 변경 | 변경 사전 공유 프로세스 수립 |
| 데이터 정제 책임 | 서버에서 정제하는 것으로 가이드 |

## 파트별 PM에게 원하는 것

| 파트 | 요청사항 |
|------|----------|
| **기획** | 실현 가능한 범위 조율, 일정 관리 |
| **디자인** | 초반 매니징 유연성, 다양한 산출물 고려한 일정 |
| **FE** | 초반 구조 확정, 기획 일관성 유지 |
| **iOS** | 컨셉 확정, 실현 가능 범위 조율 |
| **AOS** | 테스트베드 이후 변경 최소화, 배포 커뮤니케이션 |
| **BE** | 확장성 고려 시간, 기획 영향도 사전 공유 |

## 리스크 관리 매트릭스

### 리스크 분류

| 리스크 유형 | 예시 | 대응 방안 |
|------------|------|----------|
| **일정 리스크** | 기능 개발 지연, 외부 의존성 | 버퍼 일정 확보, 우선순위 조정 |
| **인력 리스크** | 담당자 이탈, 병가 | 백업 담당자 지정, 지식 공유 |
| **기술 리스크** | 신기술 도입 실패, 성능 이슈 | PoC 진행, 대안 기술 준비 |
| **커뮤니케이션 리스크** | 파트 간 오해, 요구사항 불명확 | 정기 싱크업, 문서화 |

### 리스크 평가 기준

```
영향도 (Impact):
  High (3): 프로젝트 일정에 심각한 영향
  Medium (2): 일부 기능/일정에 영향
  Low (1): 경미한 영향

발생 가능성 (Probability):
  High (3): 발생 가능성 높음 (70%+)
  Medium (2): 발생 가능성 있음 (30-70%)
  Low (1): 발생 가능성 낮음 (<30%)

리스크 점수 = 영향도 × 발생 가능성
  7-9: 즉시 대응 필요 (🔴)
  4-6: 모니터링 필요 (🟡)
  1-3: 수용 가능 (🟢)
```

## 갈등 시나리오 및 해결 방안

### 시나리오 1: 기획 변경 갈등

```
상황: 테스트베드 이후 기획에서 핵심 플로우 변경 요청
- 개발: "이미 구현 완료. 변경 시 일정 2주 추가"
- 기획: "사용자 피드백 반영 필수"

해결 방안:
1. 변경 영향도 분석 (어떤 파트에 얼마나 영향?)
2. 비용 명확화 (일정, 인력, 품질 트레이드오프)
3. 의사결정권자(PO)에게 최종 판단 위임
4. 결정 후 전체 공유 및 일정 재조율
```

### 시나리오 2: 디자인 vs 개발 시간 갈등

```
상황: 디자인에서 복잡한 애니메이션 요청
- 개발: "구현에 3일 추가 소요"
- 디자인: "UX상 필수 요소"

해결 방안:
1. 팀 목표 재확인 (빠른 출시 vs 완성도)
2. 대안 제시 요청 (간소화 버전)
3. MVP에서 제외하고 다음 버전에 반영하는 옵션 고려
4. 합의된 방향 문서화
```

### 시나리오 3: API 변경 갈등

```
상황: 백엔드 API 변경을 사전 공지 없이 배포
- 클라이언트: "갑자기 앱이 안 됨"
- 백엔드: "급하게 수정이 필요했음"

해결 방안:
1. 긴급 롤백 또는 핫픽스 판단
2. 재발 방지 프로세스 수립
   - API 변경 최소 N일 전 공지 의무화
   - Breaking Change 시 버저닝 적용
3. 변경 공지 채널 및 담당자 지정
```

## 에스컬레이션 경로

```
Level 1: 파트 내 해결
    │ 해결 불가 시
    ▼
Level 2: PM 중재
    │ 해결 불가 시
    ▼
Level 3: PO/팀리드 의사결정
    │ 해결 불가 시
    ▼
Level 4: 멘토 자문
```

### 에스컬레이션 기준

| Level | 기준 | 예시 |
|-------|------|------|
| **L1** | 파트 내 기술적 이슈 | 코드 리뷰 의견 충돌 |
| **L2** | 파트 간 협업 이슈 | 일정 조율, API 명세 불일치 |
| **L3** | 프로젝트 방향성 결정 | 기능 우선순위, 핵심 변경 |
| **L4** | 팀 내 해결 불가 | 기술적 한계, 외부 의존성 |
