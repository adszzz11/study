---

## 업무 프로세스

```
1. Feature 정의 및 업무 분장
2. PR 리뷰 공동 진행
3. Wiki 정리 (순서도, 다이어그램 등)
```

## 다른 파트에 요청하는 산출물

### 기획에게

| 산출물 | 우선순위 | 비고 |
|--------|---------|------|
| 기획서 (PPTX 선호) | 필수 | Figma는 제대로 쓸 게 아니면 지양 |
| 정책 정의서 | 필수 | 날짜 표기, 문체, String Token |
| IA | 필수 | 전체 화면 구성도 |
| 테스트 시나리오 | 권장 | QA 연계 |

### 디자인에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| 디자인시스템 (토큰화) | 필수 | 컴포넌트 통일 |
| 컴포넌트 정의 | 필수 | 텍스트박스, 버튼 등 |

### 백엔드에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| Swagger API 명세 | 필수 | 지속적 업데이트 필수 |
| Flow Chart | 권장 | 로직 흐름도 |

---

## 기획에게 원하는 것

- [ ] **초반 구조를 확실히 잡기** (후반 롤백 방지)
- [ ] **컨셉을 명확히 정의**
- [ ] **실현 가능한 범위 내에서 기획**

### DON'T
- 과도한 기능 요구
- 벤치마킹이 아닌 단순 비교 요구

---

## 디자인에게 원하는 것

- [ ] **디자인시스템 기반 구성 먼저**
- [ ] **토큰화가 잘 되어있으면 좋겠음**
- [ ] **구현 시간 고려한 대안 제시**
- [ ] 화면 제작 시 정의나 예시 기반으로 작업

### DON'T
- 과도한 Animation 요구
- 구현 시간을 고려하지 않은 인터랙티브 요구

---

## 백엔드에게 원하는 것

- [ ] **명세 문서 지속 업데이트**
- [ ] **nullable 표시 명확히**
- [ ] **로그인 토큰 갱신 로직 설계** (DB 토큰 관리)
- [ ] 소통을 잘 하자

### DON'T
- 소통 없는 API 변경
- nullable 표시 누락
- 토큰을 DB에서 관리하지 않는 설계

---

## QA에게 원하는 것

- [ ] **구체적인 에러 상황 설명**
- [ ] **재현 가능한 버그 리포트**

### DON'T
- "안 돼요"만 말하고 끝내기 (원인/상황 설명 필요)
