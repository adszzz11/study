# 백엔드 파트 가이드

프론트엔드/모바일 개발자 인터뷰 기반 백엔드 요구사항 정리

---

## 필수 산출물

| 산출물 | 형식 | 요청 파트 |
|--------|------|----------|
| **API 명세** | Swagger | 전체 |
| **API 명세** | Postman/.http | FE |

### 권장 산출물

| 산출물 | 형식 | 요청 파트 |
|--------|------|----------|
| OpenAPI Schema | JSON/YAML | FE (타입 추출) |
| Flow Chart | 다이어그램 | iOS |
| 시스템 아키텍처 문서 | 문서 | 전체 |

---

## API 명세 필수 포함 항목

| 항목 | 설명 |
|------|------|
| **nullable, type 명시** | 각 필드의 타입과 null 허용 여부 명시 |
| 공통코드 설명 | 상세하게 작성 |
| API 설명 | 용도, 사용법 |
| 에러 코드 정의 | 에러 유형별 코드 |

---

## 각 파트가 백엔드에게 원하는 것

| 요청 파트 | 핵심 요청 |
|----------|----------|
| **FE** | JSON Depth 4 이하, 서버에서 데이터 정제, OpenAPI Schema |
| **iOS** | Swagger 업데이트, 명세 상세 작성, 토큰 갱신 로직, DB 토큰 관리 |
| **AOS** | 명세 상세 작성, Entity 원형 전달, 배포 커뮤니케이션, Schema 단순화 |

---

## DO (해주세요)

### API 설계
- **nullable, type 등 명세 상세 작성**
- **사용자(프론트엔드) 입장에서 API 설계**
- **JSON Depth 4단계 이하로 구성**
- **Entity 값을 원형으로 내려주기**

### 데이터 처리
- **데이터 정제는 서버 라인에서 처리** (백엔드가 동의함)

### 문서화
- **명세 문서 지속 업데이트**
- **API 설명 상세하게 작성**
- **공통코드 설명 포함**

### 인증/보안
- **로그인 토큰 갱신 로직 집중 설계** (DB 토큰 관리)

### 커뮤니케이션
- **배포 시 커뮤니케이션**
- **API 변경 시 사전 공유**

---

## DON'T (피해주세요)

- **미공지 API 변경**
- **명세 누락 (nullable, type 등)**
- Schema 과도하게 쪼개기
- 불명확한 공통코드 설명 누락
- 토큰을 DB에서 관리하지 않는 설계

---

## API 설계 가이드라인

### JSON Depth 권장

| 구분 | 예시 | 비고 |
|------|------|------|
| **Good (Depth 4 이하)** | user → profile → name | 권장 |
| **Bad (Depth 4 초과)** | data → user → profile → details → name | 지양 |

### Schema 설계 원칙
- Entity 값을 원형으로 내려주기
- 과도하게 쪼개지 않기
- 프론트에서 필요한 데이터 선별 가능하도록

---

## 기획에게 요청

- 확장성을 고려한 기획
- 데이터모델링에 영향 주는 결정은 초반에 확정
- 기획 시 개발 영향도 사전 문의

---

## 백엔드의 협조 요청 사항

- 확장성을 고려한 설계 시간 필요
- 기능별 개발 난이도 사전 인지 필요
  - 알림 기능: 지속적인 데이터 관리 필요
  - 조회 기능: 데이터 구조에 따라 난이도 상이
- 기획 시 개발 영향도가 필요한 경우 사전 문의 요청

---

## 협업 시 주의 사항

- API 파라미터를 공지 없이 변경하는 경우
- 파라미터 스펙을 자주 변경하는 경우
- 기획 변경이 데이터 구조에 영향을 주는 경우
