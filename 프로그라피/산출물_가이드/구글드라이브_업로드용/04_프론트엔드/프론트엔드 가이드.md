# 프론트엔드 파트 가이드

> 프론트엔드 멘토 인터뷰 기반 협업 요구사항 정리

---

## 필수 산출물

| 산출물 | 형식 | 설명 |
|--------|------|------|
| **화면 구현** | 코드 | Figma 기반 UI 구현 |

### 권장 산출물

| 산출물 | 형식 |
|--------|------|
| 컴포넌트 문서 | Storybook |
| 기술 문서 | 아키텍처, 폴더 구조 |

---

## 다른 파트에 요청하는 산출물

### 기획에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| Wireframe | 필수 | 화면 구성 파악 |
| Flow Chart | 필수 | 사용자 흐름 이해 |
| 기능 명세서 | 권장 | 상세 기능 파악 |

### 디자인에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| Figma 디자인 명세 | 필수 | UI 구현 기준 |
| 디자인시스템 | 필수 | 일관된 컴포넌트 구현 |
| 디자인 플로우 | 권장 | 인터랙션 이해 |

### 백엔드에게

| 산출물 | 우선순위 | 이유 |
|--------|---------|------|
| API 명세 (Swagger) | 필수 | API 연동 |
| OpenAPI Schema | 권장 | TypeScript 타입 추출 |

---

## 기획에게 원하는 것

- [ ] **초반에 아키텍처 근간이 되는 구조 정의**
- [ ] **기획 일관성 유지** (nav bar, baseline 등)
- [ ] Figma와 기획 문서의 연동
- [ ] 세부정책 놓치면 알려주세요

### DON'T
- 중간에 기획 변경 (특히 아키텍처 근간을 흔드는 변경)
- 기획 일관성이 떨어지는 산출물 전달

---

## 디자인에게 원하는 것

- [ ] **디자인시스템 기반 재사용 컴포넌트 구성**
- [ ] **라이브러리에서 재현 가능한 수준의 디자인**
- [ ] Theme, Token을 일관성 있게 구성
- [ ] **팀 목표에 맞는 디자인 수준 조율**
  - 빠른 출시가 목표일 때: 핵심 기능 중심
  - 예술성이 중요할 때: 디자인 강조 가능

### DON'T
- 개발자의 구현 수준을 벗어나는 과도한 UX/UI 요구

---

## 백엔드에게 원하는 것

- [ ] **JSON Depth 4단계 이하로 구성**
- [ ] **데이터 정제는 서버에서 처리**
- [ ] **nullable 명시**
- [ ] **API 변경 시 사전 공유**
- [ ] 공통코드 설명 상세 작성

### DON'T
- 미공지 API 변경
- 프론트에서 과도한 데이터 정제 요구

---

## 개발 환경 권장

| 용도 | 도구 |
|------|------|
| Mock 서버 | MSW (Mock Service Worker) |
| UI 라이브러리 | Material UI, Ant Design, shadcn/ui |

---

## 협업 시 주의 사항

### 기획자 관련
- 기획 없이 임의로 개발 진행
- 합의된 기획을 일방적으로 변경 요청

### 백엔드 관련
- API 파라미터를 공지 없이 변경
- 파라미터 스펙을 자주 변경

---

## 프론트엔드의 협조 요청 사항

- 세부정책 누락 가능성이 있으므로 발견 시 공유 요청
- 서버/클라이언트 간 데이터 처리 역할을 명확히 협의 필요
- 데이터 정제는 가능한 서버에서 처리하도록 협의 (백엔드 동의)
