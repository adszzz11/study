## 협업 파트별 필요 명세

### 기획 파트

#### 필수 산출물
| 우선순위 | 산출물 | 설명 |
|---------|--------|------|
| 1순위 | WireFrame | 화면 구성의 기본 틀 |
| 1순위 | Flow Chart | 사용자 흐름도 |
| 2순위 | 기능 명세서 | 각 기능별 상세 설명 |
| 2순위 | User Story | 사용자 관점의 기능 설명 |

#### 했으면 좋겠는 것
- 기획 초반에 아키텍처의 근간이 되는 구조를 명확히 정의
- 기획 일관성 유지 (nav bar, baseline 등 핵심 요소 유지)
- Figma와 기획 문서의 연동

#### 하지 않았으면 좋겠는 것
- 중간에 기획 변경 (특히 아키텍처 근간을 흔드는 변경)
- 기획 일관성이 떨어지는 산출물 전달

---

### 디자인 파트

#### 필수 산출물
| 우선순위 | 산출물 | 설명 |
|---------|--------|------|
| 1순위 | Figma 디자인 명세 | 컴포넌트 및 화면 디자인 |
| 2순위 | 디자인 플로우 | Interaction이 많은 경우 필수 |

#### 디자인 토큰 구성
- **Primitive Token**: RGBA 값 (기본 색상)
- **Semantic Token**: 의미 기반 정의 (정의하기 나름)

#### 했으면 좋겠는 것
- 디자인시스템 기반의 재사용 컴포넌트 구성
- Theme, Token을 일관성 있게 구성
- 일반적인 라이브러리(Material UI, Ant Design, shadcn/ui)에서 재현 가능한 수준의 디자인

#### 하지 않았으면 좋겠는 것
- 개발자의 구현 수준을 벗어나는 과도한 UX/UI 요구
- 팀 목표와 맞지 않는 디자인 요구
  - 빠른 출시가 목표일 때: 과도한 디자인 지양
  - 예술성이 중요할 때: 디자인 강조 가능

### 백엔드 파트

#### 필수 산출물
| 산출물            | 도구/형식                      |
| -------------- | -------------------------- |
| API 명세         | Swagger, Postman, .http 파일 |
| 통신 프로토콜 결정     | REST API / GraphQL / gRPC  |
| OpenAPI Schema | TypeScript 타입 추출용 문서       |

#### 했으면 좋겠는 것
- API 설명을 상세하게 작성 (공통코드 포함)
- 사용자(프론트엔드) 입장에서 API 설계
- JSON Depth 4단계 이하로 구성
- 데이터 정제는 웬만하면 서버 라인에서 처리

#### 하지 않았으면 좋겠는 것
- 미공지 API 변경
- 불명확한 공통코드 설명 누락
- 프론트에서 과도한 데이터 정제 요구
