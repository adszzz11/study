# 기획 파트 요구사항 명세서

> 개발자/디자이너 인터뷰 기반 기획 파트 요구사항 통합 정리

---

## 필수 산출물 체크리스트

### 1순위 (필수)

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| Wireframe | PPTX, Figma | 화면 구성의 기본 틀, 기획자 의도가 분명해야 함 | 전체 |
| Flow Chart | 다이어그램 | 사용자 흐름도 | FE |
| IA (Information Architecture) | 다이어그램 | 전체 화면 구성도, 한눈에 볼 수 있게 | iOS |

### 2순위 (권장)

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| 기능 명세서 | 문서 | 각 기능별 상세 설명 | FE |
| User Story | 문서 | 사용자 관점의 기능 설명 | FE |
| 정책 정의서 | 문서 | 날짜 표기법, 문체, String Token 등 | iOS |
| 테스트 시나리오 | 문서 | QA와 연계 | iOS |
| 아이디어 기획서 | 문서 | 아이디어 + 진행 방향 + 세부내용 | Designer |
| 레퍼런스 | 문서/링크 | 참고할 아이디어/디자인 | Designer |

---

## 정책 정의서 포함 항목

- [ ] 날짜 표기법 (YYYY-MM-DD 등)
- [ ] 문체 (요체/다체)
- [ ] 디자인시스템 정의
- [ ] String Token 규칙

---

## DO (했으면 좋겠는 것)

### 구조 및 설계
- [ ] **초반 구조를 확실히 잡기** - 후반 롤백 방지 (iOS, FE)
- [ ] **아키텍처의 근간이 되는 구조를 명확히 정의** (FE)
- [ ] **컨셉을 명확히 정의** - 초장에 잡기 (iOS)

### 일관성
- [ ] **기획 일관성 유지** - nav bar, baseline 등 핵심 요소 유지 (FE)
- [ ] Figma와 기획 문서의 연동 (FE)

### 커뮤니케이션
- [ ] **기획 의도를 명확하게 표현** - 맞고 틀리고는 상관없이 의도 전달 (AOS)
- [ ] **의견 제시 시 이유를 명확하게** - 설득력 있는 제안 (Designer)
- [ ] 실현 가능한 범위 내에서 기획 (iOS)

---

## 협업 시 주의사항

### 변경 관련
- [ ] 테스트베드 배포 후에는 기획 변경 최소화 부탁드립니다 (AOS)
- [ ] 아키텍처에 영향을 주는 변경은 초반에 확정해주시면 좋겠습니다 (FE)

### 범위 조율
- [ ] 일정 내 구현 가능한 범위로 조율 부탁드립니다 (iOS)
- [ ] 구현 가능성에 대해 함께 논의해주시면 감사하겠습니다 (iOS)

### 일관성
- [ ] 기획 일관성이 유지되면 개발 효율이 높아집니다 (FE)
- [ ] 레퍼런스는 정리된 형태로 전달해주시면 이해가 빠릅니다 (Designer)

### 커뮤니케이션
- [ ] 의견 제시 시 배경이나 이유를 함께 공유해주시면 좋겠습니다 (AOS, Designer)
- [ ] 벤치마킹 시 구체적인 참고 포인트를 알려주시면 도움이 됩니다 (iOS)

## 기획서 포함 권장 항목 (Designer 요청)

1. 아이디어가 좋은 이유
2. 진행 방향
3. 일정
4. 세부 내용

---

## 파트별 주요 요청 요약

| 파트 | 핵심 요청 |
|------|----------|
| **FE** | 초반 구조 확정, 일관성 유지, 아키텍처 변경 최소화 |
| **iOS** | 초반 컨셉 확정, 실현 가능한 범위 조율 |
| **AOS** | 의도 명확히 전달, 테스트베드 이후 변경 최소화 |
| **Designer** | 근거 있는 제안, 정리된 레퍼런스 |
