# 통합 협업 프로세스

> 프로그라피 프로젝트 협업 플로우 정의

---

## 핵심 원칙 4가지

| 원칙 | 핵심 메시지 |
|------|------------|
| **1. 초반 확정** | 구조, 컨셉, 기술스택은 프로젝트 시작 시 확정 |
| **2. 변경 최소화** | 테스트베드 이후 기획 변경 금지 |
| **3. 사전 공유** | API 변경, 배포 전 반드시 커뮤니케이션 |
| **4. 구체적 리포트** | 버그/이슈 시 기기/환경 정보 필수 포함 |

---

## 전체 프로젝트 플로우

```
┌─────────────────────────────────────────────────────────────────┐
│                     PHASE 1: 프로젝트 시작                        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  1단계: 요구사항 협의                                             │
│  • 영업/기술지원 → 요구사항 접수                                   │
│  • 요구사항 협의 회의 (전 파트 참여)                               │
│  • PO: 비즈니스 의사결정, "왜 만드는가" 확정                       │
│  [산출물] 아이디어 기획서, 레퍼런스                                │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  2단계: 기획 확정 [초반 확정 원칙]                                  │
│  • Wireframe, IA, Flow Chart 작성                               │
│  • 개발자 + 디자이너 + 기획자 함께 리뷰                            │
│  [산출물] Wireframe, IA, Flow Chart, 정책 정의서                 │
│  * 이 시점에서 아키텍처 근간이 되는 구조 확정                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  3단계: 디자인                                                    │
│  • 디자인시스템 구축 (Token, 컴포넌트)                             │
│  • 화면별 디자인 작업                                             │
│  [산출물] 디자인시스템, Figma 명세, 프로토타입                      │
│  * Android 특성 고려 (폴더블, 해상도 다양성)                        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  4단계: 개발                                                      │
│   ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐          │
│   │   BE   │◄──►│   FE   │    │  iOS   │    │  AOS   │          │
│   └────────┘    └────────┘    └────────┘    └────────┘          │
│  [BE 산출물] Swagger API 명세 (nullable 필수)                     │
│  * API 변경 시 사전 공유 필수                                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  5단계: 테스트베드 배포 [변경 금지 시점]                            │
│  • 내부 테스트용 배포                                              │
│  * 이 시점 이후 기획 변경 최소화 (개발 일정에 큰 영향)              │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  6단계: QA                                                        │
│  • 테스트 케이스 기반 테스트 실행                                   │
│  • 버그 리포트 (기기/환경 정보 필수!)                               │
│  * 구체적인 상황 설명 필수 (단순 보고 지양)                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  7단계: 배포                                                      │
│  • 배포 커뮤니케이션 (전 파트 공유)                                 │
│  * 배포 전 커뮤니케이션 필수                                       │
└─────────────────────────────────────────────────────────────────┘
```

## 파트 간 커뮤니케이션 플로우

```
기획 ────────────────► 디자인
     Wireframe, 정책

디자인 ───────────────► 개발
      디자인시스템, Figma 명세

백엔드 ───────────────► 클라이언트
      API 명세 (nullable!)
      변경 사전 공지

개발 ────────────────► QA
    테스트 빌드

QA ──────────────────► 개발
    버그 리포트 (기기/환경 명시)
```

## 정기 싱크업 구조

| 종류 | 주기 | 참여자 | 목적 |
|------|------|--------|------|
| 데일리 스탠드업 | 매일 | 파트별 | 당일 할 일 공유, 블로커 파악 |
| 주간 리뷰 | 주 1회 | 전 파트 | 진행 상황 공유, 이슈 논의 |
| 스프린트 리뷰 | 스프린트 종료 시 | 전 파트 | 결과 데모, 회고 |

## 변경 관리 프로세스

### API 변경 시

```
변경 필요성 발생 → 영향 범위 분석 → 변경 공지 (최소 2일 전) → 변경 적용
```

### 기획 변경 시 (테스트베드 이후)

```
변경 요청 → PM 변경 가능 여부 판단 → 영향도 분석 → 전파트 공유 → 일정 재조율
```

## 이슈 에스컬레이션

| Level | 대상 | 예시 |
|-------|------|------|
| 1 | 파트 내 해결 | 코드 버그, 디자인 수정 |
| 2 | 파트 간 협의 | API 스펙 협의, 디자인-개발 조율 |
| 3 | PM 중재 | 일정 조율, 우선순위 결정 |
| 4 | PO 결정 | 기능 삭제/추가, 방향성 변경 |
