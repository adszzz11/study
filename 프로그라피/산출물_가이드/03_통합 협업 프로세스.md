# 통합 협업 프로세스

> 프로그라피 프로젝트 협업 플로우 정의

---

## 핵심 원칙 4가지

| 원칙 | 핵심 메시지 |
|------|------------|
| **1. 초반 확정** | 구조, 컨셉, 기술스택은 프로젝트 시작 시 확정 |
| **2. 변경 최소화** | 테스트베드 이후 기획 변경 금지 |
| **3. 사전 공유** | API 변경, 배포 전 반드시 커뮤니케이션 |
| **4. 구체적 리포트** | 버그/이슈 시 기기/환경 정보 필수 포함 |

## 단계별 상세 프로세스

### Phase 1: 요구사항 협의

```
영업/기술지원 → 요구사항 접수
       ↓
요구사항 협의 회의
       ↓
PO: 비즈니스 의사결정 ("왜 만드는가")
       ↓
기획자: 요구사항 구체화 시작
```

**참여자**: PO, PM, 기획자

**산출물**:
- 아이디어 기획서 (왜 이 아이디어가 좋은지, 진행 방향, 일정)
- 레퍼런스 정리

**체크포인트**:
- [ ] 아이디어의 이유가 명확한가?
- [ ] 레퍼런스가 정돈되어 있는가?

---

### Phase 2: 기획 확정

```
Wireframe, IA, Flow Chart 작성
       ↓
정책 정의서 작성
       ↓
개발자 + 디자이너 + 기획자 함께 리뷰
       ↓
리뷰 수정사항 반영
       ↓
최종 기획서 확정
```

**참여자**: 기획자, 디자이너, 개발자 전체

**산출물**:
- Wireframe
- IA (Information Architecture)
- Flow Chart
- 정책 정의서

**체크포인트**:
- [ ] 아키텍처 근간이 되는 구조가 확정되었는가?
- [ ] 모든 파트가 리뷰에 참여했는가?
- [ ] 구현 가능한 범위인가?

---

### Phase 3: 디자인

```
디자인시스템 구축
       ↓
화면별 디자인 작업
       ↓
프로토타입 제작
       ↓
디자인 리뷰 (개발자 피드백)
       ↓
수정 및 확정
```

**참여자**: 디자이너, 개발자

**산출물**:
- 디자인시스템 (Token, 컴포넌트)
- Figma 디자인 명세
- 프로토타입
- Animation Diagram (필요시)

**체크포인트**:
- [ ] 디자인시스템이 토큰화되어 있는가?
- [ ] 라이브러리에서 재현 가능한 수준인가?
- [ ] Android 특성 (폴더블, 해상도)이 고려되었는가?
- [ ] 구현 시간이 고려되었는가?

---

### Phase 4: 개발

```
BE: API 설계 및 개발
       ↓
BE: Swagger 명세 작성 (nullable 필수!)
       ↓
FE/iOS/AOS: 화면 개발
       ↓
API 연동
       ↓
통합 테스트
```

**참여자**: 백엔드, 프론트엔드, iOS, AOS

**산출물**:
- Swagger API 명세
- 화면 구현
- 기술 문서

**체크포인트**:
- [ ] API 명세에 nullable이 표시되어 있는가?
- [ ] JSON Depth가 4 이하인가?
- [ ] API 변경 시 사전 공유했는가?

---

### Phase 5: 테스트베드 배포

```
내부 테스트용 배포
       ↓
기능 동작 확인
       ↓
⚠️ 이 시점 이후 기획 변경 최소화!
```

**참여자**: PM, 개발자 전체

**체크포인트**:
- [ ] 모든 기능이 동작하는가?
- [ ] 기획 변경 사항이 있는가? (있다면 영향도 분석 필수)

---

### Phase 6: QA

```
테스트 케이스 작성/검토
       ↓
테스트 실행
       ↓
버그 리포트 (기기/환경 필수!)
       ↓
버그 수정
       ↓
재테스트
       ↓
테스트 완료
```

**참여자**: QA, 개발자

**산출물**:
- 테스트 케이스
- 테스트 결과 리포트
- 버그 리포트

**체크포인트**:
- [ ] 버그 리포트에 기기/환경 정보가 포함되어 있는가?
- [ ] 재현 단계가 명확한가?
- [ ] 모든 파트가 균등하게 테스트되었는가?

---

### Phase 7: 배포

```
최종 점검
       ↓
배포 커뮤니케이션 (전 파트 공유)
       ↓
스토어/서버 배포
       ↓
배포 완료 공유
```

**참여자**: PM, 개발자 전체

**체크포인트**:
- [ ] 모든 파트에 배포 일정이 공유되었는가?
- [ ] 롤백 계획이 있는가?

## 파트 간 커뮤니케이션 플로우

```
기획 ────────────────► 디자인
     Wireframe, 정책
     정돈된 레퍼런스

디자인 ───────────────► 개발
      디자인시스템
      Figma 명세
      (Android 특성 고려)

백엔드 ───────────────► 클라이언트
      API 명세 (nullable!)
      변경 사전 공지

개발 ────────────────► QA
    테스트 빌드
    테스트 시나리오

QA ──────────────────► 개발
    버그 리포트
    (기기/환경 명시)
```

## 정기 싱크업 구조

### 주간 리뷰 미팅

| 항목 | 내용 |
|------|------|
| 주기 | 주 1회 |
| 참여자 | 전 파트 |
| 목적 | 진행 상황 공유, 이슈 논의 |

**어젠다**:
1. 지난 주 완료 사항
2. 이번 주 계획
3. 이슈 및 블로커
4. 파트 간 협의 필요 사항

### 스프린트 리뷰

| 항목 | 내용 |
|------|------|
| 주기 | 스프린트 종료 시 |
| 참여자 | 전 파트 |
| 목적 | 스프린트 결과 데모, 회고 |

**어젠다**:
1. 완료된 기능 데모
2. 미완료 항목 및 사유
3. 다음 스프린트 계획
4. 회고 (좋았던 점, 개선할 점)

## 변경 관리 프로세스

### API 변경 시

```
변경 필요성 발생
       ↓
영향 범위 분석
       ↓
변경 공지 (최소 2일 전)
       ↓
클라이언트 대응 일정 협의
       ↓
변경 적용
       ↓
변경 완료 공지
```

**API 변경 공지 템플릿**:

```markdown
## API 변경 공지

### 변경 일시
- 예정일: YYYY-MM-DD
- 배포 시간: HH:MM

### 변경 내용
#### 추가된 API
| Endpoint | Method | 설명 |
|----------|--------|------|

#### 수정된 API
| Endpoint | 변경 전 | 변경 후 | 영향 범위 |
|----------|--------|--------|----------|

#### 삭제된 API
| Endpoint | 대체 방안 |
|----------|----------|

### 영향받는 화면/기능

### 클라이언트 대응 필요 사항
```

### 기획 변경 시 (테스트베드 이후)

```
변경 요청 발생
       ↓
PM: 변경 가능 여부 판단
       ↓
(변경 불가능) → 다음 버전으로 연기
(변경 가능) → 영향도 분석
       ↓
모든 파트에 영향도 공유
       ↓
일정 재조율
       ↓
변경 적용
```

## 이슈 에스컬레이션

### Level 1: 파트 내 해결

- 담당 파트 내에서 자체 해결
- 예: 코드 버그, 디자인 수정

### Level 2: 파트 간 협의

- 관련 파트 담당자 간 직접 협의
- 예: API 스펙 협의, 디자인-개발 조율

### Level 3: PM 중재

- PM이 중재하여 해결
- 예: 일정 조율, 우선순위 결정, 파트 간 갈등

### Level 4: PO 결정

- PO가 비즈니스 관점에서 최종 결정
- 예: 기능 삭제/추가, 방향성 변경
