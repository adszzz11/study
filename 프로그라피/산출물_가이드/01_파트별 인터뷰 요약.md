## 기획 파트

### 인터뷰 대상
- 개발자/디자이너들이 기획 파트에 요청하는 사항

### 핵심 메시지

> "초반에 구조를 확실히 잡아주세요. 나중에 롤백하기 힘들어요."

### 각 파트에서 기획에게 원하는 것

| 요청 파트 | 핵심 요청 |
|----------|----------|
| **FE** | 초반 구조 확정, 일관성 유지, 아키텍처 변경 최소화 |
| **iOS** | 초반 컨셉 확정, 실현 가능한 범위 조율 |
| **AOS** | 의도 명확히 전달, 테스트베드 이후 변경 최소화 |
| **디자인** | 근거 있는 제안, 정리된 레퍼런스 |

### DO (해주세요)

- 초반에 구조/컨셉 확정
- 기획 의도를 명확하게 표현 (맞고 틀리고 상관없이)
- 실현 가능한 범위 내에서 기획
- 의견 제시 시 이유와 배경 설명
- 레퍼런스는 정돈된 형태로 전달

### DON'T (피해주세요)

- 테스트베드 이후 기획 변경 (개발자들 "척살 대상"으로 지목)
- 아키텍처 근간을 흔드는 후반 변경
- 근거 없는 고집
- 어지러운 레퍼런스 전달
- 기획 오버치기 (과도한 기능 요구)

### 기획자가 팀에게 호소한 것

- 백엔드에게: "어디까지 기획했을 때 개발 부담이 될지 가이드해주세요"
- 프론트에게: "세부정책 놓칠 수 있으니, 발견 시 알려주세요"

## PO 파트

### 인터뷰 대상
- 디자이너가 PO에게 요청하는 사항

### 역할 정의

> "PO는 비즈니스 관점에서 '왜 이 제품을 만들어야 하는지'를 결정하는 사람"

```
영업 (요청): "초코라떼 만들어줘, 팔고 싶어"
    ↓
PO (결정): "카페 매출 올리려면 신제품을 만들자"
    ↓
기획 (구체화): "코코아랑 우유 섞으세요"
    ↓
PM (실행): "당신이 커피 만드세요"
```

### 디자이너가 PO에게 원하는 것

| 항목 | 요청 내용 |
|------|----------|
| 명확한 이유 | 의견 제시 시 배경과 이유를 함께 설명 |
| 정돈된 레퍼런스 | 어지러운 아이디어/디자인 예시 지양 |
| UX/UI 통일성 고려 | 새 컨셉 추가 시 기존 시스템과의 조화 |

### DO (해주세요)

- 아이디어가 왜 좋은지 설명할 수 있게 준비
- 레퍼런스 정리해서 전달
- 설득력 있는 제안

### DON'T (피해주세요)

- 이유 없이 의견만 던지기
- 화면이 어지러운 아이디어/디자인 예시
- 너무 컬러풀한 디자인 요구 (통일성 저해)
- 기존 디자인시스템과 맞지 않는 새 컨셉 무작정 추가

## PM 파트

### 인터뷰 대상
- 개발자/디자이너들이 PM에게 기대하는 역할

### 역할 정의

> "PM은 **일정 조율의 마왕**, 실제로 일이 되게끔 만들어주는 사람"

### PM이 조율해야 하는 주요 이슈

| 이슈 유형 | 관련 파트 | 중재 방향 |
|----------|----------|----------|
| 테스트베드 이후 기획 변경 | AOS, FE | 변경 최소화 원칙 수립 |
| 아키텍처 근간을 흔드는 변경 | FE | 초반 확정 유도 |
| UX/UI 수준 vs 개발 시간 | FE, iOS | 팀 목표에 맞춰 조율 |
| API 미공지 변경 | 전체 | 사전 공유 프로세스 수립 |
| Android 특성 미고려 | AOS | 플랫폼별 특성 교육/조율 |

### 각 파트가 PM에게 원하는 것

| 파트 | 요청 사항 |
|------|----------|
| **기획** | 실현 가능한 범위 조율, 일정 관리 |
| **디자인** | 초반 매니징 유연성, 다양한 산출물 고려한 일정 |
| **FE** | 초반 구조 확정, 기획 일관성 유지 |
| **iOS** | 컨셉 확정, 실현 가능 범위 조율 |
| **AOS** | 테스트베드 이후 변경 최소화, 배포 커뮤니케이션 |
| **BE** | 확장성 고려 시간, 기획 영향도 사전 공유 |

### PM이 알아야 할 파트별 호소 포인트

- **디자이너**: 시장조사, 설문 신뢰도 낮을 때 기한 연장 필요. 할 일이 많음 (BI/CI, 디자인, 컨텐츠 등)
- **프론트엔드**: 세부정책 놓칠 수 있음 → 알려줘서 정하게끔 지도 필요
- **백엔드**: 확장성 고려한 설계 시간 필요. 알림 등 지속적 데이터 관리 기능은 공수가 큼

## 디자인 파트

### 인터뷰 대상
- 디자이너 본인 + 개발자들이 디자인에 요청하는 사항

### 디자이너 본인 인터뷰 핵심

> "UX/UI 디자인 통일성이 가장 중요합니다"

- 1년차 디자이너 관점
- UX/UI 디자인 + Branding + Content 디자인 병행
- 순간적인 아이디에이션으로 새 디자인 제시 시 주의 필요

### 디자이너가 팀에게 호소한 것

- 시장조사, 설문 신뢰도 낮을 때 기한이 늘어날 수 있음
- 초반에는 매니징을 좀 덜 해도 이해 필요
- 왔다갔다 하는 작업이 좀 많아도 이해 부탁
- UI/UX는 디자이너가 가져가야 함 (다만 너무 고집부리지는 않겠음)

### 개발자들이 디자인에게 원하는 것

| 요청 파트 | 핵심 요청 |
|----------|----------|
| **FE** | 디자인시스템 기반, 라이브러리 재현 가능 수준, Token 일관성 |
| **iOS** | 토큰화, 컴포넌트 통일, 구현 시간 고려, Animation 조율 |
| **AOS** | Android 특성 고려, 폴더블 대응, 플랫폼별 맞춤 디자인 |

### DO (해주세요)

- 디자인시스템 기반 구성을 먼저 진행
- Theme, Token을 일관성 있게 구성
- 라이브러리에서 재현 가능한 수준의 디자인 (Material UI, Ant Design, shadcn/ui)
- 구현 시간 고려한 대안 제시
- Android 특성 고려 (폴더블, 해상도 다양성)

### DON'T (피해주세요)

- iPhone 디자인을 Android에 그대로 적용
- 과도한 Animation 요구 (구현 시간 미고려)
- 디자인시스템과 맞지 않는 새 컨셉 무작정 추가
- 너무 컬러풀한 디자인 (통일성 저해)
- 폴더블(Fold) 기기 미고려

## 프론트엔드 파트

### 인터뷰 대상
- 프론트엔드 멘토

### 핵심 메시지

> "기획 초반에 아키텍처의 근간이 되는 구조를 명확히 정의해주세요"

### FE가 각 파트에게 원하는 것

**기획에게**
- 초반에 아키텍처 근간이 되는 구조 정의
- 기획 일관성 유지 (nav bar, baseline 등)
- Figma와 기획 문서의 연동

**디자인에게**
- 디자인시스템 기반 재사용 컴포넌트 구성
- 일반적인 라이브러리에서 재현 가능한 수준의 디자인
- 팀 목표에 맞는 디자인 수준 조율

**백엔드에게**
- JSON Depth 4단계 이하로 구성
- 데이터 정제는 서버에서 처리
- nullable 명시
- API 변경 시 사전 공유

### FE가 팀에게 호소한 것

- 세부정책 놓칠 수 있음 → 발견 시 알려주세요
- 서버에서 처리해야 할 것과 프론트에서 처리해야 할 것을 명확히 협의하고 싶음
- 웬만하면 서버 라인에서 정제하도록 강제하면 좋겠음 (백엔드가 동의함)

### 긁힘 포인트 (힘들어하는 상황)

- 기획자가 자기 마음대로 개발할 때
- 귀찮아서 기능 개발 못한다고 거짓말할 때
- 기획된 것을 자꾸 없애자고 주장할 때
- 백엔드가 API param을 몰래 하나씩 없앨 때
- 파라미터를 자꾸 바꿀 때

## iOS 파트

### 인터뷰 대상
- iOS 멘토 (팀 구성 3명 기준)

### 업무 프로세스

```
1. Feature 정의 및 업무 분장
2. PR 리뷰 공동 진행
3. Wiki 정리 (순서도, 다이어그램 등)
```

### 프로젝트 초기 세팅

- 모듈화 (Modulation)
- 프레임워크 선정
- 아키텍처 설계
- CI/CD 자동화 세팅

### 배포 일정 가이드

| 상황 | 권장 여유 기간 |
|------|---------------|
| 신규 앱 | 최소 1개월 (서버 안정화 포함) |
| 신규 Feature | 1~2주 전 |

### iOS가 각 파트에게 원하는 것

**기획에게**
- 초반 구조를 확실히 잡기 (후반 롤백 방지)
- 컨셉을 명확히 정의
- 실현 가능한 범위 내에서 기획
- 기획서는 PPTX 선호 (Figma는 제대로 쓸 게 아니면 지양)

**디자인에게**
- 디자인시스템 기반 구성 먼저
- 토큰화가 잘 되어있으면 좋겠음
- 구현 시간 고려한 대안 제시
- 과도한 Animation 지양

**백엔드에게**
- 명세 문서 지속 업데이트
- nullable 표시 명확히
- 로그인 토큰 갱신 로직 설계 (DB 토큰 관리)
- 소통 없는 API 변경 지양

**QA에게**
- 구체적인 에러 상황 설명
- 재현 가능한 버그 리포트
- "안 돼요"만 말하고 끝내지 않기

## AOS 파트

### 인터뷰 대상
- AOS 멘토 (모바일 멀티플랫폼 환경)

### 업무 프로세스

```
1. PM 업무 하달
2. 테크리드/챕터리드의 인원 & 공수 산정
3. 스프린트 리뷰 (기획자 소통)
4. 스프린트 시작 (백엔드/디자이너/기획자/QA 소통)
5. 배포
```

### Android 특수 고려사항

| 기기 유형 | 고려 수준 | 비고 |
|----------|----------|------|
| 일반 스마트폰 | 기본 | |
| 폴더블 (Fold) | **필수** | 반드시 고려 |
| 플립 | 선택 | 상대적으로 여유 |
| 특수 해상도 | 케이스별 협의 | |

### AOS가 각 파트에게 원하는 것

**기획에게**
- 기획 의도를 명확하게 표현 (맞고 틀리고 상관없이)
- 테스트베드 이후 기획 변경 지양 ("척살 대상")
- 근거 없는 고집 지양

**디자인에게**
- Android 특성 고려 (iOS와 다름을 인지해달라!)
- 폴더블(Fold) 기기 고려
- 다양한 화면 크기 고려
- iPhone 디자인 그대로 적용 지양

**백엔드에게**
- nullable 표시 정확하게
- 배포 시 커뮤니케이션
- Entity 값을 원형으로 내려주기
- Schema 과도하게 쪼개지 않기

**QA에게**
- 구체적인 버그 리포트 (기기 명시)
- 모든 파트에 균등한 테스트 (서운함 유발하지 말아주세요)
- 해상도 차이로 인한 이슈 고려

### AOS 개발자의 호소

> "Android 개발자라는 것을 인지해주세요. 폴더블 기기를 고려해주세요. 화면 크기 다양성을 고려해주세요."

## 백엔드 파트

### 인터뷰 대상
- 프론트엔드/모바일 개발자들이 백엔드에 요청하는 사항

### 가장 많이 언급된 요구사항

> **nullable 표시** - FE, iOS, AOS 전체에서 공통적으로 가장 많이 요청

### 각 파트가 백엔드에게 원하는 것

| 요청 파트 | 핵심 요청 |
|----------|----------|
| **FE** | JSON Depth 4 이하, 서버에서 데이터 정제, OpenAPI Schema, 공통코드 설명 |
| **iOS** | Swagger 업데이트, nullable 명시, 토큰 갱신 로직, DB 토큰 관리 |
| **AOS** | nullable 명시, Entity 원형 전달, 배포 커뮤니케이션, Schema 단순화 |

### DO (해주세요)

- nullable 표시 정확하게 (가장 중요!)
- 사용자(프론트엔드) 입장에서 API 설계
- JSON Depth 4단계 이하로 구성
- 데이터 정제는 서버 라인에서 처리
- 명세 문서 지속 업데이트
- 배포 시 커뮤니케이션
- 로그인 토큰 갱신 로직 집중 설계 (DB 토큰 관리)

### DON'T (피해주세요)

- 미공지 API 변경
- 불명확한 공통코드 설명 누락
- Schema 과도하게 쪼개기
- 토큰을 DB에서 관리하지 않는 설계

### 백엔드가 기획에게 호소한 것

- 확장성을 고려한 기획 부탁
- 데이터모델링에 영향 주는 결정은 초반에 확정
- 기획 시 개발 영향도 사전 문의
- 알림 기능은 지속적인 데이터 관리 필요 (공수가 큼)

### 긁힘 포인트 (힘들어하는 상황)

- 기획자가 API param을 자꾸 몰래 하나씩 없애는 경우
- 파라미터를 자꾸 바꾸는 경우
- 기획 변경이 데이터 구조에 영향을 주는 경우

## QA 파트

### 인터뷰 대상
- 모바일 개발자들이 QA에 요청하는 사항

### 품질 목표 (AOS 요청)

| 지표 | 목표 |
|------|------|
| 테스트 성공률 | 100% |
| Crash-free 비율 | 98% 이상 |

### 각 파트가 QA에게 원하는 것

| 요청 파트 | 핵심 요청 |
|----------|----------|
| **iOS** | 재현 가능한 버그 리포트, 에러/상황 명세, 구체적인 상황 설명 |
| **AOS** | 자체 프로세스 정의, 기기 명시, 해상도 고려, 균등한 테스트 |

### DO (해주세요)

- 구체적인 에러 상황 설명
- 재현 가능한 버그 리포트 (기기/환경 정보 포함)
- 자신만의 프로세스 정의 및 운영 (AOS가 권장)
- 모든 파트에 균등한 테스트

### DON'T (피해주세요)

- "안 돼요"만 시전하기 (원인/상황 설명 필요)
- "기획에서는 저렇게 되어있는데"만 말하기 (해상도 다양성 고려 필요)
- 특정 파트만 테스트 누락 (서운함 유발)
- 기기 정보 없이 버그 리포트

### 버그 리포트 예시

**Bad Case**
- "안 돼요" → 구체적인 상황 추가 필요
- "기획이랑 다른데요" → 기기/해상도 정보 추가 필요
- "제 폰에서는 안 돼요" → 기기 모델명 추가 필요

**Good Case**
- "Galaxy S23 (Android 14)에서 결제 버튼 클릭 시 앱이 크래시됩니다. 재현 단계: 1. 장바구니 진입 2. 결제하기 클릭 3. 크래시 발생"
- "기획서에는 A로 되어있는데, 해상도 320dp 이하 기기에서는 레이아웃이 깨져서 B로 대응하는 것이 어떨까요?"

## 인터뷰 종합 인사이트

### 모든 파트가 공통으로 강조한 것

1. **초반 확정**: 구조, 컨셉, 기술스택은 프로젝트 시작 시 확정
2. **사전 커뮤니케이션**: 변경 사항은 사전에 공유
3. **구체적인 정보**: 버그 리포트, 요청사항 모두 구체적으로

### 가장 많이 언급된 Pain Point

| 순위 | Pain Point | 관련 파트 |
|------|------------|----------|
| 1 | nullable 미표시 | BE → FE, iOS, AOS |
| 2 | 테스트베드 이후 기획 변경 | 기획 → 개발 전체 |
| 3 | Android 특성 미고려 | 디자인 → AOS |
| 4 | API 미공지 변경 | BE → 클라이언트 전체 |
| 5 | 불구체적인 버그 리포트 | QA → 개발 전체 |
