# PM(Project Manager) 파트 요구사항 명세서

> 개발자/디자이너 인터뷰 기반 PM 파트 요구사항 통합 정리

---

## 역할 정의

### PM의 핵심 역할

> PM은 **일정 조율의 마왕**, 실제로 일이 되게끔 만들어주는 사람

| 역할 | 핵심 업무 |
|------|----------|
| **PM** | 일정 조율, 리소스 관리, 실행 가능하게 조율 |
| **PO** | 비즈니스 의사결정, 제품 방향성 |
| **기획** | 요구사항 구체화, 정책 수립 |

---

## 업무 프로세스

### 일반적인 프로젝트 플로우

```
영업/기술지원 → 요구사항 접수
       ↓
요구사항 협의 회의
       ↓
기획자가 구체화해서 기획 진행
       ↓
개발자 + 디자이너 + 기획자 함께 리뷰
       ↓ (리뷰 수정사항 반영)
최종 기획서 기반 디자이너 디자인
       ↓
개발
       ↓
QA → 배포
```

## 필수 산출물 체크리스트

### 프로젝트 관리

| 산출물 | 형식 | 설명 |
|--------|------|------|
| 프로젝트 일정표 | 문서/툴 | 전체 마일스톤 및 세부 일정 |
| 리소스 할당표 | 문서/툴 | 파트별 담당자 및 업무 분장 |
| 리뷰 일정 | 캘린더 | 정기적인 리뷰 미팅 일정 |

### 커뮤니케이션

| 산출물 | 형식 | 설명 |
|--------|------|------|
| 회의록 | 문서 | 주요 결정사항 기록 |
| 이슈 트래킹 | 툴 | Jira, Redmine 등 |
| 진행 상황 공유 | 문서/슬랙 | 정기적인 상태 업데이트 |

## PM이 조율해야 하는 파트 간 요구사항

### 기획 ↔ 개발

| 요청 파트 | 기획에게 요청하는 사항 |
|----------|----------------------|
| **FE** | 초반 구조 확정, 일관성 유지, 아키텍처 변경 최소화 |
| **iOS** | 초반 컨셉 확정, 실현 가능한 범위 조율 |
| **AOS** | 의도 명확히 전달, 테스트베드 이후 변경 최소화 |

### 디자인 ↔ 개발

| 요청 파트 | 디자인에게 요청하는 사항 |
|----------|----------------------|
| **FE** | 디자인시스템 기반, 라이브러리 재현 가능 수준 |
| **iOS** | 토큰화, 컴포넌트 통일, 구현 시간 고려 |
| **AOS** | Android 특성 고려, 폴더블 대응 |

### 백엔드 ↔ 프론트엔드/모바일

| 요청 파트 | 백엔드에게 요청하는 사항 |
|----------|----------------------|
| **전체** | nullable 명시, API 변경 사전 공유 |
| **FE** | JSON Depth 4 이하, 서버에서 데이터 정제 |
| **iOS** | Swagger 업데이트, 토큰 갱신 로직 |
| **AOS** | 배포 커뮤니케이션 |

## DO (PM이 했으면 좋겠는 것)

### 일정 조율

- [ ] **테스트베드 배포 후 기획 변경 최소화** 조율 (AOS 요청)
- [ ] **구현 가능한 범위 내 기획** 조율 (iOS 요청)
- [ ] **Animation/인터랙티브 요소 구현 시간** 조율 (iOS, FE 요청)

### 커뮤니케이션

- [ ] **API 변경 시 사전 공유** 프로세스 수립 (전체)
- [ ] **배포 커뮤니케이션** 프로세스 수립 (AOS)
- [ ] **정기적인 리뷰 미팅** 진행

### 초반 확정

- [ ] **초반 구조/컨셉 확정** 유도 (iOS, FE)
- [ ] **아키텍처에 영향을 주는 변경은 초반에 확정** (FE)

## 협업 시 PM이 중재해야 하는 이슈

### 1. 기획 변경 관련

| 이슈 | 관련 파트 | PM 중재 방향 |
|------|----------|-------------|
| 테스트베드 이후 기획 변경 | AOS, FE | 변경 최소화 원칙 수립, 불가피한 경우 일정 재조율 |
| 아키텍처 근간을 흔드는 변경 | FE | 초반 확정 유도, 변경 시 영향도 분석 |

### 2. 디자인 ↔ 개발 갈등

| 이슈 | 관련 파트 | PM 중재 방향 |
|------|----------|-------------|
| UX/UI 수준 vs 개발 시간 | FE, iOS | 팀 목표에 맞춰 조율 |
| Android 특성 미고려 | AOS | 플랫폼별 특성 고려하도록 조율 |

### 3. 백엔드 ↔ 프론트엔드 갈등

| 이슈 | 관련 파트 | PM 중재 방향 |
|------|----------|-------------|
| API 미공지 변경 | 전체 | 변경 사전 공유 프로세스 수립 |
| 데이터 정제 책임 | FE, BE | 서버에서 정제하는 것으로 가이드 |

## 주요 긁힘 포인트 정리 (PM이 알아야 할 것)

### 개발자들이 기획에게 원하는 것

- 초반 구조/컨셉 확정
- 테스트베드 이후 변경 최소화
- 의도 명확히 전달
- 실현 가능한 범위 조율

### 개발자들이 디자인에게 원하는 것

- 디자인시스템 기반 작업
- 라이브러리에서 재현 가능한 수준
- 플랫폼별 특성 고려
- 구현 시간 고려

### 개발자들이 백엔드에게 원하는 것

- nullable 명시
- API 변경 사전 공유
- Swagger 업데이트
- 배포 커뮤니케이션

## 파트별 호소 포인트 (PM이 이해해야 할 것)

### 디자이너

- 시장조사, 설문 신뢰도가 낮을 때 기한 연장 필요
- 초반에는 매니징을 좀 덜 해도 이해 필요
- 할 일이 많음 (BI/CI, 디자인, 컨텐츠 등)

### 프론트엔드

- 세부정책 놓칠 수 있음 → 알려줘서 정하게끔 지도 필요

### 백엔드

- 확장성을 고려한 설계 시간 필요
- 어디까지 기획했을 때 부담이 될지 상상하기 어려움
- 알림 등 지속적 데이터 관리가 필요한 기능은 공수가 큼

## PM 체크리스트

### 프로젝트 시작 시

- [ ] 파트별 인원 확인 및 역할 분담
- [ ] 초반 구조/컨셉 확정 유도
- [ ] 커뮤니케이션 채널 수립
- [ ] 정기 리뷰 일정 수립

### 프로젝트 진행 중

- [ ] 기획 변경 영향도 관리
- [ ] 파트 간 갈등 중재
- [ ] API 변경 공유 프로세스 관리
- [ ] 일정 조율 및 리스크 관리

### 배포 전

- [ ] 테스트베드 배포 후 변경 사항 최소화
- [ ] QA 일정 확보
- [ ] 배포 커뮤니케이션

## 파트별 주요 요청 요약

| 파트 | PM에게 원하는 것 |
|------|-----------------|
| **기획** | 실현 가능한 범위 조율, 일정 관리 |
| **디자인** | 초반 매니징 유연성, 다양한 산출물 고려한 일정 |
| **FE** | 초반 구조 확정, 기획 일관성 유지 |
| **iOS** | 컨셉 확정, 실현 가능 범위 조율 |
| **AOS** | 테스트베드 이후 변경 최소화, 배포 커뮤니케이션 |
| **BE** | 확장성 고려 시간, 기획 영향도 사전 공유 |
