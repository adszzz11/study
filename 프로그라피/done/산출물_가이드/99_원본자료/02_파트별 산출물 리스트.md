# 파트별 산출물 리스트

> 각 파트에서 생성해야 하는 산출물 정리
> 우선순위: 필수 > 권장 > 선택

---

## 한눈에 보는 필수 산출물

| 파트 | 필수 산출물 |
|------|------------|
| **기획/PO** | Wireframe, IA, Flow Chart |
| **PM** | 프로젝트 일정표, 리소스 할당표, 정기 리뷰 일정 |
| **디자인** | 디자인시스템, Figma 디자인 명세, 프로토타입 |

| **AOS** | 앱 구현 |
| **백엔드** | Swagger API 명세 (nullable 필수!) |
| **QA** | 테스트 케이스, 테스트 결과 리포트 |

## 기획/PO 파트

### 필수 산출물

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| **Wireframe** | PPTX/Figma | 화면 구성의 기본 틀, 기획자 의도 명확해야 함 | 전체 |
| **IA (Information Architecture)** | 다이어그램 | 전체 화면 구성도, 한눈에 볼 수 있게 | iOS, 전체 |
| **Flow Chart** | 다이어그램 | 사용자 흐름도 | FE, 전체 |

### 권장 산출물

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| 기능 명세서 | 문서 | 각 기능별 상세 설명 | FE |
| 정책 정의서 | 문서 | 날짜 표기법, 문체, String Token 규칙 | iOS |
| User Story | 문서 | 사용자 관점의 기능 설명 | FE |
| 아이디어 기획서 | 문서 | 아이디어 + 진행 방향 + 일정 + 세부내용 | 디자인 |
| 테스트 시나리오 | 문서 | QA 연계용 | QA |

### 선택 산출물

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| 레퍼런스 정리 | 문서/링크 | 참고할 서비스/디자인 | 디자인 |

### 정책 정의서 포함 항목

```markdown
## 정책 정의서

### 1. 표기법
- 날짜: YYYY.MM.DD / YYYY-MM-DD
- 시간: 24시간제 / 12시간제 (오전/오후)
- 숫자: 천 단위 콤마 사용 여부

### 2. 문체
- 요체 (합니다/입니다)
- 다체 (한다/이다)

### 3. String Token 규칙
- 버튼: btn_[기능]_[상태]
- 라벨: lbl_[화면]_[위치]
- 메시지: msg_[유형]_[번호]

### 4. 에러 메시지 규칙
- 형식: [에러 코드] 사용자 친화적 메시지
```

### 선택 산출물

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| 브랜드 가이드 | 문서 | BI/CI, 컬러, 폰트 | 전체 |
| 컨텐츠 디자인 | - | 마케팅용, 인스타 피딩 등 | 마케팅 |

### 디자인시스템 구성 요소

```markdown
## 디자인시스템 구성

### 1. 토큰 (Token)
- **Primitive Token**: RGBA 값 (기본 색상)
- **Semantic Token**: 의미 기반 정의 (primary, secondary, error 등)

### 2. 컴포넌트
- 버튼 (Button): Primary, Secondary, Tertiary, Ghost
- 입력 필드 (Input): Text, Password, Search, TextArea
- 카드 (Card): 기본, 리스트, 상세
- 모달 (Modal): Alert, Confirm, Bottom Sheet
- 네비게이션: Tab Bar, Navigation Bar, Bottom Navigation

### 3. 레이아웃
- 그리드 시스템
- 여백 규칙 (Spacing)
- 타이포그래피 스케일
```

## 백엔드 파트

### 필수 산출물

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| **API 명세** | Swagger | nullable 표시 **필수** | 전체 |
| **API 명세** | Postman/.http | 테스트 가능한 형태 | FE |

### 권장 산출물

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| OpenAPI Schema | JSON/YAML | TypeScript 타입 추출용 | FE |
| Flow Chart | 다이어그램 | 로직 흐름도 | iOS |
| 시스템 아키텍처 문서 | 문서 | 전체 구조 설명 | 전체 |

### API 명세 필수 포함 항목

| 항목 | 중요도 | 설명 |
|------|--------|------|
| **nullable 표시** | 최상 | 모든 파트에서 가장 많이 요청! |
| 공통코드 설명 | 높음 | 상세하게 작성 |
| API 설명 | 높음 | 용도, 사용법 |
| 에러 코드 정의 | 높음 | 에러 유형별 코드 |

### API 설계 가이드라인

```json
// Good - Depth 4 이하
{
  "user": {
    "profile": {
      "name": "홍길동",
      "avatar": "url"
    }
  }
}

// Bad - Depth 4 초과
{
  "data": {
    "user": {
      "profile": {
        "details": {
          "name": "홍길동"  // Depth 5 - 지양
        }
      }
    }
  }
}
```

### 버그 리포트 템플릿

```markdown
## 버그 리포트

### 환경 (필수!)
- 기기: (예: Galaxy S23, iPhone 15 Pro)
- OS 버전: (예: Android 14, iOS 17.2)
- 앱 버전: (예: 1.2.3)

### 재현 단계
1. ...
2. ...
3. ...

### 예상 결과
-

### 실제 결과
-

### 심각도
- [ ] Critical (앱 크래시, 데이터 손실)
- [ ] High (주요 기능 불가)
- [ ] Medium (기능 제한적 사용 가능)
- [ ] Low (사소한 UI 문제)

### 첨부
- 스크린샷/영상
```

### QA 워크플로우 권장 구성

```
1. 테스트 케이스 작성
   ↓
2. 테스트 실행
   ↓
3. 결과 리포트 작성
   ↓
4. 버그 등록 (기기/환경 명시)
   ↓
5. 수정 확인
   ↓
6. 재테스트
```

## RACI 매트릭스

> R: Responsible (실행) | A: Accountable (책임) | C: Consulted (자문) | I: Informed (통보)

| 산출물 | 기획/PO | PM | 디자인 | FE | iOS | AOS | BE | QA |
|--------|---------|-----|--------|-----|------|------|-----|-----|
| Wireframe | R/A | I | C | I | I | I | I | I |
| IA | R/A | I | C | C | C | C | I | I |
| Flow Chart | R/A | I | C | C | C | C | C | I |
| 정책 정의서 | R/A | I | C | C | C | C | C | I |
| 디자인시스템 | C | I | R/A | C | C | C | I | I |
| Figma 디자인 | C | I | R/A | C | C | C | I | I |
| API 명세 | I | I | I | C | C | C | R/A | I |
| 테스트 케이스 | C | I | I | C | C | C | C | R/A |
| 프로젝트 일정 | C | R/A | I | I | I | I | I | I |
