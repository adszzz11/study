### 기타 필수 항목

- 공통코드 설명 상세 작성 (FE)

## DO (했으면 좋겠는 것)

### API 설계

- **사용자(프론트엔드) 입장에서 API 설계** (FE)
- **JSON Depth 4단계 이하로 구성** (FE)
- **Entity 값을 원형으로 최대한 내려주기** (AOS)
  - 프론트 입장에서 필요한 데이터만 선별 가능하도록

### 데이터 처리

- **데이터 정제는 웬만하면 서버 라인에서 처리** (FE)
  - 백엔드가 동의함

### 문서화

- **명세 문서 지속 업데이트** (iOS)
- **API 설명을 상세하게 작성** (FE)
- **공통코드 설명 포함** (FE)

### 인증/보안

- **로그인 토큰 갱신 로직 집중 설계** (iOS)
  - 디비에서 토큰 관리 필수

### 커뮤니케이션

- **배포 시 커뮤니케이션** (AOS)
- 소통을 잘 하자 (iOS)

## 협업 시 요청사항

### API 변경 관련

- API 변경 시 사전 공유 부탁드립니다 (FE)
- 변경사항은 미리 소통해주시면 대응이 수월합니다 (iOS)
- 배포 관련 사항은 사전 공유 부탁드립니다 (AOS)

### nullable 관련 (전체 공통)

- nullable 표시를 명확히 해주시면 클라이언트 개발이 수월합니다 (FE, iOS, AOS 전체)

### Schema 설계

- Schema는 적절한 단위로 구성해주시면 좋겠습니다 (AOS)
- 데이터 정제는 가능하면 서버에서 처리해주시면 감사하겠습니다 (FE)

### 문서화

- 공통코드 설명을 함께 작성해주시면 이해가 빠릅니다 (FE)

### 인증

- 토큰은 DB에서 관리하는 방식을 권장드립니다 (iOS)

## 주요 이슈 포인트

### 1. nullable (최다 언급)

모든 파트(FE, iOS, AOS)에서 공통적으로 가장 많이 요청한 사항

**해결책**: 모든 API 응답 필드에 nullable 여부 명시

### 2. 변경 사항 공유

모든 파트에서 공통적으로 요청하는 사항:
- API 변경 시 사전 공유
- 배포 전 커뮤니케이션

### 3. 로그인 토큰 갱신

- 토큰 갱신 로직에 집중 필요
- DB에서 토큰 관리 필수

## API 설계 가이드라인

### JSON Depth 권장

| 구분 | 예시 | 비고 |
|------|------|------|
| **Good (Depth 4 이하)** | user → profile → name | 권장 |
| **Bad (Depth 4 초과)** | data → user → profile → details → name | 지양 |

### Schema 설계 원칙 (AOS)

- Entity 값을 원형으로 내려주기
- 과도하게 쪼개지 않기
- 프론트에서 필요한 데이터 선별 가능하도록

## 파트별 주요 요청 요약

| 파트 | 핵심 요청 |
|------|----------|

## 백엔드로서 기획자에게 요청하는 것

PM.md raw 데이터 기반 백엔드 관점 정리

### 확장성 고려

- 확장성을 고려한 기획을 해주시면 감사하겠습니다
- 데이터모델링에 영향을 주는 결정은 초반에 확정해주세요

### 시스템 아키텍처

- 어떤 구조로 되어있는지 질문해주시면 설명드리겠습니다
- 시스템 아키텍처 문서화를 진행할 예정입니다

### 기획 영향도

- 어디까지 기획했을 때 부담이 되는지 상상이 어려우실 수 있습니다
  - 알림 기능: 지속적인 데이터 관리 필요
  - 조회 기능: 데이터 구조에 따라 난이도 상이
- 기획 시 개발 영향도가 궁금하시면 먼저 여쭤봐 주세요

## 협업 시 주의 사항 (기획자가 주의할 것)

백엔드 개발자가 어려움을 겪는 상황

- API param을 자꾸 몰래 하나씩 없애는 경우
- 파라미터를 자꾸 바꾸는 경우
- 기획 변경이 데이터 구조에 영향을 주는 경우
