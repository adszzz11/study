# 기획 파트 요구사항 명세서

> 개발자/디자이너 인터뷰 기반 기획 파트 요구사항 통합 정리

---

## 필수 산출물 체크리스트

### 1순위 (필수)

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| Wireframe | PPTX, Figma | 화면 구성의 기본 틀, 기획자 의도가 분명해야 함 | 전체 |
| Flow Chart | 다이어그램 | 사용자 흐름도 | FE |
| IA (Information Architecture) | 다이어그램 | 전체 화면 구성도, 한눈에 볼 수 있게 | iOS |

### 2순위 (권장)

| 산출물 | 형식 | 설명 | 요청 파트 |
|--------|------|------|----------|
| 기능 명세서 | 문서 | 각 기능별 상세 설명 | FE |
| User Story | 문서 | 사용자 관점의 기능 설명 | FE |
| 정책 정의서 | 문서 | 날짜 표기법, 문체, String Token 등 | iOS |
| 테스트 시나리오 | 문서 | QA와 연계 | iOS |
| 아이디어 기획서 | 문서 | 아이디어 + 진행 방향 + 세부내용 | Designer |
| 레퍼런스 | 문서/링크 | 참고할 아이디어/디자인 | Designer |

---

## 정책 정의서 포함 항목

- [ ] 날짜 표기법 (YYYY-MM-DD 등)
- [ ] 문체 (요체/다체)
- [ ] 디자인시스템 정의
- [ ] String Token 규칙

---

## DO (했으면 좋겠는 것)

### 구조 및 설계
- [ ] **초반 구조를 확실히 잡기** - 후반 롤백 방지 (iOS, FE)
- [ ] **아키텍처의 근간이 되는 구조를 명확히 정의** (FE)
- [ ] **컨셉을 명확히 정의** - 초장에 잡기 (iOS)

### 일관성
- [ ] **기획 일관성 유지** - nav bar, baseline 등 핵심 요소 유지 (FE)
- [ ] Figma와 기획 문서의 연동 (FE)

### 커뮤니케이션
- [ ] **기획 의도를 명확하게 표현** - 맞고 틀리고는 상관없이 의도 전달 (AOS)
- [ ] **의견 제시 시 이유를 명확하게** - 설득력 있는 제안 (Designer)
- [ ] 실현 가능한 범위 내에서 기획 (iOS)

---

## DON'T (하지 않았으면 좋겠는 것)

### 변경 관련 (심각)
- [ ] ❌ **테스트베드 올라간 후 기획 변경** (AOS - 주의 필요)
- [ ] ❌ **중간에 기획 변경** - 특히 아키텍처 근간을 흔드는 변경 (FE)

### 과도한 요구
- [ ] ❌ **기획 오버치기** - 과도한 기능 요구 (iOS)
- [ ] ❌ 할 수 있을 것 같은 라인만 (iOS)

### 일관성 문제
- [ ] ❌ 기획 일관성이 떨어지는 산출물 전달 (FE)
- [ ] ❌ 화면이 어지러운 아이디어/디자인 예시 (Designer)

### 커뮤니케이션
- [ ] ❌ **근거 없는 고집** (AOS - 주의 필요)
- [ ] ❌ 이유 없이 의견만 던지기 (Designer)
- [ ] ❌ 벤치마킹이 아닌 단순 비교 요구 (iOS)

---

## 기획서 포함 권장 항목 (Designer 요청)

1. 아이디어가 좋은 이유
2. 진행 방향
3. 일정
4. 세부 내용

---

## 파트별 주요 요청 요약

| 파트 | 핵심 요청 |
|------|----------|
| **FE** | 초반 구조 확정, 일관성 유지, 아키텍처 변경 금지 |
| **iOS** | 초반 컨셉 확정, 실현 가능한 기획, 오버 기획 금지 |
| **AOS** | 의도 명확히, 테스트베드 이후 변경 금지, 고집 금지 |
| **Designer** | 근거 있는 제안, 정리된 레퍼런스, 어지러운 예시 금지 |
