## 협업 파트별 필요 명세

### 백엔드 파트

#### 필수 산출물
| 산출물 | 형식 |
|--------|------|
| API 문서 | Docs, Swagger |

#### 했으면 좋겠는 것
- nullable 표시 정확하게 작성
- 배포 시 커뮤니케이션
- Entity 값을 원형으로 최대한 내려주기
  - 프론트 입장에서 필요한 데이터만 선별 가능하도록

#### 하지 않았으면 좋겠는 것
- Schema 과도하게 쪼개기
- nullable 표시 누락
- 배포 관련 소통 없이 변경

---

### 디자인 파트

#### 필수 산출물
| 산출물 | 설명 |
|--------|------|
| 프로토타입 | WireFrame 포함 |
| Animation Diagram | 인터랙션 정의 |
| 일관된 화면 디자인 | 모나지 않은 통일된 UI |

#### 했으면 좋겠는 것
- 디자인 통일성 유지
- Android 특성 고려한 디자인
- 다양한 화면 크기 고려 (넓은 화면 / 좁은 화면)

#### 하지 않았으면 좋겠는 것
- iPhone 디자인만 보고 Android에 동일하게 요구
- 폴더블(Fold) 기기 미고려
  - 플립은 상대적으로 괜찮음
- 말도 안 되는 특수 기종 미고려

#### Android 개발자로서 호소
- Android 개발자라는 것을 인지해주세요
- 폴더블 기기를 고려해주세요
- 화면 크기 다양성을 고려해주세요

---

### 기획자 파트

#### 필수 산출물
| 산출물 | 비고 |
|--------|------|
| 디테일한 Wireframe | 기획자 의도가 분명해야 함 |

#### 했으면 좋겠는 것
- 기획 의도를 명확하게 표현
- 맞고 틀리고는 상관없으니 의도 전달

#### 하지 않았으면 좋겠는 것 (척살 대상)
- 테스트베드 올라간 후 기획 변경
- 근거 없는 고집

---

### QA 파트

#### 필수 산출물
| 산출물 | 설명 |
|--------|------|
| QA 워크플로우 | 업무 프로세스 정의 (QA가 직접 정의 가능) |
| 테스트 케이스 | 기능별 테스트 항목 |
| 테스트 결과 | 실행 결과 리포트 |

#### 품질 목표
- **100% 이상**의 테스트 성공률
- **98% 이상**의 Crash-free 비율
  - 예측 못하는 이슈는 발생할 수 있으나 최대한 잡고 싶음

#### 했으면 좋겠는 것
- 자신만의 프로세스 정의 및 운영
- 구체적인 버그 리포트

#### 하지 않았으면 좋겠는 것
- "안 돼요"만 시전하기
  - 내 기기에서 안 되는데 다른 기기에서는 됨 → 기기 명시 필요
- "기획에서는 저렇게 되어있는데" 만 말하기
  - 해상도 다양성으로 인한 이슈 고려 필요
- 특정 파트만 테스트 누락
  - (서운함 유발)

## Android 특수 고려사항

### 기기 다양성
| 유형 | 고려 수준 |
|------|----------|
| 일반 스마트폰 | 기본 |
| 폴더블 (Fold) | 필수 고려 |
| 플립 | 상대적으로 여유 |
| 특수 해상도 기기 | 케이스별 협의 |

### 화면 대응
- 넓은 화면 / 좁은 화면 모두 고려
- iPhone 기준 디자인을 그대로 적용하지 않기
