# iOS 파트 협업 가이드

> iOS 멘토 인터뷰 기반 정리 (팀 구성: 3명 기준)

---

## 업무 프로세스

### 일반적인 워크플로우
1. Feature 정의 및 업무 분장
2. PR 리뷰 공동 진행
3. Wiki 정리 (순서도, 다이어그램 등)

### 프로젝트 단계별 업무
| 단계 | 주요 업무 |
|------|----------|
| 초기 | 와이어프레임 기반 프로젝트 세팅 |
| 중기 | 화면 개발 및 기능 구현 |
| 말기 | 테스트 및 배포 준비 |

### 프로젝트 초기 세팅
- Modulation (모듈화)
- Framework 선정
- Architecture 설계
- 자동화 세팅 (CI/CD)

## 협업 파트별 필요 명세

### 기획 파트

#### 필수 산출물
| 산출물 | 형식 | 비고 |
|--------|------|------|
| 기획서 | PPTX (선호) | Figma는 제대로 쓸 게 아니면 지양 |
| 정책 정의서 | 문서 | 날짜 표기법, 문체, String Token 등 |
| IA (Information Architecture) | 다이어그램 | 전체 화면 구성도 |
| 테스트 시나리오 | 문서 | QA와 연계 |

#### 정책 정의서 포함 항목
- 날짜 표기법
- 문체 (요체/다체)
- 디자인시스템 정의
- String Token 규칙

#### 했으면 좋겠는 것
- 초반 구조를 확실히 잡기 (후반 롤백 방지)
- 컨셉을 명확히 정의
- 실현 가능한 범위 내에서 기획

#### 하지 않았으면 좋겠는 것
- 기획 오버치기 (과도한 기능 요구)
- 벤치마킹이 아닌 단순 비교 요구

---

### 디자인 파트

#### 필수 산출물
- 토큰화된 디자인시스템
- 컴포넌트 정의 (텍스트박스, 버튼 등)
- 통일된 컴포넌트 구성

#### 했으면 좋겠는 것
- 디자인시스템 기반 구성을 먼저 진행
- 화면 제작 시 정의나 예시 기반으로 작업
- 구현 시간 고려한 대안 제시

#### 하지 않았으면 좋겠는 것
- 과도한 Animation 요구
- 구현 시간을 고려하지 않은 인터랙티브 요구

#### 추가 협업 방안
- 디자이너가 프론트 CSS를 직접 작성하는 경우도 긍정적으로 고려

### 백엔드 파트

#### 필수 산출물
| 산출물 | 비고 |
|--------|------|
| Swagger API 명세 | 지속적 업데이트 필수 |
| Flow Chart | 로직 흐름도 |

#### 했으면 좋겠는 것
- 명세 문서 지속 업데이트
- nullable 표시 명확히 작성
- 로그인 토큰 갱신 로직 집중 설계

#### 하지 않았으면 좋겠는 것
- 소통 없는 API 변경
- nullable 표시 누락
- 토큰을 DB에서 관리하지 않는 설계

---

### QC(QA) 파트

#### 필수 산출물
| 산출물 | 설명 |
|--------|------|
| 테스트 케이스(시나리오) | 기능별 테스트 항목 |
| 에러 명세 | 에러 유형 및 대응 방안 |
| 결함 처리 방식 | 버그 리포트 및 처리 프로세스 |
| 상황 명세 | 엣지 케이스 정의 |

#### 했으면 좋겠는 것
- 구체적인 에러 상황 설명
- 재현 가능한 버그 리포트

#### 하지 않았으면 좋겠는 것
- "안 돼요"만 말하고 끝내기 (원인/상황 설명 필요)
