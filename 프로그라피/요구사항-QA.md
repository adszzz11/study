# QA 파트 요구사항 명세서

> 모바일 개발자 인터뷰 기반 QA 파트 요구사항 통합 정리

---

## 필수 산출물 체크리스트

### 테스트 관련

| 산출물 | 설명 | 요청 파트 |
|--------|------|----------|
| 테스트 케이스(시나리오) | 기능별 테스트 항목 | iOS, AOS |
| 테스트 결과 | 실행 결과 리포트 | AOS |
| 에러 명세 | 에러 유형 및 대응 방안 | iOS |
| 상황 명세 | 엣지 케이스 정의 | iOS |

### 프로세스 관련

| 산출물 | 설명 | 요청 파트 |
|--------|------|----------|
| QA 워크플로우 | 업무 프로세스 정의 | AOS |
| 결함 처리 방식 | 버그 리포트 및 처리 프로세스 | iOS |

---

## 품질 목표 (AOS)

| 지표 | 목표 |
|------|------|
| 테스트 성공률 | **100% 이상** |
| Crash-free 비율 | **98% 이상** |

> 예측 못하는 이슈는 발생할 수 있으나 최대한 잡고 싶음

---

## DO (했으면 좋겠는 것)

### 버그 리포트

- [ ] **구체적인 에러 상황 설명** (iOS)
- [ ] **재현 가능한 버그 리포트** (iOS)
- [ ] **구체적인 버그 리포트** (AOS)

### 프로세스

- [ ] **자신만의 프로세스 정의 및 운영** (AOS)
  - QA가 업무 프로세스를 정의해도 됨
  - 자신의 프로세스를 가져가는게 훨씬 명확할 수 있다

---

## DON'T (하지 않았으면 좋겠는 것)

### 커뮤니케이션 (공통 - 가장 많이 언급)

- [ ] ❌ **"안 돼요"만 말하고 끝내기** (iOS, AOS)
  - 원인/상황 설명 필요
  - 내 기기에서 안 되는데 다른 기기에서는 됨 → **기기 명시 필요**

### 기획 기반 판단

- [ ] ❌ **"기획에서는 저렇게 되어있는데" 만 말하기** (AOS)
  - 해상도 다양성으로 인한 이슈 고려 필요

### 테스트 누락

- [ ] ❌ **특정 파트만 테스트 누락** (AOS)
  - (서운함 유발)
- [ ] ❌ 내 테스트만 안해줄 때 있음 (AOS)

## 버그 리포트 가이드라인

### 필수 포함 항목

```markdown
## 버그 리포트

### 환경
- 기기: (예: Galaxy S23, iPhone 15 Pro)
- OS 버전: (예: Android 14, iOS 17.2)
- 앱 버전: (예: 1.2.3)

### 재현 단계
1. ...
2. ...
3. ...

### 예상 결과
-

### 실제 결과
-

### 첨부
- 스크린샷/영상
```

### Bad Case

```markdown
❌ "안 돼요"
❌ "기획이랑 다른데요"
❌ "제 폰에서는 안 돼요"
```

### Good Case

```markdown
✅ "Galaxy S23 (Android 14)에서 결제 버튼 클릭 시 앱이 크래시됩니다.
   재현 단계: 1. 장바구니 진입 2. 결제하기 클릭 3. 크래시 발생"

✅ "기획서에는 A로 되어있는데, 해상도 320dp 이하 기기에서는
   레이아웃이 깨져서 B로 대응하는 것이 어떨까요?"
```

## 테스트 시 고려사항 (AOS)

### Android 기기 다양성

| 유형 | 테스트 필요 수준 |
|------|-----------------|
| 일반 스마트폰 | 기본 |
| 폴더블 (Fold) | 필수 |
| 플립 | 선택 |
| 다양한 해상도 | 필수 |

### 해상도 다양성

- 넓은 화면 / 좁은 화면 모두 테스트
- 해상도로 인한 이슈는 기획 문제가 아닐 수 있음

---

## 파트별 주요 요청 요약

| 파트 | 핵심 요청 |
|------|----------|
| **iOS** | 재현 가능한 버그 리포트, 에러/상황 명세, "안 돼요"만 금지 |
| **AOS** | 자체 프로세스 정의, 기기 명시, 해상도 고려, 공평한 테스트 |

---

## QA 워크플로우 권장 구성 (AOS)

```
1. 테스트 케이스 작성
   ↓
2. 테스트 실행
   ↓
3. 결과 리포트 작성
   ↓
4. 버그 등록 (기기/환경 명시)
   ↓
5. 수정 확인
   ↓
6. 재테스트
```

> QA가 직접 업무 프로세스를 정의하면 더 명확해질 수 있음
