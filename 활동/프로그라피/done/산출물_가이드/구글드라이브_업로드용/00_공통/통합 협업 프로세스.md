# 통합 협업 프로세스

프로그라피 프로젝트 협업 플로우 정의

---

## 핵심 원칙 4가지

| 원칙 | 핵심 메시지 |
|------|------------|
| **1. 초반 확정** | 구조, 컨셉, 기술스택은 프로젝트 시작 시 확정 |
| **2. 변경 최소화** | 테스트베드 이후 기획 변경 금지 |
| **3. 사전 공유** | API 변경, 배포 전 반드시 커뮤니케이션 |
| **4. 구체적 리포트** | 버그/이슈 시 기기/환경 정보 필수 포함 |

---

## 전체 프로젝트 플로우

| 단계                | 내용                                    | 산출물                               | 주의사항           |
| ----------------- | ------------------------------------- | --------------------------------- | -------------- |
| **1단계: 요구사항 협의**  | 요구사항 접수, 협의 회의 (전 파트 참여), PO 의사결정     | 아이디어 기획서, 레퍼런스                    | -              |
| **2단계: 기획 확정**    | Wireframe, IA, Flow Chart 작성, 전 파트 리뷰 | Wireframe, IA, Flow Chart, 정책 정의서 | 아키텍처 근간 구조 확정  |
| **3단계: 디자인**      | 디자인시스템 구축, 화면별 디자인                    | 디자인시스템, Figma 명세, 프로토타입           | Android 특성 고려  |
| **4단계: 개발**       | BE, FE, iOS, AOS 개발                   | Swagger API 명세                    | API 변경 시 사전 공유 |
| **5단계: 테스트베드 배포** | 내부 테스트용 배포                            | -                                 | 이후 기획 변경 최소화   |
| **6단계: QA**       | 테스트 케이스 기반 테스트, 버그 리포트                | -                                 | 구체적인 상황 설명 필수  |
| **7단계: 배포**       | 배포 및 전 파트 공유                          | -                                 | 배포 전 커뮤니케이션 필수 |

---

## 파트 간 커뮤니케이션 플로우

| From | To | 전달 내용 |
|------|-----|----------|
| 기획 | 디자인 | Wireframe, 정책 |
| 디자인 | 개발 | 디자인시스템, Figma 명세 |
| 백엔드 | 클라이언트 | API 명세, 변경 사전 공지 |
| 개발 | QA | 테스트 빌드 |
| QA | 개발 | 버그 리포트 (기기/환경 명시) |

## 정기 싱크업 구조

| 종류 | 주기 | 참여자 | 목적 |
|------|------|--------|------|
| 데일리 스탠드업 | 매일 | 파트별 | 당일 할 일 공유, 블로커 파악 |
| 주간 리뷰 | 주 1회 | 전 파트 | 진행 상황 공유, 이슈 논의 |
| 스프린트 리뷰 | 스프린트 종료 시 | 전 파트 | 결과 데모, 회고 |

## 변경 관리 프로세스

### API 변경 시

| 순서 | 단계 |
|------|------|
| 1 | 변경 필요성 발생 |
| 2 | 영향 범위 분석 |
| 3 | 변경 공지 (최소 2일 전) |
| 4 | 변경 적용 |

| 순서 | 단계 |
|------|------|
| 1 | 변경 요청 |
| 2 | PM 변경 가능 여부 판단 |
| 3 | 영향도 분석 |
| 4 | 전파트 공유 |
| 5 | 일정 재조율 |

## 이슈 에스컬레이션

| Level | 대상 | 예시 |
|-------|------|------|
| 1 | 파트 내 해결 | 코드 버그, 디자인 수정 |
| 2 | 파트 간 협의 | API 스펙 협의, 디자인-개발 조율 |
| 3 | PM 중재 | 일정 조율, 우선순위 결정 |
| 4 | PO 결정 | 기능 삭제/추가, 방향성 변경 |
