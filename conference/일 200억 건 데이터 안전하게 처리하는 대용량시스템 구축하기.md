## 데이터 통합 (Data Integration)

### Back Pressure
**정의:** 생산자 속도를 조절해 소비자가 안정적으로 데이터를 처리할 수 있게 도와주는 메커니즘

### EDA (Event-Driven Architecture)

**아키텍처 플로우:**
```
Data Input → Publisher → Kafka Event
          → Listener → Data Process Event
```

#### Publisher 서버를 앞에 두면 좋은 점
1. **기술 추상화 가능**
2. **메시지 형상 관리 가능**
3. **WOOWA RPC Server** 사용
	- 자바 인터페이스가 IDL
4. **순서 보장 가능**

#### Listener 서버를 앞에 두면 좋은 점
1. **Back Pressure 가능**
	- Concurrency: Consumer가 처리할 thread 개수를 따로 가져갈 수 있음
2. **시간대로 다시 생성 가능**
3. **컨슘 제어 가능**

## 데이터 처리 (Data Processing)

### Task 기반 Workflow
- 유향 비순환 그래프(DAG) 구조

### 고가용성 (High Availability)
- **HA** (High Availability)
- **부하 분산** (Load Balancing)
- **Partitioning**

### SPOF (Single Point of Failure) 해결

**Sharding 전략:**
- **App-level Sharding** 도입
	- Key-based Sharding
	- Range-based Sharding
	- Directory-based Sharding

**우아한형제들 채택:**
- **Key + Directory** 방식

**Hot Partition & Cold Partition 고려:**
- Shard 정책을 2개로 분리 운영

### Woowa Persistence API
- 자체 개발 데이터 처리 API

## 서비스 구축

### 우아한형제들의 고민

**핵심 과제:**
- 비즈니스 요구사항에 따라 다양한 케이스를 구분해야 함
- **수많은 데이터 유형을 안전하고 유연하고 빠르게 처리하는 방법**

### Data Driven Development

**우아한 워크플로우 설계 원칙:**
1. **데이터 처리 흐름을 유연하게 변경**
2. **트랜잭션 단위를 유연하게 구성**
3. **특정 동작만 별도로 수행**

### 결합도 낮추기
- 호출에 대한 결합도도 낮추자
- 유연한 확장이 가능한 구조

## 💡 인사이트 & 배운 점

- **Back Pressure의 중요성**:
	- 대용량 트래픽에서 안정적인 처리를 위해 필수
	- Publisher/Listener 분리로 제어 가능
- **Sharding 전략의 현실**:

- **워크플로우 설계 철학**:
	- 유연성과 확장성을 최우선으로 고려
	- 비즈니스 요구사항 변화에 빠르게 대응
