# AI Dev
AI 개발의 현주소

## 주로 도입하고 있는 기술들
- RAG
	- Advanced RAG
		- index : Contextual Embedding, Metadata Extraction
		- preprocess : Query Decomposition, Query Expansion
		- retrieval : Chunking hierarchy, Hybrid Search
		- post processing : Reranking Agentic Reasoning
	- picks
		- RAG Builder : lnagchain(langgraph), LLamaIndex
		- Vector DB : cloud vendors, milvus(opensource), pinecone(close source, 잘 안씀)
		- Tracing : Langsmith, Langfuse,
		- Prompt Management(Visioning) : SaaS(Vellum, Prompt Hub, etc)
		- Evaluation : RAGAS(OpenSource), W&B(GUI)
## 타 회사외 커피챗(면접) 때 만났던 FAQ & Guides
- FAQ
	- data preprocess
		- 완벽한 parser는 없음
		- Contextual하게 쪼개기 매우 중요
		- 노가다 쩔수없음
		- 멱등한 데이터 환경이 중요
		- 메타데이터 활용 중요
			- embedding할 데이터 구조 잡기
			- 도메인 전문가와 협업 필요
	- retrieval
		- 결국 좋은 데이터를 가져오는게 중요
		- 여러 Retrieval 방식을 앙상블하는게 좀 더 낫더라
		- Hierarchy를 두는 식으로 많이 함
		- 관련 관심사 많이 가져와서 Filter, Reranking 하는 방식을 적용하는게 낫더라
		- Semantic, Keyword, Hybrid 중 뭐가 낫다고 할 수 없음, 케바케
			- 보통 Semantic이 조금 더 잘 나오는 편인듯
	- RAG Tech
		- 최신 RAG기술을 찾아보는게 나을까
			- 시간이 별로 없다면 Advanced RAG로 충분함
			- 신기술보단 기존의 영역 강화방안을 고민해보자
			- 좋아보이는 기술들은 보통 케바케인 경우가 많았음
		- GraphRAG에 대한 생각
			- 1 depth retrieval에 비해 추가적인 context를 요하는 쿼리를 잘함
			- 복잡도, 데이터 관리 측면에서 비용이 꽤 들어감
	- Evaluation
		- 크게 Retrieval, Generation 두개를 평가하는 방식으로 나뉨
		- 테스트셋 구성을 잘하는게 중요함
		- 평가기준 세우기
			- ML 논문들에서 보이는 metric이나 평가방식을 그대로 차용하기보단 문제에 맞게 재정의하는것이 중요
			- 1-5, true/false, % 등 demension은 다양할 수 있음
			- 정상적인 평가 기준으로 보면, LLM을 사용하는게 나은듯
		- 우선 수기로 사람이 해보기
		- 메커니컬한 지점들을 찾아 자동화
	- Tool Selection
		- 선택기준
			- 안정적인가?
			- 툴에 크게 의존하는가?
			- 커뮤니티 지원이 충분한가?
		- 익숙하지 않다면, 좋아보이는 거 보다 대중적인 툴 사용하기(like LangChain)
		- 문제들이 보이고 나서 새로운 툴로 옮겨도 괜찮음. RAG는 웹개발처럼 마이그레이션이 복잡하지 않음
		- 데이터 엔지니어링 인프라는 충분히 고민하고 선택하자

- Retrieval이 힘들더라
	- 사람을 써야한다

	- 인프라 비용
	- 자동화 파이브라인 생성도 좀 비용임

Vector Database 비용처리
- 정답이 ㄹㅇ 없음

- 맞음. 데이터가 많아지면 느려지는건 사실

- 잘 모름
- 배포하는 것도 직접 개발했음

- 자동화 비용이 크게 들진 않았던 거 같음
