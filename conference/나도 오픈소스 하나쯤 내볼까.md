## 🎯 핵심 내용

### 오픈소스의 개념 변화

**Original (과거):**
- 동작하는 프로그램 코드

**Now (현재):**
- GitHub에 올릴 수 있는 내용
- 도움말, 즐겨찾기(Awesome), Versioning, Personal Blog
- **배포 방식 중 하나**로 생각

### Release Mastering

**기술 명세서의 사례:**
- **HeadVer**: Versioning 규칙을 GitHub에 release
- SemVer 호환 고려

---

## 오픈소스로 배포 시 장점

**아키텍처 플로우:**
```
Source → Private Repo → Public Repo → CI/CD
```

### 1. 아키텍처
- **비즈니스 로직 의존성을 낮출 수 있다**
- **추상화를 조금 더 하게 된다**
- **보안에 조금 더 신경 쓰게 된다**
	- Secret Key 커밋 방지

### 2. 라이선스 자유도
- **사용권리의 법무 체크 부담을 낮출 수 있음**
- 관계가 복잡한 회사에서 다른 법인이 개발한 소프트웨어를 쓰는 경우 유용

### 3. 지속가능성
- 팀 내 코드가 아닌, **회사 내 모두의 코드**
- 회사 내 모두의 코드가 아닌, **회사 이름의 코드**
- **유용한 기술에 공감하는 사람들과 함께 만든다!**

## ABC User Feedback 사례

**정의:** 사용자 의견을 수집하는 도구 (오픈소스)

**기술 스택:**
- Frontend: React
- Backend: Node.js
- Monorepo: Turborepo

### 기존 SaaS 대안

**Uservoice.com:**
- 신고받은 버그 수정
- 의견으로 기능 생성
- **문제점**: 개인정보를 다루는 회사에서 SaaS 사용이 어려움
- **비용**: 1000명 의견 처리에 월 $899

---

## 오픈소스 설계 원칙

### MVP (Minimum Viable Product)
**"동료를 설득 못하면 유저도 설득 못한다"**

**핵심 원칙:**
- 쉬운 설치 경험
- 데이터 레이어 분리

### 소프트웨어 구성 옵션

1. **일반적인 SaaS**
2. **DB 선택 가능**
3. **설치형 소프트웨어**
4. **오픈소스**

### ABC User Feedback의 해결점

**개인정보 취급:**
- 사용자가 DB 직접 관리

**회사 의존성 제거:**
- 지적재산권 법률 검토 없이 사용 가능
- 필드 편집 기능으로 비즈니스 의존성 제거
- 1개 회사에서 여러 프로젝트 적용 가능

**확장성:**
- SSO, OAuth 지원

---

## 오픈소스 시작 가이드

### 첫 시작

**1. 자동화부터 해보자**
- 업무 1개 자동화
- 비슷한 업무 자동화
- 다른 환경 업무 자동화

**2. 이름 이상한 거 짓지 말자**
- About 좀 잘 써보자

**3. 로직 없이 설정파일부터 작성해보자**
- 이거 잘 되면 **추상화 훈련이 매우 잘 됨**

### 소프트웨어 유형별 구조

1. **유틸리티 라이브러리**
2. **데이터 의존 라이브러리**
3. **API/CLI Application**
4. **End User가 있는 Application**

## 💡 인사이트 & 배운 점

- **오픈소스는 배포 전략**:
	- 코드 공유를 넘어, 라이선스, 지속가능성, 아키텍처 개선의 도구
- **추상화 훈련**:
	- 설정 파일부터 작성하면 비즈니스 로직과 기술 로직 분리 연습
- **MVP의 중요성**:
	- 동료 설득이 사용자 설득의 전제조건
- **데이터 레이어 분리**:
	- 사용자가 DB를 직접 관리하도록 하여 개인정보 문제 해결
