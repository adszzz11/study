## 🎯 핵심 내용

### 주로 도입하고 있는 기술들

#### RAG (Retrieval-Augmented Generation)

**Advanced RAG 구성요소:**
- **Index**
	- Contextual Embedding
	- Metadata Extraction
- **Preprocess**
	- Query Decomposition
	- Query Expansion
- **Retrieval**
	- Chunking hierarchy
	- Hybrid Search
- **Post Processing**
	- Reranking
	- Agentic Reasoning

**기술 스택 (Picks):**
- **RAG Builder**: Langchain(Langgraph), LLamaIndex
- **Vector DB**: Cloud vendors, Milvus(OpenSource), Pinecone(Closed Source, 잘 안씀)
- **Tracing**: Langsmith, Langfuse
- **Prompt Management (Versioning)**: SaaS (Vellum, Prompt Hub 등)
- **Evaluation**: RAGAS(OpenSource), W&B(GUI)

### 타 회사와의 커피챗(면접) FAQ & Guides

#### 1. Data Preprocess
- 완벽한 parser는 없음
- Contextual하게 쪼개기 매우 중요
- 노가다 불가피
- 멱등한 데이터 환경이 중요
- **메타데이터 활용 중요**
	- Embedding할 데이터 구조 잡기
	- 도메인 전문가와 협업 필요

#### 2. Retrieval
- 결국 좋은 데이터를 가져오는 게 중요
- 여러 Retrieval 방식을 앙상블하는 게 나음
- Hierarchy를 두는 식으로 많이 함
- 관련 관심사 많이 가져와서 Filter, Reranking 하는 방식이 나음
- Semantic, Keyword, Hybrid 중 뭐가 낫다고 할 수 없음 (케바케)
	- 보통 Semantic이 조금 더 잘 나오는 편

#### 3. RAG Tech
**최신 RAG 기술을 찾아보는 게 나을까?**
- 시간이 별로 없다면 Advanced RAG로 충분함
- 신기술보단 기존 영역 강화 방안을 고민해보자
- 좋아 보이는 기술들은 보통 케바케인 경우가 많음

**GraphRAG에 대한 생각**
- 1 depth retrieval에 비해 추가적인 context를 요하는 쿼리를 잘함
- 복잡도, 데이터 관리 측면에서 비용이 꽤 들어감

#### 4. Evaluation
- 크게 Retrieval, Generation 두 개를 평가하는 방식으로 나뉨
- **테스트셋 구성을 잘하는 게 중요함**
- **평가 기준 세우기**
	- ML 논문들에서 보이는 metric이나 평가 방식을 그대로 차용하기보단 문제에 맞게 재정의하는 것이 중요
	- 1-5, true/false, % 등 dimension은 다양할 수 있음
	- 정상적인 평가 기준으로 보면, LLM을 사용하는 게 나음
- 우선 수기로 사람이 해보기
- 메커니컬한 지점들을 찾아 자동화

#### 5. Tool Selection
**선택 기준:**
- 안정적인가?
- 툴에 크게 의존하는가?
- 커뮤니티 지원이 충분한가?

**권장 사항:**
- 익숙하지 않다면, 좋아 보이는 것보다 대중적인 툴 사용하기 (like LangChain)
- 문제들이 보이고 나서 새로운 툴로 옮겨도 괜찮음
- RAG는 웹 개발처럼 마이그레이션이 복잡하지 않음
- **데이터 엔지니어링 인프라는 충분히 고민하고 선택하자**

## 💡 인사이트 & 배운 점

### 사전 질문

**1. AI가 개발자에게 미치는 영향은?**
- 걱정보다는 기대감이 조금 더 듦
- 모르거나, 잘하고 싶은 영역들을 좀 더 쉽게 접근하게 도와줌
- 개발 관련 context에 대한 이해도가 빠르게 늘고 있음
- "어떻게 잘 부려먹을까?"에 대해 생각해보자
- **기본기가 더 중요해짐**
	- AI의 응답에 대한 판단력 필요 (비판적 시점)
- GPT 안 쓰고 개발해보기 ㅋㅋ

**2. LLMOps, AI Software 엔지니어로 커리어 전환? 기존 커리어 활용?**
- 소프트웨어 엔지니어링 역량이 조금 더 중요해지고 있는 것 같음
- 데이터, 백엔드 엔지니어링 역량이 조금 되면, RAG도 생각보다 금방 잘 만들 수 있음
- 토이 프로젝트로 해본다거나, 회사에서 선적용하는 것도 괜찮음
- 앞으로 이쪽 커리어가 유지될지도 잘은 모르겠음
- **데이터를 잘하는 게 조금 더 중요해질 것**

## ❓ 질의응답 (Q&A)

**Q: GraphRAG를 안 쓴 이유를 좀 더 딥하게 말해달라**
**A:**
- Retrieval이 힘들더라
- 사람을 써야 한다
- 자동화하면 좀 덜하지만, 비용이 좀 더 크다
- 인프라 비용
- 자동화 파이프라인 생성도 비용임

**Q: Vector Database 비용 처리**
**A:**
- 정답이 진짜 없음
- PG Vector도 어느 정도 ability 있음

**Q: 만들어진 RAG에서 좋은 평가가 나왔는데, 데이터를 추가로 적재했을 때 평가가 안 좋아질 수 있는가?**
**A:**
- 맞음. 데이터가 많아지면 느려지는 건 사실

**Q: RAG를 배포할 때 MLflow로 배포 자동화 한다면, 할 게 더 많을까?**
**A:**
- 잘 모름
- 배포하는 것도 직접 개발했음 (버저닝, 트레이싱 등등)
- 자동화 비용이 크게 들진 않았던 것 같음

**Q: 도메인 전문가와 소통하는 방식 중 사건이 있었을까?**
**A:**
- 도메인 전문가에게 AI화 하는 게 중요한 걸 알려야 함
- 큰 난항은 없었던 듯
