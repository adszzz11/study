## 📌 개요

**GitHub Flow**는 GitHub에서 제안한 브랜치 전략으로, `main` 브랜치를 기준으로 **짧은-lived의 feature 브랜치**를 만들고, **Pull Request(PR)** 기반으로 리뷰 후 병합하는 방식입니다.

이 전략은 배포 속도가 중요한 **웹 서비스(SaaS)**, **스타트업**, **지속적 배포(CD)** 환경에서 특히 잘 어울립니다.

---

## 🧱 기본 원칙

1. `main` 브랜치는 **항상 배포 가능한 상태**를 유지해야 한다.
    
2. 모든 기능 개발은 `main`에서 **새로운 브랜치를 생성**해서 시작한다.
    
3. 브랜치 작업이 완료되면 **Pull Request(PR)**를 통해 코드 리뷰를 요청한다.
    
4. 코드 리뷰 후 CI가 통과하면 **`main`으로 병합(Merge)**한다.
    
5. 병합과 동시에 자동 배포(옵션)를 수행한다.
    

---

## 🌲 브랜치 구조

- `main`: 운영 환경에 배포되는 **유일한 중심 브랜치**
    
- `feature/로그인`, `fix/오류수정`: 작업 목적에 따라 만들어지는 **짧은-lived 브랜치**

```
main
 └──▶ feature/add-login-page
        └──▶ Pull Request
               └──▶ main 병합
```

### ✅ 브랜치 유형 정의

| 브랜치 유형 | 접두어 (`type`) | 설명                           |
| ------ | ------------ | ---------------------------- |
| 기능 개발  | `feature/`   | 신규 기능 구현 시                   |
| 버그 수정  | `fix/`       | 기존 기능 오류 수정 시                |
| 문서 작업  | `docs/`      | README, 주석, 내부 위키 등 문서 관련 작업 |
| 리팩토링   | `refactor/`  | 로직 변경 없는 구조 개선               |
| 스타일 수정 | `style/`     | 포맷팅, Lint, 코드 정리 등           |
| 테스트 추가 | `test/`      | 테스트 코드 작성 및 수정               |

### ❌ 왜 Git Flow는 선택하지 않았는가?

**Git Flow**는 다음과 같은 한계로 인해 현재 우리 팀의 개발 및 운영 환경에 적합하지 않습니다:

| 한계점                         | 설명                                                                                 |
| --------------------------- | ---------------------------------------------------------------------------------- |
| **브랜치 구조가 복잡함**             | `main`, `develop`, `feature/*`, `release/*`, `hotfix/*` 등 다수의 브랜치가 필요하여 관리 복잡도가 증가 |
| **정기 릴리즈에 최적화됨**            | 빠른 배포 주기를 지닌 웹서비스나 스타트업 환경과 맞지 않음                                                  |
| **Pull Request 기반 협업과 어색함** | GitHub의 PR 중심 흐름과 자연스럽게 통합되지 않음                                                    |
| **CI/CD 자동화가 번거로움**         | 병합 브랜치가 많아 배포 파이프라인 설계가 복잡해짐                                                       |
| **충돌 위험 증가**                | 병합 지연 및 병렬 브랜치 작업으로 충돌 가능성이 높음                                                     |

---

현재 개발 단계까지는 이렇게 진행할 수 있지만, 이후 본격적으로 서비스가 운영되고 비즈니스가 커진다면 변경의 필요가 생깁니다. 필요할 경우, 그 때 다시 논의합니다.
