## Elasticsearch 아키텍처

### 핵심 개념

#### 1. Cluster (클러스터)

하나 이상의 노드로 구성된 Elasticsearch 인스턴스 그룹입니다.

**특징**:
- 고가용성 제공
- 데이터 복제 및 분산
- 자동 장애 조치
- 부하 분산

**클러스터 상태**:
- **Green**: 모든 샤드가 할당됨
- **Yellow**: 일부 복제본 샤드가 할당되지 않음
- **Red**: 일부 기본 샤드가 할당되지 않음 (데이터 손실 위험)

```bash
# 클러스터 상태 확인
curl -X GET "localhost:9200/_cluster/health?pretty"

# 클러스터 설정 확인
curl -X GET "localhost:9200/_cluster/settings?pretty"
```

**출처**: [Elasticsearch Architecture | Coralogix](https://coralogix.com/guides/elasticsearch/elasticsearch-architecture-8-key-components-and-putting-them-to-work/)

#### 2. Nodes (노드)

클러스터를 구성하는 개별 Elasticsearch 인스턴스입니다.

**노드 타입별 상세**:

##### Master Node
- **역할**: 클러스터 상태 관리, 인덱스 생성/삭제, 노드 추가/제거
- **설정**:
  ```yaml
  node.roles: [ master ]
  ```
- **권장 사항**:
  - 최소 3개의 master-eligible 노드
  - 전용 마스터 노드 사용 (데이터 저장 안 함)
  - Quorum 기반 의사결정 (n/2 + 1)

**출처**: [Elasticsearch Master Nodes | Idlemind.dev](https://idlemind.dev/posts/elasticsearch_master_nodes/)

##### Data Node
- **역할**: 데이터 저장, 검색, 집계, CRUD 작업
- **타입별 분류**:
  ```yaml
  # Hot 노드 (활발한 읽기/쓰기)
  node.roles: [ data_hot ]

  # Warm 노드 (읽기 전용)
  node.roles: [ data_warm ]

  # Cold 노드 (드문 액세스)
  node.roles: [ data_cold ]

  # Frozen 노드 (아카이브)
  node.roles: [ data_frozen ]
  ```

##### Coordinating Node
- **역할**: 요청 라우팅, 결과 집계
- **동작 방식**:
  1. 클라이언트 요청 수신
  2. 관련 샤드가 있는 데이터 노드로 요청 전달
  3. 각 노드의 결과를 수집 및 병합
  4. 최종 결과를 클라이언트에 반환

```yaml
# 전용 Coordinating 노드
node.roles: [ ]  # 빈 배열
```

**출처**: [Elasticsearch How Coordinating node works? | Stack Overflow](https://stackoverflow.com/questions/66649669/elasticsearch-how-coordinating-node-works)

##### Ingest Node
- **역할**: 인덱싱 전 데이터 전처리
- **설정**:
  ```yaml
  node.roles: [ ingest ]
  ```

**출처**: [Node roles | Elastic Docs](https://www.elastic.co/docs/deploy-manage/distributed-architecture/clusters-nodes-shards/node-roles)

#### 3. Indices (인덱스)

관련 문서의 논리적 컬렉션입니다.

**인덱스 구조**:
```
Index
  ├── Mappings (스키마 정의)
  ├── Settings (구성 설정)
  └── Shards (물리적 파티션)
       ├── Primary Shard
       └── Replica Shard
```

**인덱스 생성 예시**:
```json
PUT /logs-2025.10.06
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "index.codec": "best_compression",
    "refresh_interval": "30s"
  },
  "mappings": {
    "properties": {
      "@timestamp": {
        "type": "date",
        "format": "strict_date_optional_time||epoch_millis"
      },
      "level": {
        "type": "keyword"
      },
      "message": {
        "type": "text",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      },
      "user_id": {
        "type": "keyword"
      },
      "response_time": {
        "type": "integer"
      }
    }
  }
}
```

#### 4. Shards (샤드)

인덱스를 여러 조각으로 나눈 물리적 파티션입니다.

**샤드 유형**:

##### Primary Shard
- 원본 데이터 저장
- 쓰기 작업 처리
- 인덱스 생성 시 설정 (이후 변경 불가)

##### Replica Shard
- Primary 샤드의 복사본
- 읽기 성능 향상
- 고가용성 제공
- 동적으로 변경 가능

**샤드 크기 권장사항**:
- 10-50GB per shard (최적)
- 최대 샤드 크기: 50-100GB
- 노드당 샤드 수: 20-25개 per GB heap

```bash
# 샤드 상태 확인
curl -X GET "localhost:9200/_cat/shards?v"

# 샤드 할당 확인
curl -X GET "localhost:9200/_cat/allocation?v"
```

**출처**: [Demystifying Elasticsearch: nodes, clusters, shards | Medium](https://learncsdesigns.medium.com/demystifying-elasticsearch-understanding-nodes-clusters-shards-and-indices-part-2-c1835e1e061e)

### Elasticsearch 내부 동작

#### 인덱싱 프로세스

```
1. 클라이언트 요청
   ↓
2. Coordinating Node가 요청 수신
   ↓
3. 문서 ID 해싱하여 Primary Shard 결정
   ↓
4. Primary Shard에 문서 기록
   ↓
5. Replica Shard에 복제
   ↓
6. 클라이언트에 응답 반환
```

**세부 단계**:

1. **Document Routing**:
   ```
   shard_num = hash(_routing) % num_primary_shards
   ```

2. **Write to Primary**:
   - Memory buffer에 기록
   - Translog에 기록 (내구성)
   - Refresh 시 검색 가능

3. **Replication**:
   - Primary → Replica 동기화
   - 모든 Replica 성공 후 응답

4. **Refresh**:
   - Memory buffer → Filesystem cache
   - 기본 1초마다 실행
   - 검색 가능 상태로 전환

5. **Flush**:
   - Filesystem cache → Disk
   - Translog 비우기
   - 기본 30분마다 또는 Translog 크기 초과시

#### 검색 프로세스

```
1. 검색 요청
   ↓
2. Query Phase (각 샤드에서 독립적 검색)
   ↓
3. Fetch Phase (실제 문서 가져오기)
   ↓
4. 결과 병합 및 정렬
   ↓
5. 클라이언트 응답
```

**Query Phase**:
```json
POST /logs-*/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "level": "ERROR" } }
      ],
      "filter": [
        { "range": { "@timestamp": { "gte": "now-1h" } } }
      ]
    }
  },
  "size": 100,
  "from": 0,
  "sort": [
    { "@timestamp": { "order": "desc" } }
  ]
}
```

**Fetch Phase**:
- Top N 문서의 실제 내용 가져오기
- 각 샤드에서 관련 문서 검색
- Coordinating Node가 최종 병합

### Mapping (매핑)

문서의 필드 타입과 인덱싱 방법을 정의합니다.

**주요 필드 타입**:

| 타입 | 용도 | 예시 |
|-----|------|-----|
| `text` | 전문 검색 | 로그 메시지, 설명 |
| `keyword` | 정확한 매칭, 집계 | 상태 코드, 태그, ID |
| `date` | 날짜/시간 | 타임스탬프 |
| `long`/`integer` | 정수 | 응답 시간, 카운트 |
| `float`/`double` | 실수 | 비율, 점수 |
| `boolean` | 참/거짓 | 플래그 |
| `ip` | IP 주소 | 클라이언트 IP |
| `geo_point` | 지리 좌표 | 위도/경도 |
| `object` | 중첩 객체 | JSON 객체 |
| `nested` | 독립 문서 배열 | 복잡한 구조 |

**Dynamic Mapping vs Explicit Mapping**:

```json
// Explicit Mapping (권장)
PUT /logs
{
  "mappings": {
    "dynamic": "strict",  // 새 필드 거부
    "properties": {
      "@timestamp": { "type": "date" },
      "level": { "type": "keyword" },
      "message": { "type": "text" }
    }
  }
}

// Dynamic Mapping (자동)
// Elasticsearch가 데이터 타입 추론
// 문제: 타입 추론 오류 가능
```

## Logstash 파이프라인

### 파이프라인 아키텍처

```
[Input Thread] → [Queue] → [Filter Workers] → [Output Workers]
```

#### Input Stage

**동작 방식**:
- 각 Input 플러그인은 독립 스레드에서 실행
- 이벤트 생성 및 Queue로 전달
- 다양한 소스에서 동시 수집 가능

**주요 Input 플러그인 상세**:

##### 1. Beats Input
```ruby
input {
  beats {
    port => 5044
    type => "beats"

    # SSL/TLS 설정
    ssl => true
    ssl_certificate => "/path/to/cert.crt"
    ssl_key => "/path/to/cert.key"

    # 클라이언트 인증
    ssl_verify_mode => "force_peer"
    ssl_certificate_authorities => ["/path/to/ca.crt"]
  }
}
```

##### 2. HTTP Input
```ruby
input {
  http {
    port => 8080
    codec => json

    # 기본 인증
    user => "logstash"
    password => "secret"

    # 추가 메타데이터
    additional_codecs => {
      "application/json" => "json"
    }
  }
}
```

##### 3. File Input
```ruby
input {
  file {
    path => "/var/log/app/*.log"
    start_position => "beginning"  # 또는 "end"
    sincedb_path => "/dev/null"    # 개발용 (처음부터 읽기)

    # 파일 디스커버리
    discover_interval => 15

    # 멀티라인 처리
    codec => multiline {
      pattern => "^%{TIMESTAMP_ISO8601}"
      negate => true
      what => "previous"
    }
  }
}
```

**출처**: [Logstash Pipeline Configuration | Elastic Docs](https://www.elastic.co/guide/en/logstash/current/pipeline.html)

#### Filter Stage

파이프라인의 핵심으로, 데이터를 파싱하고 변환합니다.

##### 1. Grok Filter - 비구조화 데이터 파싱

**Grok 패턴**:
```ruby
filter {
  grok {
    match => {
      "message" => "%{PATTERN:field_name}"
    }
  }
}
```

**내장 패턴 예시**:
- `%{IP:clientip}` - IP 주소
- `%{TIMESTAMP_ISO8601:timestamp}` - ISO 타임스탬프
- `%{NUMBER:response_time}` - 숫자
- `%{WORD:method}` - 단어
- `%{GREEDYDATA:message}` - 나머지 모든 데이터

**Apache 로그 파싱**:
```ruby
filter {
  grok {
    match => {
      "message" => "%{COMBINEDAPACHELOG}"
    }
  }

  # 실패시 태그 추가
  if "_grokparsefailure" in [tags] {
    mutate {
      add_field => { "parse_error" => "grok failed" }
    }
  }
}
```

**커스텀 패턴**:
```ruby
filter {
  grok {
    patterns_dir => ["/etc/logstash/patterns"]
    match => {
      "message" => "%{CUSTOM_PATTERN:field}"
    }
  }
}

# /etc/logstash/patterns/custom
CUSTOM_PATTERN \[%{TIMESTAMP_ISO8601:timestamp}\] %{LOGLEVEL:level} %{GREEDYDATA:msg}
```

##### 2. Mutate Filter - 필드 조작

```ruby
filter {
  mutate {
    # 필드 이름 변경
    rename => {
      "old_field" => "new_field"
    }

    # 필드 제거
    remove_field => [ "unnecessary_field", "another_field" ]

    # 필드 추가
    add_field => {
      "environment" => "production"
      "indexed_at" => "%{@timestamp}"
    }

    # 태그 추가
    add_tag => [ "processed", "production" ]

    # 타입 변환
    convert => {
      "response_time" => "integer"
      "cpu_usage" => "float"
    }

    # 문자열 대소문자 변환
    lowercase => [ "level" ]
    uppercase => [ "environment" ]

    # 문자열 치환
    gsub => [
      "message", "/", "_"
    ]
  }
}
```

##### 3. Date Filter - 날짜 파싱

```ruby
filter {
  date {
    match => [
      "timestamp",
      "dd/MMM/yyyy:HH:mm:ss Z",
      "ISO8601",
      "UNIX"
    ]
    target => "@timestamp"
    timezone => "Asia/Seoul"

    # 파싱 실패시
    tag_on_failure => [ "_dateparsefailure" ]
  }
}
```

##### 4. GeoIP Filter - 지리 정보 추가

```ruby
filter {
  geoip {
    source => "clientip"
    target => "geoip"

    # 특정 필드만 추가
    fields => [
      "city_name",
      "country_name",
      "continent_code",
      "location"
    ]
  }
}
```

##### 5. JSON Filter - JSON 파싱

```ruby
filter {
  json {
    source => "message"
    target => "parsed"

    # 파싱 실패시 무시
    skip_on_invalid_json => true
  }
}
```

##### 6. 조건부 처리

```ruby
filter {
  if [level] == "ERROR" {
    mutate {
      add_tag => [ "error" ]
    }
  } else if [level] == "WARN" {
    mutate {
      add_tag => [ "warning" ]
    }
  }

  # 특정 필드 존재 여부
  if [user_id] {
    elasticsearch {
      hosts => ["localhost:9200"]
      query => "user_id:%{[user_id]}"
      fields => { "username" => "user_name" }
    }
  }
}
```

**출처**: [Logstash Filters | Tutorials Point](https://www.tutorialspoint.com/logstash/logstash_filters.htm)

#### Output Stage

처리된 이벤트를 목적지로 전송합니다.

##### Elasticsearch Output (상세 설정)

```ruby
output {
  elasticsearch {
    hosts => ["es-node1:9200", "es-node2:9200", "es-node3:9200"]

    # 인덱스 설정
    index => "logs-%{[service]}-%{+YYYY.MM.dd}"
    document_id => "%{[@metadata][fingerprint]}"  # 중복 방지

    # 보안
    user => "logstash_writer"
    password => "${ES_PASSWORD}"
    ssl => true
    cacert => "/etc/logstash/certs/ca.crt"

    # 성능 튜닝
    bulk_size => 5000
    flush_size => 5000
    idle_flush_time => 1

    # 재시도 설정
    retry_on_conflict => 3

    # ILM 사용
    ilm_enabled => true
    ilm_rollover_alias => "logs"
    ilm_pattern => "000001"
    ilm_policy => "logs-policy"
  }
}
```

##### 조건부 Output

```ruby
output {
  # 에러 로그는 별도 인덱스
  if [level] == "ERROR" {
    elasticsearch {
      hosts => ["localhost:9200"]
      index => "errors-%{+YYYY.MM.dd}"
    }

    # 알림도 전송
    email {
      to => "ops@example.com"
      subject => "Error detected: %{[message]}"
    }
  } else {
    elasticsearch {
      hosts => ["localhost:9200"]
      index => "logs-%{+YYYY.MM.dd}"
    }
  }
}
```

**출처**: [Logstash Output Stage | Tutorials Point](https://www.tutorialspoint.com/logstash/logstash_output_stage.htm)

### 파이프라인 성능 튜닝

#### 1. Pipeline Workers

```yaml
# /etc/logstash/logstash.yml
pipeline.workers: 4  # CPU 코어 수와 동일하게
pipeline.batch.size: 125  # 기본값
pipeline.batch.delay: 50  # 밀리초
```

#### 2. Queue 설정

```yaml
# Memory Queue (기본)
queue.type: memory
queue.max_bytes: 1gb

# Persistent Queue (권장 - 프로덕션)
queue.type: persisted
queue.max_bytes: 4gb
path.queue: /var/lib/logstash/queue
```

#### 3. JVM 설정

```bash
# /etc/logstash/jvm.options
-Xms2g  # 최소 힙
-Xmx2g  # 최대 힙 (최소와 동일)
```

## Beats 상세

### Filebeat 고급 기능

#### 1. Filebeat Modules

사전 구성된 로그 수집 및 파싱 설정입니다.

**활성화 방법**:
```bash
filebeat modules enable nginx
filebeat modules enable mysql
filebeat modules enable system
```

**모듈 설정 커스터마이징**:
```yaml
# /etc/filebeat/modules.d/nginx.yml
- module: nginx
  access:
    enabled: true
    var.paths: ["/var/log/nginx/access.log*"]
  error:
    enabled: true
    var.paths: ["/var/log/nginx/error.log*"]
```

#### 2. Filebeat Processors

이벤트를 전송 전에 전처리합니다.

```yaml
processors:
  # 호스트 메타데이터 추가
  - add_host_metadata:
      netinfo.enabled: true

  # 클라우드 메타데이터
  - add_cloud_metadata: ~

  # 필드 삭제
  - drop_fields:
      fields: ["agent.ephemeral_id", "agent.id"]

  # 조건부 삭제
  - drop_event:
      when:
        regexp:
          message: "^DEBUG"

  # 필드 이름 변경
  - rename:
      fields:
        - from: "message"
          to: "log_message"

  # 스크립트 프로세서 (JavaScript)
  - script:
      lang: javascript
      source: >
        function process(event) {
          event.Put("custom_field", "value");
        }
```

#### 3. Input 유형

```yaml
filebeat.inputs:
  # 로그 파일
  - type: log
    enabled: true
    paths: ["/var/log/*.log"]

  # Container 로그
  - type: container
    enabled: true
    paths:
      - '/var/lib/docker/containers/*/*.log'

  # Syslog
  - type: syslog
    protocol.tcp:
      host: "0.0.0.0:514"

  # TCP
  - type: tcp
    host: "0.0.0.0:9000"

  # Kafka
  - type: kafka
    hosts: ["kafka1:9092", "kafka2:9092"]
    topics: ["logs"]
    group_id: "filebeat"
```

### Metricbeat 고급 기능

#### System Module 상세 설정


### 3. 장애 대응

**일반적인 문제와 해결책**:

| 문제 | 원인 | 해결책 |
|-----|------|-------|
| Yellow 상태 | Replica 미할당 | 노드 추가 또는 replica 수 감소 |
| High Heap | 메모리 부족 | JVM 힙 증가 또는 노드 추가 |
| Slow Search | 큰 인덱스, 비효율적 쿼리 | 인덱스 분할, 쿼리 최적화 |
