# Agentic Document Scraper - 과제 분석

## 📋 과제 개요

Georgia 주의 지방자치단체 웹사이트에서 도시계획 및 구역설정(Zoning) 회의 문서를 자동으로 수집하는 **에이전트 기반 시스템** 구축

**대상 지역:**
- Cherokee County (비법인 지역)
- City of Holly Springs
- City of Alpharetta
- City of Marietta

**수집 대상 문서:**
- 회의록 (Meeting minutes) - PDF 또는 HTML
- 회의 안건 (Meeting agendas) - PDF 또는 HTML
- 안건 패킷 (Agenda packets) - 대용량 PDF
- 비디오 녹화 링크 - Granicus, YouTube, Vimeo 등

---

## 🎯 핵심 요구사항 분석

### 1. Agentic System 구축
**핵심 특징:**
- 단순 스크래퍼가 아닌 "자율적으로 판단하고 행동하는 에이전트"
- 각 지자체 사이트의 구조 변화에 적응할 수 있어야 함
- 문서 타입을 자동으로 분류하고 적절한 수집 전략 선택

**Agentic의 의미:**
```
일반 스크래퍼: 고정된 규칙 기반 → 사이트 변경 시 즉시 실패
에이전트 시스템: 목표 기반 → 사이트 구조 이해하고 적응적으로 대응
```

### 2. 두 단계 운영 모드

#### Phase 1: Backfill (과거 데이터 수집)
- **목표**: 최근 1-2년 간의 모든 회의 문서 수집
- **도전과제**:
  - 대량의 문서를 효율적으로 처리
  - 중복 다운로드 방지
  - 페이지네이션 처리
  - Rate limiting 고려

#### Phase 2: Continuous Updates (지속적 업데이트)
- **목표**: 신규 회의 문서 자동 감지 및 수집
- **도전과제**:
  - 변경 감지 메커니즘 (polling, webhook 등)
  - 스케줄링 전략
  - 실패 재시도 로직
  - 알림/모니터링

### 3. 웹사이트 다양성 문제

**예상되는 다양한 패턴:**
- 정적 HTML 페이지
- JavaScript 렌더링 필요한 SPA
- 검색/필터 기능이 있는 동적 페이지
- PDF 직접 링크 vs 중간 페이지 경유
- 다양한 날짜 형식 및 레이아웃

**각 지자체별 특성 파악 필요:**
- 사이트 구조 분석
- DOM 구조 패턴 식별
- 문서 명명 규칙
- 업데이트 주기

## 🛠️ 기술적 도전 과제

### 1. 웹 스크래핑 기술 스택
**고려 사항:**
- **정적 페이지**: BeautifulSoup, lxml
- **동적 페이지**: Playwright, Selenium
- **API 발견**: DevTools로 XHR/Fetch 요청 분석

### 2. AI/LLM 활용 포인트
**Agentic 특성을 구현하기 위한 LLM 활용:**

1. **페이지 구조 이해**
   - HTML을 분석하여 문서 링크 위치 자동 파악
   - 테이블, 리스트, 카드 등 다양한 레이아웃 패턴 인식

2. **문서 분류**
   - 파일명/링크 텍스트로 문서 타입 판단
   - Minutes vs Agenda vs Packet 자동 분류

3. **날짜 추출**
   - 다양한 형식의 날짜 표현 정규화
   - 회의 날짜 추출 및 파싱

4. **에러 복구**
   - 스크래핑 실패 시 대안 전략 수립
   - DOM 구조 변경 감지 및 새로운 선택자 생성

### 3. 데이터 파이프라인
```
Discovery → Extraction → Classification → Storage → Validation
```

**각 단계별 고려사항:**
- **Discovery**: 어떤 페이지를 방문할지 결정
- **Extraction**: 문서 링크 및 메타데이터 추출
- **Classification**: 문서 타입 및 중요도 분류
- **Storage**: 파일 저장 구조 및 메타데이터 DB
- **Validation**: 다운로드 완료 검증, 중복 체크

### 4. 확장성 및 유지보수
- **모듈화**: 각 지자체별 어댑터 패턴
- **설정 관리**: 지자체별 설정 외부화
- **로깅**: 상세한 실행 로그 및 에러 추적
- **테스트**: 각 지자체별 테스트 케이스

## 🏗️ 아키텍처 설계 방향

### Option 1: Rule-based with AI Fallback
```
1. 각 지자체별 규칙 정의 (CSS selector, XPath)
2. 규칙 실패 시 LLM에게 페이지 분석 요청
3. LLM이 새로운 규칙 제안 → 검증 → 적용
```

**장점**: 빠르고 비용 효율적 (규칙 작동 시)
**단점**: 초기 규칙 작성 시간 필요

### Option 2: Full Agentic Approach
```
1. LLM에게 목표 제시 ("이 페이지에서 모든 회의록 찾기")
2. LLM이 페이지 분석하고 액션 수행
3. 결과 검증 및 피드백
```

**장점**: 높은 적응성, 새 지자체 추가 용이
**단점**: 비용, 속도, 안정성

### Option 3: Hybrid (추천)
```
1. 첫 방문: LLM이 사이트 구조 학습
2. 패턴 추출 및 규칙 생성
3. 이후: 규칙 기반으로 실행
4. 변경 감지 시: LLM 재개입으로 규칙 업데이트
```

**균형잡힌 접근**: 비용 효율 + 적응성

## 📊 구현 우선순위

### MVP (Minimum Viable Product)
**목표**: 1개 지자체에서 1가지 문서 타입 성공적으로 수집

1. **단일 지자체 선택** (예: City of Alpharetta)
   - 사이트 구조가 상대적으로 단순한 곳 우선
   - 수동으로 구조 분석

2. **회의록만 수집** (가장 일반적인 문서)
   - PDF 다운로드 및 저장
   - 메타데이터 추출 (날짜, 제목)

3. **Backfill 기능**
   - 최근 6개월 데이터로 제한
   - 순차 처리 (병렬 처리 X)

4. **기본 에러 핸들링**
   - 네트워크 에러 재시도
   - 실패 로깅

### Phase 2
5. **다른 문서 타입 추가**
   - Agendas, Packets, Video links

6. **2번째 지자체 추가**
   - 코드 일반화 및 리팩토링
   - 공통 인터페이스 정의

7. **LLM 통합**
   - 문서 분류
   - 페이지 구조 분석

### Phase 3
8. **나머지 지자체 추가**
9. **Continuous update 모드**
10. **모니터링 및 알림**

## 💡 기술 스택 제안

### Core
- **Language**: Python 3.11+
- **Scraping**:
  - `httpx` - 비동기 HTTP 클라이언트
  - `playwright` - 동적 페이지 처리
  - `beautifulsoup4` - HTML 파싱
- **AI/LLM**:
  - Anthropic Claude API (구조화된 출력에 강점)
  - OpenAI GPT-4 (대안)

### 데이터 관리
- **Database**: SQLite (MVP) → PostgreSQL (프로덕션)
- **Storage**: Local filesystem → S3
- **Queue**: Redis (재시도 큐)

### 스케줄링 & 모니터링
- **Scheduler**: APScheduler (MVP) → Airflow (프로덕션)
- **Logging**: structlog
- **Monitoring**: Sentry (에러 추적)

### 개발 도구
- **Testing**: pytest, pytest-asyncio
- **Code quality**: ruff, mypy
- **Environment**: poetry or uv
