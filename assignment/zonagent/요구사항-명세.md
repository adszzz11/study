## 개요

### 프로젝트명
**ZonAgent - Agentic Document Scraper for Georgia Municipalities**

### 배경
Georgia주 지자체의 Planning & Zoning 회의 문서를 자동으로 수집하는 시스템 구축. 각 지자체 웹사이트는 구조와 형식이 다양하므로, LLM을 활용한 Agentic 접근 방식이 필요함.

### 평가 기준
1. **B - Agentic 접근 방식**: LLM을 어떻게 활용했는지
2. **D - 우선순위 판단 능력**: 무엇을 선택하고 포기했는지

---

## 핵심 목표

### 1. 완전한 커버리지

**4개 지자체 전부 구현**:
- Cherokee County (unincorporated) - Granicus 플랫폼
- City of Marietta - CivicEngage 플랫폼
- City of Alpharetta - CivicClerk SPA
- City of Holly Springs - CivicClerk SPA

**4개 문서 타입 전부 수집**:
- Meeting Minutes (PDF/HTML)
- Meeting Agendas (PDF/HTML)
- Agenda Packets (PDF)
- Video recording links (Granicus, YouTube, Vimeo)

### 2. 운영 모드

**Backfill 모드**:
- 과거 1-2년 회의 데이터 수집
- 초기 데이터 구축

**Continuous 모드**:
- 새 회의 자동 감지 및 수집
- 지자체별 동적 트리거 설정:
  - **MANUAL**: API 호출 시에만 실행
  - **PERIODIC**: 고정 주기 (6시간, 12시간 등)
  - **CRON**: Cron 표현식 (매일 자정, 매주 월요일 등)
  - **EVENT**: 외부 이벤트 기반 (추후 구현)

### 3. 확장 가능성

새로운 지자체 추가 시:
- 수동 코딩 최소화
- LLM이 자동으로 규칙 생성
- 플러그인 방식으로 추가

### 실행 흐름

```
1. Spring Boot Scheduler
   → 트리거 조건 충족 (Cron/Periodic/Manual)

2. ScraperExecutor
   → ProcessBuilder로 Python Agent 실행
   → python scraper.py --jurisdiction cherokee --api-url http://localhost:8080/api

3. Python Agent
   → 규칙 생성/로드
   → 웹 스크래핑 (Playwright)
   → LLM 데이터 추출
   → 신뢰성 검증
   → PDF → .md 변환
   → 중요 데이터 추출

4. Python → Spring Boot
   → POST /api/meetings/bulk (회의 데이터)
   → POST /api/files/upload (파일 업로드)
   → POST /api/rules/save (규칙 저장)
   → POST /api/extracted/save (추출 데이터)

5. Spring Boot
   → PostgreSQL 저장 (메타데이터)
   → MongoDB 저장 (추출 데이터)
   → File System 저장 (원본 + .md)
```

### 동적 스케줄러

```kotlin
@Service
class DynamicScraperScheduler(
    private val jurisdictionConfigRepository: JurisdictionConfigRepository,
    private val scraperExecutor: ScraperExecutor,
    private val taskScheduler: TaskScheduler
) {

    private val scheduledTasks = ConcurrentHashMap<String, ScheduledFuture<*>>()

    @PostConstruct
    fun initializeSchedulers() {
        jurisdictionConfigRepository.findAll()
            .filter { it.enabled }
            .forEach { registerScheduler(it) }
    }

    fun registerScheduler(config: JurisdictionConfig) {
        cancelScheduler(config.name)

        when (config.triggerType) {
            TriggerType.MANUAL -> {
                // 스케줄 등록 안 함
            }

            TriggerType.PERIODIC -> {
                val task = taskScheduler.scheduleAtFixedRate(
                    { scraperExecutor.execute(config.name) },
                    Duration.ofMinutes(config.periodMinutes!!.toLong())
                )
                scheduledTasks[config.name] = task
            }

            TriggerType.CRON -> {
                val task = taskScheduler.schedule(
                    { scraperExecutor.execute(config.name) },
                    CronTrigger(config.cronExpression!!)
                )
                scheduledTasks[config.name] = task
            }

            TriggerType.EVENT -> {
                // 추후 구현
            }
        }
    }

    fun cancelScheduler(jurisdictionName: String) {
        scheduledTasks[jurisdictionName]?.cancel(false)
        scheduledTasks.remove(jurisdictionName)
    }

    fun updateScheduler(config: JurisdictionConfig) {
        jurisdictionConfigRepository.save(config)
        registerScheduler(config)
    }
}
```

## 기능 요구사항

### 1. 규칙 생성 (Rule Generation)

**입력**:
- 지자체 웹사이트 URL
- 샘플 HTML

**LLM 태스크**:
1. 회의 목록 페이지에서 회의 행들 찾기
2. 각 회의 행에서 제목, 날짜, 링크 추출
3. 문서 다운로드 링크들 찾기

**출력 형식** (JSON):
```json
{
  "jurisdiction": "cherokee_county",
  "description": "Cherokee County 회의 목록 추출 규칙",
  "steps": [
    {
      "step": 1,
      "action": "find_container",
      "selector": "#AgendaCenterContent",
      "validation": "컨테이너가 존재하고 table을 포함해야 함"
    },
    {
      "step": 2,
      "action": "find_meetings",
      "selector": "table tbody tr",
      "description": "각 tr이 하나의 회의"
    },
    {
      "step": 3,
      "action": "extract_fields",
      "fields": {
        "title": {
          "selector": "td:nth-child(2) a",
          "type": "text",
          "required": true
        },
        "date": {
          "selector": "td:nth-child(1)",
          "type": "date",
          "format": "MM/DD/YYYY"
        },
        "url": {
          "selector": "td:nth-child(2) a",
          "attribute": "href"
        }
      }
    }
  ]
}
```

**실행 시점**:
- 최초 1회 생성 → JSON 파일 저장
- 이후 재사용
- 신뢰성 검증 실패 시 재생성

### 2. 데이터 스크래핑 (Scraping)

**프로세스**:
1. 저장된 규칙 로드
2. LLM에게 규칙 + HTML 전달
3. LLM이 규칙에 따라 데이터 추출
4. 추출된 데이터 반환

### 3. 신뢰성 검증 (Reliability Validation)

**검증 시점**:
- 데이터 추출 후 매번

**LLM 태스크**:
- 추출된 데이터가 올바른 회의 정보인지 검증
- 필드별 신뢰도 평가

**출력 형식**:
```json
{
  "overall_confidence": 0.85,
  "field_validation": {
    "title": {
      "valid": true,
      "confidence": 0.95
    },
    "date": {
      "valid": true,
      "confidence": 0.90
    },
    "url": {
      "valid": false,
      "confidence": 0.50,
      "issue": "일부 URL이 상대경로"
    }
  }
}
```

**재생성 트리거**:
- `overall_confidence < 0.8`
- 또는 특정 필드 `confidence < 0.7`

### 4. 데이터 정제 (Data Cleaning)

**Minutes PDF → .md 변환**:
- LLM에게 PDF 내용 전달
- Markdown 형식으로 변환
- 구조화된 포맷 적용

**기타 PDF (Agenda, Packet)**:
- 원본 다운로드
- .md 파일에 링크 걸기

**파일 구조**:
```
data/
├── cherokee_county/
│   ├── meeting-2024-01-15/
│   │   ├── minutes.md        # LLM 변환
│   │   ├── agenda.pdf        # 원본 다운로드
│   │   ├── packet.pdf
│   │   └── metadata.json
```

### 5. 중요 데이터 추출 (Key Data Extraction)

**추출 항목**:

1. **회의 메타 정보**:
   - 회의 일시
   - 참석자
   - 장소
   - 회의 종류

2. **안건 및 결정사항**:
   - 논의된 안건 목록
   - 투표 결과
   - 승인/거부 항목

3. **Planning & Zoning 특화 정보**:
   - 개발 프로젝트명
   - 신청자/개발자
   - 위치/주소
   - 승인/조건부승인/거부

4. **요약**:
   - 회의 전체 요약 (3-5문장)
   - 주요 결정사항 요약

## 기술 스택

### Backend (Spring Boot - Kotlin)

**프레임워크**:
- Spring Boot 3.x
- Kotlin 1.9+

**주요 기능**:
- REST API 서버 (Python Agent 데이터 수신)
- 동적 스케줄러 (Cron, Periodic)
- 스크래퍼 실행기 (ProcessBuilder)
- Database 관리

**라이브러리**:
- Spring Data JPA (PostgreSQL)
- Spring Data MongoDB
- Spring WebFlux (비동기 HTTP)
- Kotlin Coroutines

### Scraper (Python - CLI Agent)

**실행 방식**:
- 독립 CLI 프로세스
- Spring Boot가 subprocess로 실행

**주요 라이브러리**:
- **Playwright**: JavaScript SPA 스크래핑
- **BeautifulSoup4**: HTML 파싱
- **httpx**: HTTP 요청 (정적 페이지)
- **Anthropic SDK**: Claude API 호출
- **PyPDF2 / pdfplumber**: PDF 텍스트 추출
- **requests**: Kotlin API 호출

### Database

**PostgreSQL**:
- 구조화된 메타데이터
- 회의, 문서, 지자체 테이블
- 관계형 쿼리

**MongoDB**:
- 정제/추출된 JSON 데이터
- 회의록 분석 결과
- 유연한 스키마

**파일 시스템**:
- 원본 PDF, HTML
- 변환된 .md 파일

### AI

**Anthropic Claude API**:
- 모델: Claude 3.5 Sonnet (또는 최신)
- 용도:
  - 규칙 생성
  - 데이터 추출
  - 신뢰성 검증
  - PDF → Markdown 변환
  - 중요 데이터 추출

6. 검증 실패 → 규칙 재생성 → 재실행
7. Kotlin API로 데이터 전송

**출력**:
- PostgreSQL: 회의 메타데이터
- 파일 시스템: 원본 HTML, PDF

### 2단계: Data Cleaning

**프로세스**:
- **Minutes PDF**:
  1. PDF 텍스트 추출
  2. LLM에게 전달
  3. Markdown 변환
  4. `minutes.md` 저장
  5. Kotlin API로 파일 업로드

- **기타 PDF (Agenda, Packet)**:
  1. 원본 다운로드
  2. 파일 저장
  3. `minutes.md`에 링크 추가

**출력**:
- 파일 시스템:
  ```
  meeting-2024-01-15/
  ├── minutes.md
  ├── agenda.pdf
  ├── packet.pdf
  └── metadata.json
  ```

### 3단계: Key Data Extraction

**프로세스**:
1. LLM에게 `minutes.md` 전달
2. 구조화된 정보 추출
3. JSON 생성
4. Kotlin API로 전송

**출력**:
- MongoDB: 추출된 JSON 데이터
- PostgreSQL: 추출 완료 상태 업데이트

## Agentic 시스템 설계

### Agentic의 정의

**시나리오 C - 매번 LLM 사용 + 태스크 분할**:
- 문서 수집 시마다 LLM 호출
- 작은 태스크로 나눠서 질문
- 실시간 데이터 추출

### 규칙 기반 + LLM 하이브리드

**프로세스 1: 규칙 정의** (최초 1회):
```
LLM에게 질문:
"이 HTML에서 회의 목록을 추출하려면 어떤 단계가 필요한가?"

LLM 응답:
{
  "steps": [
    {"action": "find_container", "selector": "..."},
    {"action": "find_meetings", "selector": "..."},
    {"action": "extract_fields", "fields": {...}}
  ]
}

→ JSON 파일로 저장
→ POST /api/rules/save
```

**프로세스 2: 데이터 추출** (매번):
```
LLM에게 전달:
- 규칙 (JSON)
- 현재 HTML

LLM 응답:
- 추출된 회의 데이터 (JSON)

→ POST /api/meetings/bulk
```

### Self-Healing (자가 치유)

**신뢰성 검증**:
```
데이터 추출 후:
→ LLM에게 질문: "이 데이터가 올바른 회의 정보인가?"
→ 필드별 신뢰도 점수 반환
```

**재생성 트리거**:
```
if overall_confidence < 0.8:
    1. 규칙이 잘못되었다고 판단
    2. 규칙 재생성 (프로세스 1)
    3. 새 규칙으로 재실행 (프로세스 2)
    4. POST /api/rules/save (새 규칙)
```

## 스토리지 전략

### PostgreSQL (구조화된 메타데이터)

**테이블**:

```sql
-- 지자체 설정
CREATE TABLE jurisdiction_config (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL UNIQUE,
    display_name VARCHAR(255),
    base_url TEXT,
    trigger_type VARCHAR(50),
    cron_expression VARCHAR(100),
    period_minutes INTEGER,
    enabled BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 회의
CREATE TABLE meetings (
    id SERIAL PRIMARY KEY,
    jurisdiction_id INTEGER REFERENCES jurisdiction_config(id),
    title TEXT,
    meeting_date TIMESTAMP,
    url TEXT,
    status VARCHAR(50),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 문서
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    meeting_id INTEGER REFERENCES meetings(id),
    doc_type VARCHAR(50),
    url TEXT,
    file_path TEXT,
    file_size BIGINT,
    checksum VARCHAR(64),
    status VARCHAR(50),
    created_at TIMESTAMP DEFAULT NOW()
);

-- 스크래핑 실행 이력
CREATE TABLE scraping_runs (
    id SERIAL PRIMARY KEY,
    jurisdiction_id INTEGER REFERENCES jurisdiction_config(id),
    run_type VARCHAR(50),
    started_at TIMESTAMP,
    finished_at TIMESTAMP,
    status VARCHAR(50),
    meetings_found INTEGER,
    documents_downloaded INTEGER,
    errors JSONB
);
```
