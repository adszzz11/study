## 개요

### 프로젝트명
**ZonAgent - Agentic Document Scraper for Georgia Municipalities**

### 배경
Georgia주 지자체의 Planning & Zoning 회의 문서를 자동으로 수집하는 시스템 구축. 각 지자체 웹사이트는 구조와 형식이 다양하므로, LLM을 활용한 Agentic 접근 방식이 필요함.

### 평가 기준
1. **B - Agentic 접근 방식**: LLM을 어떻게 활용했는지
2. **D - 우선순위 판단 능력**: 무엇을 선택하고 포기했는지

---

## 핵심 목표

### 1. 완전한 커버리지

**4개 지자체 전부 구현**:
- Cherokee County (unincorporated) - Granicus 플랫폼
- City of Marietta - CivicEngage 플랫폼
- City of Alpharetta - CivicClerk SPA
- City of Holly Springs - CivicClerk SPA

**4개 문서 타입 전부 수집**:
- Meeting Minutes (PDF/HTML)
- Meeting Agendas (PDF/HTML)
- Agenda Packets (PDF)
- Video recording links (Granicus, YouTube, Vimeo)

### 2. 운영 모드

**Backfill 모드**:
- 과거 1-2년 회의 데이터 수집
- 초기 데이터 구축

**Continuous 모드**:
- 새 회의 자동 감지 및 수집
- 지자체별 트리거 선택:
  - **Cron**: 매일 정해진 시간 (예: Cherokee)
  - **Polling**: 주기적 체크 (예: Marietta)
  - **Event-based**: 변경 감지 시 (예: Alpharetta)
  - **Manual API**: 수동 트리거 (예: Holly Springs)

### 3. 확장 가능성

새로운 지자체 추가 시:
- 수동 코딩 최소화
- LLM이 자동으로 규칙 생성
- 플러그인 방식으로 추가

**Spring Boot ↔ Python FastAPI**:
- REST API (HTTP/JSON)
- 엔드포인트:
  - `POST /rules/generate` - 규칙 생성
  - `POST /scrape/execute` - 데이터 추출

## 기능 요구사항

### 1. 규칙 생성 (Rule Generation)

**입력**:
- 지자체 웹사이트 URL
- 샘플 HTML

**LLM 태스크**:
1. 회의 목록 페이지에서 회의 행들 찾기
2. 각 회의 행에서 제목, 날짜, 링크 추출
3. 문서 다운로드 링크들 찾기

**출력 형식** (JSON):
```json
{
  "jurisdiction": "cherokee_county",
  "description": "Cherokee County 회의 목록 추출 규칙",
  "steps": [
    {
      "step": 1,
      "action": "find_container",
      "selector": "#AgendaCenterContent",
      "validation": "컨테이너가 존재하고 table을 포함해야 함"
    },
    {
      "step": 2,
      "action": "find_meetings",
      "selector": "table tbody tr",
      "description": "각 tr이 하나의 회의"
    },
    {
      "step": 3,
      "action": "extract_fields",
      "fields": {
        "title": {
          "selector": "td:nth-child(2) a",
          "type": "text",
          "required": true
        },
        "date": {
          "selector": "td:nth-child(1)",
          "type": "date",
          "format": "MM/DD/YYYY"
        },
        "url": {
          "selector": "td:nth-child(2) a",
          "attribute": "href"
        }
      }
    }
  ]
}
```

**실행 시점**:
- 최초 1회 생성 → JSON 파일 저장
- 이후 재사용
- 신뢰성 검증 실패 시 재생성

### 2. 데이터 스크래핑 (Scraping)

**프로세스**:
1. 저장된 규칙 로드
2. LLM에게 규칙 + HTML 전달
3. LLM이 규칙에 따라 데이터 추출
4. 추출된 데이터 반환

**추출 데이터**:
```json
{
  "meetings": [
    {
      "title": "Planning Commission Meeting",
      "date": "2024-01-15",
      "url": "https://...",
      "documents": [
        {
          "type": "minutes",
          "url": "https://.../minutes.pdf"
        },
        {
          "type": "agenda",
          "url": "https://.../agenda.pdf"
        },
        {
          "type": "packet",
          "url": "https://.../packet.pdf"
        },
        {
          "type": "video",
          "url": "https://granicus.com/..."
        }
      ]
    }
  ]
}
```

### 3. 신뢰성 검증 (Reliability Validation)

**검증 시점**:
- 데이터 추출 후 매번

**LLM 태스크**:
- 추출된 데이터가 올바른 회의 정보인지 검증
- 필드별 신뢰도 평가

**출력 형식**:
```json
{
  "overall_confidence": 0.85,
  "field_validation": {
    "title": {
      "valid": true,
      "confidence": 0.95
    },
    "date": {
      "valid": true,
      "confidence": 0.90
    },
    "url": {
      "valid": false,
      "confidence": 0.50,
      "issue": "일부 URL이 상대경로"
    }
  }
}
```

**재생성 트리거**:
- `overall_confidence < 0.8`
- 또는 특정 필드 `confidence < 0.7`

### 4. 데이터 정제 (Data Cleaning)

**대상**:
- 스크래핑된 원본 데이터

**작업**:

**Minutes PDF → .md 변환**:
- LLM에게 PDF 내용 전달
- Markdown 형식으로 변환
- 구조화된 포맷 적용

**기타 PDF (Agenda, Packet)**:
- 원본 다운로드
- .md 파일에 링크 걸기

**파일 구조**:
```
data/
├── cherokee_county/
│   ├── meeting-2024-01-15/
│   │   ├── minutes.md        # LLM이 변환
│   │   ├── agenda.pdf        # 원본 다운로드
│   │   ├── packet.pdf
│   │   └── metadata.json
```

### 5. 중요 데이터 추출 (Key Data Extraction)

**입력**:
- minutes.md (변환된 회의록)

**LLM 태스크**:
- 중요 정보 추출 + 요약

**추출 항목**:

1. **회의 메타 정보**:
   - 회의 일시
   - 참석자
   - 장소
   - 회의 종류

2. **안건 및 결정사항**:
   - 논의된 안건 목록
   - 투표 결과
   - 승인/거부 항목

3. **Planning & Zoning 특화 정보**:
   - 개발 프로젝트명
   - 신청자/개발자
   - 위치/주소
   - 승인/조건부승인/거부

4. **요약**:
   - 회의 전체 요약 (3-5문장)
   - 주요 결정사항 요약

**출력 형식** (JSON → MongoDB 저장):
```json
{
  "meeting_id": "cherokee-2024-01-15",
  "metadata": {
    "date": "2024-01-15T19:00:00",
    "attendees": ["John Doe", "Jane Smith"],
    "location": "City Hall",
    "type": "Planning Commission"
  },
  "agenda_items": [
    {
      "item": "Rezoning Application - 123 Main St",
      "applicant": "ABC Development",
      "location": "123 Main St, Cherokee County",
      "decision": "Approved",
      "vote": "5-0"
    }
  ],
  "summary": "Planning Commission approved 3 rezoning applications..."
}
```

**파일 시스템**:
- 원본 PDF, HTML
- 변환된 .md 파일

### AI

**Anthropic Claude API**:
- 모델: Claude 3.5 Sonnet (또는 최신)
- 용도:
  - 규칙 생성
  - 데이터 추출
  - 신뢰성 검증
  - PDF → Markdown 변환
  - 중요 데이터 추출

## 데이터 파이프라인

### 3단계 파이프라인

```
1. Scraping & Storage
   ↓
2. Data Cleaning
   ↓
3. Key Data Extraction
```

### 1단계: Scraping & Storage

**입력**:
- 지자체 URL
- 저장된 규칙 (JSON)

**프로세스**:
1. 규칙 로드
2. Playwright로 페이지 접근
3. LLM + 규칙으로 데이터 추출
4. 신뢰성 검증
5. 검증 실패 → 규칙 재생성 → 재실행

**출력**:
- PostgreSQL: 회의 메타데이터
- 파일 시스템: 원본 HTML, PDF

### 2단계: Data Cleaning

**입력**:
- 1단계에서 수집된 원본 데이터

**프로세스**:
- **Minutes PDF**:
  1. PDF 텍스트 추출
  2. LLM에게 전달
  3. Markdown 변환
  4. `minutes.md` 저장

- **기타 PDF (Agenda, Packet)**:
  1. 원본 다운로드
  2. 파일 저장
  3. `minutes.md`에 링크 추가

**출력**:
- 파일 시스템:
  ```
  meeting-2024-01-15/
  ├── minutes.md      # 변환됨
  ├── agenda.pdf      # 원본
  ├── packet.pdf      # 원본
  └── metadata.json
  ```

### 3단계: Key Data Extraction

**입력**:
- `minutes.md` (변환된 회의록)

**프로세스**:
1. LLM에게 `minutes.md` 전달
2. 구조화된 정보 추출
3. JSON 생성

**출력**:
- MongoDB: 추출된 JSON 데이터
- PostgreSQL: 추출 완료 상태 업데이트

## Agentic 시스템 설계

### Agentic의 정의

**시나리오 C - 매번 LLM 사용 + 태스크 분할**:
- 문서 수집 시마다 LLM 호출
- 작은 태스크로 나눠서 질문
- 실시간 데이터 추출

### 규칙 기반 + LLM 하이브리드

**프로세스 1: 규칙 정의** (최초 1회):
```
LLM에게 질문:
"이 HTML에서 회의 목록을 추출하려면 어떤 단계가 필요한가?"

LLM 응답:
{
  "steps": [
    {"action": "find_container", "selector": "..."},
    {"action": "find_meetings", "selector": "..."},
    {"action": "extract_fields", "fields": {...}}
  ]
}

→ JSON 파일로 저장
```

**프로세스 2: 데이터 추출** (매번):
```
LLM에게 전달:
- 규칙 (JSON)
- 현재 HTML

LLM 응답:
- 추출된 회의 데이터 (JSON)
```

### Self-Healing (자가 치유)

**신뢰성 검증**:
```
데이터 추출 후:
→ LLM에게 질문: "이 데이터가 올바른 회의 정보인가?"
→ 필드별 신뢰도 점수 반환
```

**재생성 트리거**:
```
if overall_confidence < 0.8:
    1. 규칙이 잘못되었다고 판단
    2. 규칙 재생성 (프로세스 1)
    3. 새 규칙으로 재실행 (프로세스 2)
```

### LLM 호출 최적화

**캐싱 전략**:
- 규칙은 파일로 저장 (재사용)
- 동일 HTML → 동일 규칙 → 캐시된 결과 사용 가능
- 변경 감지 시에만 재생성

**비용 관리**:
- 규칙 생성: 지자체당 1회 (총 4회)
- 데이터 추출: 회의당 1회
- 신뢰성 검증: 회의당 1회
- PDF 변환: Minutes만 (회의당 1회)

## 스토리지 전략

### PostgreSQL (구조화된 메타데이터)

**테이블**:

```sql
-- 지자체
CREATE TABLE jurisdictions (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    platform VARCHAR(100),
    base_url TEXT,
    trigger_type VARCHAR(50), -- 'cron', 'polling', 'event', 'manual'
    trigger_config JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 회의
CREATE TABLE meetings (
    id SERIAL PRIMARY KEY,
    jurisdiction_id INTEGER REFERENCES jurisdictions(id),
    title TEXT,
    meeting_date TIMESTAMP,
    url TEXT,
    status VARCHAR(50), -- 'pending', 'scraped', 'cleaned', 'extracted'
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 문서
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    meeting_id INTEGER REFERENCES meetings(id),
    doc_type VARCHAR(50), -- 'minutes', 'agenda', 'packet', 'video'
    url TEXT,
    file_path TEXT,
    file_size BIGINT,
    checksum VARCHAR(64),
    status VARCHAR(50), -- 'pending', 'downloaded', 'converted'
    created_at TIMESTAMP DEFAULT NOW()
);

-- 스크래핑 실행 이력
CREATE TABLE scraping_runs (
    id SERIAL PRIMARY KEY,
    jurisdiction_id INTEGER REFERENCES jurisdictions(id),
    run_type VARCHAR(50), -- 'backfill', 'continuous'
    started_at TIMESTAMP,
    finished_at TIMESTAMP,
    status VARCHAR(50), -- 'running', 'success', 'failed'
    meetings_found INTEGER,
    documents_downloaded INTEGER,
    errors JSONB
);
```
