## 3. Architecture Notes

### 3.1 정의
> "How does your agentic approach work?"

시스템의 아키텍처와 **Agentic 접근법**을 설명하는 기술 문서

### 3.2 필수 섹션

#### 구조 템플릿
```markdown
# Architecture Notes

## 1. System Overview
### 1.1 High-Level Architecture
```
[다이어그램]
User → Main Orchestrator → Jurisdiction Scrapers
                         ↓
                    LLM Agent (Claude)
                         ↓
                    Storage Layer (SQLite + FS)
```

### 1.2 Key Components
- **Orchestrator**: 작업 흐름 관리
- **Scraper**: 웹 페이지 처리
- **LLM Agent**: 지능형 분석
- **Storage**: 문서 및 메타데이터 저장

## 2. Agentic Approach
### 2.1 What Makes It "Agentic"?
> 단순 규칙 기반 스크래퍼와의 차별점 설명

**Traditional Scraper:**
```python
# 고정된 CSS selector
links = soup.select("table.meetings a.pdf-link")
```

**Agentic Scraper:**
```python
# LLM이 페이지 분석
prompt = f"이 HTML에서 회의록 PDF 링크를 찾아줘: {html}"
response = claude.complete(prompt)
links = extract_links(response)
```

### 2.2 LLM Integration Strategy
**사용 시점:**
1. 초기 사이트 분석 (1회)
2. 문서 타입 분류 (각 문서)
3. 사이트 구조 변경 감지 시 (에러 발생)

**프롬프트 예시:**
```
You are analyzing a municipal website for meeting documents.

HTML:
{html_snippet}

Task: Extract all meeting minutes links from this page.

Output format:
{
  "links": [
    {"url": "...", "meeting_date": "2024-01-15", "title": "..."},
    ...
  ]
}
```

**비용 최적화:**
- 규칙 캐싱 (한 번 학습하면 재사용)
- HTML 길이 제한 (중요 부분만 전송)
- Batch processing

### 2.3 Adaptability Mechanisms
**사이트 변경 감지:**
```python
try:
    documents = scraper.scrape_with_rules(cached_rules)
except ParsingError:
    # LLM 재개입
    new_rules = llm_agent.analyze_page(url)
    update_cached_rules(new_rules)
    documents = scraper.scrape_with_rules(new_rules)
```

## 3. Data Pipeline
```
Discovery → Extraction → Classification → Storage → Validation
```

### 3.1 Discovery
- 어떤 페이지를 방문할지 결정
- 페이지네이션 처리

### 3.2 Extraction
- HTML 파싱
- 링크 및 메타데이터 추출

### 3.3 Classification (Agentic)
- LLM으로 문서 타입 판단
- Minutes vs Agenda vs Packet

### 3.4 Storage
- 파일 저장 (계층적 구조)
- 메타데이터 DB 저장

### 3.5 Validation
- Checksum 검증
- 중복 체크

## 4. Technology Stack
| Layer | Technology | Rationale |
|-------|------------|-----------|
| Language | Python 3.11 | 풍부한 라이브러리 생태계 |
| Scraping | Playwright | 동적 페이지 지원 |
| LLM | Claude 3.5 Sonnet | 긴 컨텍스트, 구조화된 출력 |
| Database | SQLite | MVP에 적합, 경량 |
| Storage | Local FS | 간단, 나중에 S3로 이전 가능 |

## 5. Error Handling
- Network errors: Exponential backoff
- Parsing errors: LLM fallback
- Rate limiting: Delay + retry

## 6. Scalability Considerations
- 플러그인 아키텍처 (새 지자체 추가 용이)
- 상태 저장으로 재개 가능
- 병렬 처리 준비 (현재 순차)
```

### 3.3 다이어그램 포함 (권장)

**시스템 아키텍처:**
```
┌─────────────┐
│   User      │
└──────┬──────┘
       │
┌──────▼──────────────────────────┐
│   Main Orchestrator             │
│   - Backfill scheduler          │
│   - Error handling              │
└──────┬──────────────────────────┘
       │
┌──────▼─────────────┬─────────────┬─────────────┐
│  Jurisdiction 1    │   Jur. 2    │   Jur. 3    │
│  Scraper           │   Scraper   │   Scraper   │
└──────┬─────────────┴─────────────┴─────────────┘
       │
┌──────▼──────────────────┐
│   LLM Agent (Claude)    │
│   - Page analysis       │
│   - Doc classification  │
└──────┬──────────────────┘
       │
┌──────▼──────────────────┐
│   Storage Layer         │
│   - SQLite (metadata)   │
│   - Filesystem (files)  │
└─────────────────────────┘
```

**Agentic Decision Flow:**
```
Start
  │
  ├─→ Try cached rules
  │   │
  │   ├─ Success? → Continue
  │   │
  │   └─ Fail? → LLM Analysis
  │              │
  │              ├─ Extract new rules
  │              ├─ Cache rules
  │              └─ Retry with new rules
  │
  └─→ End
```
