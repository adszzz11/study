# 운영 모드 명세

> Backfill 및 Continuous Updates 모드 정의

## 🔄 운영 모드 개요

시스템은 두 가지 운영 모드를 지원해야 합니다:

| 모드 | 목적 | 실행 빈도 | 우선순위 |
|------|------|-----------|----------|
| **Backfill** | 과거 데이터 일괄 수집 | 1회 (지자체당) | 1순위 |
| **Continuous Updates** | 신규 데이터 지속 감지 | 주기적 (일/시간 단위) | 2순위 |

---

## 1. Backfill Mode (과거 데이터 수집)

### 1.1 정의
**Backfill**은 새로운 지자체를 시스템에 추가할 때, 과거 회의 문서를 일괄적으로 수집하는 초기 작업입니다.

### 1.2 명세

| 항목 | 세부사항 | 확실성 |
|------|----------|--------|
| **목표 기간** | 1-2년 | ⚠️ 90% (범위 조정 가능) |
| **실행 시점** | 지자체 온보딩 시 1회 | ✅ 100% |
| **처리 방식** | 배치 작업 | ✅ 100% |
| **종료 조건** | 목표 기간 전체 수집 완료 | ✅ 100% |

### 1.3 프로세스

```
[Backfill 프로세스]

1. 시작 날짜 계산
   └─> today - N년 (N = 1 or 2)

2. 역순 탐색 (최신 → 과거)
   └─> 페이지네이션 처리
   └─> 각 회의별 문서 다운로드

3. 진행 상황 추적
   └─> 수집된 문서 수
   └─> 커버 기간
   └─> 실패한 항목

4. 완료 조건 확인
   └─> 목표 날짜까지 도달
   └─> 더 이상 문서 없음
```

### 1.4 기술적 요구사항

#### 필수 기능
- ✅ **날짜 범위 지정**: 시작/종료 날짜 설정
- ✅ **중복 방지**: 이미 수집된 문서 스킵
- ✅ **진행률 표시**: 현재 진행 상황 로깅
- ✅ **재개 가능**: 중단 시 이어서 수집
- ✅ **실패 처리**: 일부 실패해도 계속 진행

#### 선택적 기능
- ⚠️ **병렬 처리**: 여러 페이지 동시 수집 (Phase 2)
- ⚠️ **속도 제한**: Rate limiting (Phase 2)

### 1.6 예상 작업량 (1개 지자체 기준)

| 항목 | 보수적 추정 | 낙관적 추정 |
|------|-------------|-------------|
| **회의 빈도** | 월 2회 | 월 4회 |
| **1년 회의 수** | 24개 | 48개 |
| **2년 회의 수** | 48개 | 96개 |
| **문서 수 (Minutes만)** | 48개 | 96개 |
| **모든 문서 타입** | 144-192개 | 288-384개 |

**스토리지 추정 (PDF 기준):**
- Minutes: 500KB - 2MB 각
- Agendas: 200KB - 1MB 각
- Packets: 5MB - 50MB 각

**총 스토리지 (2년, 모든 타입):**
- 최소: ~500MB
- 최대: ~5GB

- 진행률 대시보드

## 2. Continuous Updates Mode (지속적 업데이트)

### 2.1 정의
**Continuous Updates**는 Backfill 완료 후, 신규로 게시되는 회의 문서를 자동으로 감지하고 수집하는 지속적 운영 모드입니다.

### 2.2 명세

| 항목 | 세부사항 | 확실성 |
|------|----------|--------|
| **목표** | 신규 문서 자동 수집 | ✅ 100% |
| **실행 방식** | 주기적 폴링 | ⚠️ 80% (구현 방법 선택 필요) |
| **빈도** | TBD (일/시간 단위) | ❓ 클라이언트 질의 필요 |
| **지속 기간** | 무한 (서비스 종료까지) | ✅ 100% |

### 2.3 프로세스

```
[Continuous Update 프로세스]

1. 스케줄러 실행 (예: 매일 새벽 2시)
   │
2. 마지막 수집 시점 확인
   └─> DB에서 last_update_date 조회
   │
3. 신규 문서 감지
   └─> 사이트 방문
   └─> last_update_date 이후 문서만 필터
   │
4. 신규 문서 다운로드
   └─> 변경사항 있으면 수집
   └─> 없으면 조기 종료
   │
5. 상태 업데이트
   └─> last_update_date 갱신
   └─> 수집 통계 기록
```

### 2.4 변경 감지 전략

#### Option A: Polling (추천 - MVP)
```python
# 정기적으로 사이트 방문
def check_for_updates():
    last_check = get_last_update_date("cherokee_county")
    new_docs = scraper.get_documents_since(last_check)
    if new_docs:
        download_and_store(new_docs)
        update_last_check_date()
```

**장점:**
- ✅ 구현 간단
- ✅ 모든 사이트에 적용 가능
- ✅ 안정적

**단점:**
- ⚠️ 불필요한 요청 발생 가능
- ⚠️ 실시간 아님 (지연 있음)

#### Option B: RSS/Atom Feed (사이트 지원 시)
```python
# RSS 피드 구독
def check_rss_feed():
    feed = parse_rss("https://example.gov/meetings.rss")
    for entry in feed.entries:
        if entry.published > last_check:
            download_document(entry.link)
```

**장점:**
- ✅ 효율적 (변경 시만 감지)
- ✅ 표준 방식

**단점:**
- ❌ 지자체 사이트가 지원 안 할 가능성 높음

#### Option C: HTML Diff (고급)
```python
# 페이지 변경 감지
def detect_page_changes():
    current_html = fetch_page()
    previous_html = load_cached_html()
    diff = compare_html(current_html, previous_html)
    if diff:
        extract_new_documents(diff)
```

**장점:**
- ✅ 정확한 변경 감지

**단점:**
- ⚠️ 복잡도 높음
- ⚠️ 사소한 변경도 감지됨

**결정: Option A (Polling) 채택**

### 2.5 스케줄링 옵션

| 빈도 | 실시간성 | 비용 | 적합성 |
|------|----------|------|--------|
| **시간 단위** | ⚡ 높음 | 💰💰💰 | 실시간 요구사항 있을 때 |
| **일 1회** | ⚡ 중간 | 💰 | ⭐⭐⭐⭐⭐ 추천 (MVP) |
| **주 2-3회** | ⚡ 낮음 | 💰 | 긴급성 낮을 때 |

**MVP 제안: 매일 1회 (새벽 2시)**

### 2.7 최적화 전략

#### Intelligent Scheduling
```python
# 회의 패턴 학습
if no_changes_for_30_days:
    reduce_check_frequency()  # 월 2회로 감소

if meeting_season:  # 회기 중
    increase_check_frequency()  # 주 3회로 증가
```

```python
# 변경 없으면 조기 종료
if page_etag == cached_etag:
    return  # 페이지 파싱 안 함
```

## 4. 비기능적 요구사항

### 4.1 성능

| 항목 | 목표 | 측정 방법 |
|------|------|-----------|
| **Backfill 속도** | 시간당 50+ 문서 | 로그 분석 |
| **Update 응답시간** | 10분 이내 | 스케줄러 로그 |
| **에러율** | < 5% | 실패/총 문서 수 |

### 4.2 안정성

- ✅ **재시작 가능**: 중단 시 이어서 수집
- ✅ **멱등성**: 같은 문서 여러 번 실행해도 중복 저장 안 됨
- ✅ **부분 실패 허용**: 일부 문서 실패해도 전체 계속

### 4.3 모니터링

**Backfill 모니터링:**
- 진행률 (%)
- 수집 속도 (문서/시간)
- 실패율
- 예상 완료 시각

**Continuous Update 모니터링:**
- 마지막 체크 시각
- 신규 문서 수
- 연속 무변경 횟수
- 에러 발생 여부
