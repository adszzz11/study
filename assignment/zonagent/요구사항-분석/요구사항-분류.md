## 🔍 2. 기술적 검토가 필요한 사항

### 2.1 "Agentic" 시스템 구현 범위

**문제:**
- "Agentic"의 정확한 정의가 모호함
- 얼마나 자율적이어야 하는가?
- LLM 활용 범위는?

**선택지:**

| 접근법 | 설명 | 장점 | 단점 | 추천도 |
|--------|------|------|------|--------|
| **Minimal Agentic** | LLM으로 문서 분류만 | 빠르고 저렴 | Agentic 특성 약함 | ⭐⭐ |
| **Moderate Agentic** | LLM으로 페이지 구조 이해 + 규칙 생성 | 균형적 | 복잡도 중간 | ⭐⭐⭐⭐⭐ |
| **Full Agentic** | LLM이 모든 결정 수행 | 완전 자율적 | 비용/속도/안정성 문제 | ⭐⭐⭐ |

**결정 필요:**
1. 어느 수준의 자율성을 목표로 하는가?
2. 사이트 변경 시 얼마나 빠르게 적응해야 하는가?
3. 비용 대비 효과를 어떻게 평가할 것인가?

**제안:**
- **Moderate Agentic 접근법 (Hybrid)** 채택
  - 초기: LLM이 사이트 분석 → 규칙 생성
  - 이후: 규칙 기반 실행 (빠르고 저렴)
  - 변경 감지: LLM 재개입
  - 비용 효율 + 적응성 균형

### 2.2 아키텍처 설계

**문제:**
- 모듈 구조를 어떻게 설계할 것인가?
- 확장 가능한 구조는?

**선택지:**

#### Option A: 단일 스크립트 (Monolithic)
```
main.py
├── scrape_cherokee()
├── scrape_holly_springs()
├── scrape_alpharetta()
└── scrape_marietta()
```
**장점**: 빠른 프로토타이핑
**단점**: 확장성 낮음, 중복 코드

#### Option B: 플러그인 아키텍처 (추천)
```
scraper/
├── core/
│   ├── base_scraper.py      # 추상 클래스
│   ├── llm_agent.py          # LLM 통합
│   └── storage.py            # 문서 저장
├── jurisdictions/
│   ├── cherokee.py
│   ├── holly_springs.py
│   ├── alpharetta.py
│   └── marietta.py
└── main.py                   # Orchestrator
```
**장점**: 확장 용이, 재사용 가능
**단점**: 초기 설계 시간 필요

#### Option C: 완전 동적 (Full Agentic)
```
scraper/
├── agent.py                  # LLM 에이전트
├── tools/                    # 에이전트가 사용할 도구들
└── config.json              # 지자체 URL만 저장
```
**장점**: 새 지자체 추가 시 코드 변경 불필요
**단점**: 복잡도 높음, 디버깅 어려움

**제안:**
- **Option B (플러그인 아키텍처) 채택**
  - MVP: 1-2개 지자체로 패턴 검증
  - 공통 인터페이스 추출
  - 나머지 지자체는 플러그인으로 추가

### 2.3 LLM 통합 방식

**결정 필요:**

| 항목 | 선택지 | 고려사항 |
|------|--------|----------|
| **LLM 제공자** | Claude API vs OpenAI vs Both | - Claude: 긴 컨텍스트, 구조화 출력<br>- OpenAI: 함수 호출, 생태계 |
| **호출 시점** | 매 페이지 vs 초기만 vs 실패시만 | - 비용 vs 적응성 트레이드오프 |
| **프롬프트 전략** | Zero-shot vs Few-shot vs Chain-of-Thought | - 정확도 vs 토큰 사용량 |
| **출력 형식** | JSON vs 자연어 vs 구조화된 출력 | - 파싱 안정성 |

**제안:**
```python
# Hybrid 전략
1. 첫 방문 시: LLM이 페이지 분석
   - Input: HTML + 목표 ("회의록 링크 찾기")
   - Output: JSON (선택자 규칙 + 신뢰도)

2. 규칙 저장 및 재사용
   - 성공 시: 규칙 캐싱
   - 실패 시: LLM 재호출

3. 정기 검증
   - 주 1회: LLM으로 규칙 유효성 확인
   - 변경 감지: 자동 업데이트
```

**LLM 선택:**
- **Primary**: Claude 3.5 Sonnet
  - 긴 HTML 처리 가능 (200K 토큰)
  - 구조화된 출력 (JSON mode)
- **Fallback**: GPT-4 Turbo

### 2.5 에러 핸들링 및 재시도 전략

**결정 필요:**

| 에러 타입 | 전략 | 구현 복잡도 |
|-----------|------|-------------|
| **네트워크 에러** | Exponential backoff (3회) | 낮음 |
| **파싱 실패** | LLM 재분석 요청 | 중간 |
| **404 Not Found** | 로깅만 (재시도 X) | 낮음 |
| **Rate Limiting** | 대기 후 재시도 | 중간 |
| **사이트 구조 변경** | LLM 재학습 | 높음 |

**제안:**
```python
# 재시도 우선순위
1. 네트워크 에러: 즉시 재시도 (exponential backoff)
2. 파싱 실패: LLM 재분석 (1회)
3. 구조 변경: 알림 + 수동 확인 (MVP)
               → 자동 재학습 (나중)
```

### 2.6 스케줄링 및 Continuous Update 구현

**결정 필요:**

| 방법 | 설명 | 적합성 |
|------|------|--------|
| **Polling** | 정기적으로 사이트 방문 (예: 매일) | ⭐⭐⭐⭐ 간단하고 안정적 |
| **Webhook** | 사이트가 변경 알림 (불가능) | ⭐ 지자체 사이트는 미지원 |
| **RSS/Atom** | 피드 구독 | ⭐⭐ 일부 사이트만 지원 |
| **Change Detection** | HTML diff 비교 | ⭐⭐⭐ 복잡하지만 정확 |

**제안:**
- **Polling 방식 (매일 1회)**
  - 새벽 시간대 실행 (서버 부하 낮음)
  - 마지막 수집 날짜 이후만 확인
  - 변경 없으면 조기 종료

**구현:**
```python
# Cron job or APScheduler
@scheduler.scheduled_job('cron', hour=2)  # 매일 새벽 2시
def daily_update():
    for jurisdiction in jurisdictions:
        scraper = get_scraper(jurisdiction)
        scraper.scrape_since(last_update_date)
```

# 테스트 우선순위
1. Unit Tests (핵심 로직)
   - 날짜 파싱

   - 파일명 생성

2. Integration Tests (Mock HTML)
   - 저장된 HTML 샘플로 테스트
   - 실제 사이트 부하 주지 않음

3. Smoke Tests (실제 사이트)
   - CI/CD에서 주 1회 실행
   - 사이트 구조 변경 감지
